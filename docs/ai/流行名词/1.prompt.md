# Prompt

> Prompt 通常翻译成“提示词”或“提示”。  
> 在大模型应用里，Prompt 不是一句魔法咒语，而是人类和模型之间的任务接口。

## 先说结论

Prompt 可以理解为：

```text
我把任务、背景、规则、示例和输出要求，
用自然语言组织成模型能理解的输入。
```

普通人看 Prompt，容易以为它只是“问问题的方式”。

但从 AI 工程角度看，Prompt 更像是：

```text
需求说明书 + 上下文 + 执行规则 + 输出协议
```

写好 Prompt 的核心，不是堆很多漂亮的话，而是把任务边界讲清楚，让模型知道：

- 要做什么；
- 基于什么资料做；
- 不要做什么；
- 按什么步骤做；
- 输出成什么格式；
- 遇到不确定时怎么处理。

## Prompt 什么时候出现的

Prompt 这个词本身并不是大模型时代才出现的。

在计算机领域，Prompt 很早就存在。比如命令行里的：

```text
C:\>
$
请输入用户名：
```

这些都是 prompt，意思是“系统提示用户输入”。

早期的 prompt 是：

```text
机器提示人：请你输入内容。
```

到了大模型时代，Prompt 的含义反过来了：

```text
人提示模型：请你按我的要求完成任务。
```

所以 Prompt 有两个层面的历史：

| 阶段           | Prompt 的含义                              | 典型场景                            |
| -------------- | ------------------------------------------ | ----------------------------------- |
| 传统计算机时代 | 系统提示用户输入                           | 命令行、表单、交互式程序            |
| 早期 NLP 时代  | 用模板引导模型完成分类、填空等任务         | 情感分类、文本匹配、问答            |
| 大语言模型时代 | 用自然语言描述任务，让模型直接完成复杂工作 | ChatGPT、Claude、Gemini、Agent、RAG |

真正让 Prompt 变成一个热门概念的，是大语言模型的能力变化。

尤其是 GPT-3 之后，研究者发现模型可以通过上下文里的少量示例学会任务，这叫：

```text
In-context Learning
```

也就是：

```text
不用重新训练模型，
只要在输入里给任务说明和示例，
模型就能临时学会怎么做。
```

ChatGPT 普及之后，Prompt 从研究概念变成了普通用户也能感知的能力。  
于是“Prompt Engineering”开始流行，很多人开始研究如何把问题说清楚、把模型用好。

## prompt engineering 概念是什么?

Prompt Engineering 通常翻译成“提示词工程”。

它不是简单地“写一句提示词”，而是：

```text
为了让大模型稳定完成任务，
对输入内容、上下文结构、输出格式和约束规则进行设计、测试和优化的过程。
```

如果说 Prompt 是一次具体输入，那么 Prompt Engineering 就是设计这类输入的方法论。

可以这样区分：

| 概念 | 含义 | 例子 |
| --- | --- | --- |
| Prompt | 给模型的一次具体提示 | “请解释 RAG 是什么” |
| Prompt Template | 可复用的提示模板 | “请解释 {concept}，要求先说结论、再举例” |
| Prompt Engineering | 设计、测试、优化 Prompt 的过程 | 调整任务说明、示例、格式、约束，让输出更稳定 |
| Context Engineering | 更大范围的上下文设计 | 把系统规则、用户问题、历史对话、RAG 资料、工具说明一起组织好 |

Prompt Engineering 解决的不是“怎么把话说得好听”，而是“怎么让模型更可靠”。

它关注的问题包括：

- 模型是否理解任务；
- 模型是否知道边界；
- 模型是否能按格式输出；
- 模型是否会编造；
- 模型是否能处理复杂步骤；
- 模型输出是否稳定；
- Prompt 是否方便复用、测试和维护。

## 为什么需要 Prompt Engineering

大模型不是传统函数。

传统函数更像：

```ts
function add(a: number, b: number) {
  return a + b;
}
```

只要输入一样，输出通常就是确定的。

大模型更像：

```text
给它一段上下文，
它基于概率生成最可能的后续内容。
```

这带来几个问题：

1. **同一个问题可能有不同回答**：模型输出有随机性。
2. **任务描述不清时会自己脑补**：模型会补全缺失信息。
3. **没有边界时容易编造**：尤其是事实类问题。
4. **格式不清时输出不可控**：比如你想要 JSON，它可能输出解释文字。
5. **复杂任务容易漏步骤**：模型可能只完成一部分。

Prompt Engineering 的作用，就是把模糊任务变成更清晰的模型输入。

可以把它理解成：

```text
把“我想要什么”
翻译成
“模型应该如何执行”
```

## Prompt Engineering 包含什么

Prompt Engineering 通常包括这几类工作：

### 1. 任务定义

先明确模型到底要完成什么。

不好的写法：

```text
帮我优化一下。
```

更好的写法：

```text
请优化下面这段文章的表达，让它更适合技术博客读者阅读。
要求保留原意，不要增加新观点。
```

### 2. 背景补充

模型不知道你的真实上下文，所以需要补充必要背景。

例如：

```text
我是前端开发者，正在学习 AI 工程。
请用我已有的前端经验类比解释 Agent。
```

### 3. 约束设计

告诉模型哪些事情不能做。

例如：

```text
如果资料中没有答案，请直接说“资料不足”，不要根据常识补充。
```

### 4. 输出格式设计

让模型按固定结构输出，方便阅读或程序处理。

例如：

```text
请只输出 JSON，格式如下：

{
  "title": string,
  "summary": string,
  "tags": string[]
}
```

### 5. 示例设计

通过示例告诉模型“什么是我想要的结果”。

例如：

```text
示例：
输入：Prompt 是什么？
输出：Prompt 是人类给模型的任务说明，用来控制模型理解什么、怎么做、怎么输出。
```

### 6. 评估和迭代

Prompt 写完后不能只看一次结果。

应该准备几组测试输入，观察：

- 是否都能正确完成任务；
- 输出格式是否稳定；
- 有没有幻觉；
- 有没有漏掉边界情况；
- 换一个模型是否仍然可用。

这一步才真正体现“工程”两个字。

## Prompt Engineering 和写作文案的区别

Prompt Engineering 看起来像写文字，但本质更接近设计接口。

写作文案关注：

```text
表达是否优美
读者是否喜欢
语气是否自然
```

Prompt Engineering 关注：

```text
模型是否理解
任务是否可执行
输出是否稳定
结果是否可验证
失败是否可处理
```

所以好的 Prompt 不一定华丽，但一定清楚、可执行、可测试。

例如：

```text
你是一个世界顶级专家，请深入全面地分析这个问题。
```

这句话看起来很强，但工程价值有限，因为它没有说明：

- 分析什么；
- 从哪些维度分析；
- 输出多长；
- 给谁看；
- 用什么格式；
- 如何处理不确定信息。

更工程化的写法是：

```text
请从“定义、适用场景、优点、缺点、工程落地”五个维度分析 RAG。
读者是有前端经验但 AI 基础较弱的开发者。
输出使用 Markdown，先给结论，再用表格对比，最后给项目实践建议。
如果某一点无法确定，请明确说明不要猜测。
```

## Prompt Engineering 的目标

Prompt Engineering 的最终目标不是“写出万能提示词”。

真正的目标是：

```text
让模型输出更可控、更稳定、更符合任务需求。
```

具体来说，就是提升这几件事：

| 目标 | 含义 |
| --- | --- |
| 准确性 | 回答更接近事实和任务要求 |
| 稳定性 | 多次运行结果风格和格式更一致 |
| 可控性 | 能控制语气、长度、结构、边界 |
| 可解析性 | 输出能被程序继续处理 |
| 可复用性 | 同一模板能处理同类任务 |
| 可评估性 | 能判断输出好不好 |
| 可维护性 | Prompt 变化后容易测试和回滚 |

## Prompt Engineering 的局限

Prompt Engineering 很重要，但它不是万能的。

它不能完全解决：

- 模型本身能力不足；
- 资料缺失；
- 训练知识过时；
- 工具结果错误；
- 长上下文遗忘；
- 高风险场景中的事实校验。

所以在真实系统里，Prompt Engineering 往往要和其他工程手段配合：

- RAG：补充外部知识；
- Tool Calling：调用真实工具；
- Structured Output：约束输出格式；
- Guardrails：做安全和格式校验；
- Evaluation：用测试集评估效果；
- Human Review：高风险结果人工确认。

用自己的话说：

```text
Prompt Engineering 是把模型从“会聊天”
推向“能稳定完成任务”的第一层工程化手段。
```

## Prompt 的演进

Prompt 的演进，本质上是人类和模型协作方式的演进。

### 阶段一：关键词式提问

最早很多人使用模型，就像使用搜索引擎：

```text
RAG 是什么
JavaScript 闭包
大模型厂商
```

这种方式很简单，但问题也明显：

- 任务不清楚；
- 输出风格不稳定；
- 深度不可控；
- 模型容易自由发挥。

这种 Prompt 适合查概念，不适合做复杂任务。

### 阶段二：指令式 Prompt

后来大家开始给模型明确任务：

```text
请用通俗语言解释 RAG 是什么，并举一个例子。
```

比关键词式更好，因为它说明了：

- 要解释什么；
- 用什么语言风格；
- 需要举例子。

这是 Prompt 最基础也最常用的形态。

### 阶段三：角色式 Prompt

再往后，人们开始给模型设定角色：

```text
你是一名资深 AI 工程师，请用适合前端开发者理解的方式解释 RAG。
```

角色式 Prompt 的作用不是让模型真的“变成某个人”，而是给它一个回答标准：

- 使用什么视角；
- 假设读者是谁；
- 该关注哪些重点；
- 该避免哪些表达。

角色设定有用，但不能迷信。  
只写“你是专家”不够，还要说明任务、背景和输出要求。

### 阶段四：结构化 Prompt

当任务变复杂时，Prompt 开始变得结构化：

```text
背景：
我正在学习 AI 工程。

任务：
解释 Prompt 的含义。

要求：
1. 先说结论。
2. 用例子解释。
3. 对比 Prompt、RAG、Agent。
4. 最后总结。

输出格式：
使用 Markdown。
```

结构化 Prompt 的好处是：

- 信息更清楚；
- 模型更容易遵守；
- 输出更稳定；
- 便于复用和调试。

这也是工程项目中更推荐的写法。

### 阶段五：Few-shot Prompt

Few-shot 是在 Prompt 里给几个示例，让模型模仿。

例如：

```text
请把用户输入转换成 JSON。

示例 1：
输入：明天下午三点提醒我开会
输出：{"task":"开会","time":"明天下午三点"}

示例 2：
输入：周五上午九点提醒我提交周报
输出：{"task":"提交周报","time":"周五上午九点"}

现在处理：
输入：今晚八点提醒我复习 Prompt
输出：
```

Few-shot 的价值在于：

- 模型能看到你想要的格式；
- 模型能模仿你的判断标准；
- 对分类、抽取、转换类任务特别有用。

### 阶段六：Chain-of-Thought 和推理提示

当任务需要推理时，人们会让模型先分析再回答。

早期常见写法是：

```text
请一步一步思考。
```

现在更推荐的方式是：

```text
请先分析问题，最后只输出结论。
```

或者：

```text
请给出简洁的推理依据，不要展开隐藏思考过程。
```

原因是：  
我们需要模型更认真地解决问题，但不一定需要它把所有中间思考都暴露出来。

在工程里，更重要的是：

- 拆解任务；
- 验证每一步；
- 让模型输出可检查的结果；
- 对关键结论要求依据。

### 阶段七：工具调用 Prompt

进入 Agent 和 Tool Calling 阶段后，Prompt 不只是让模型回答，还要让模型决定是否调用工具。

例如：

```text
你可以使用以下工具：

1. searchDocs：搜索知识库
2. calculator：计算数学表达式
3. getWeather：查询天气

当问题需要实时信息或外部数据时，优先调用工具。
如果知识不足，不要编造。
```

这时 Prompt 变成了模型和工具系统之间的调度规则。

### 阶段八：系统级 Prompt

在真实产品里，Prompt 往往不再是一段简单文字，而是一套系统：

```text
System Prompt
+ 用户输入
+ 历史对话
+ 检索资料
+ 工具说明
+ 输出格式约束
+ 安全规则
```

也就是说，Prompt 已经从“写一句好提示词”，演进成了：

```text
上下文工程
```

这也是更高层次理解 Prompt 的关键。

## 如何写好 Prompt

写好 Prompt，可以先记住一个公式：

```text
角色 + 背景 + 任务 + 约束 + 示例 + 输出格式 + 失败处理
```

不是每次都要全部写上，但复杂任务最好尽量明确。

### 1. 先说明任务目标

不要只说：

```text
帮我看看这个。
```

更好的写法：

```text
请帮我检查这段代码是否有潜在 bug，并重点关注异步请求、空值处理和类型错误。
```

任务越清楚，模型越容易给出可用答案。

### 2. 给必要背景

模型不知道你的真实处境。  
如果背景不说，它只能猜。

例如：

```text
我是前端开发者，正在学习 AI 工程。
请用我能理解的方式解释 Embedding，不要使用太多数学公式。
```

这比单纯问“Embedding 是什么”效果更好。

### 3. 明确输出格式

如果你希望输出可以直接使用，就要写清楚格式。

例如：

```text
请用 Markdown 输出，结构如下：

## 先说结论
## 核心解释
## 举例说明
## 常见误区
## 总结
```

如果是程序处理，最好要求 JSON：

```text
请只输出 JSON，不要输出额外解释。

{
  "title": string,
  "tags": string[],
  "summary": string
}
```

### 4. 给示例

> 当你发现模型输出不稳定时，示例比解释更有效。

例如你想要某种摘要风格，不要只说“简洁一点”，可以给一个样例：

```text
示例：
输入：Prompt 是大模型的输入指令。
输出：Prompt 是人类给模型的任务说明，用来控制模型理解什么、怎么做、怎么输出。
```

模型很擅长模仿模式。

### 5. 写清楚不要做什么

> 很多时候，Prompt 不只是告诉模型做什么，还要告诉它不要做什么。

例如：

```text
如果资料中没有答案，请直接说“资料不足，无法确定”，不要根据常识补充。
```

这对 RAG、客服、法律、医疗、金融等场景尤其重要。

### 6. 把复杂任务拆小

> 不要把所有事情塞进一句话：

```text
帮我学习大模型并做一个项目。
```

更好的方式是拆成步骤：

```text
请按下面步骤帮我：
1. 先列出需要学习的知识点。
2. 再按优先级排序。
3. 每个知识点说明为什么重要。
4. 最后给出第一周学习计划。
```

模型处理结构化步骤，通常比处理模糊大任务更稳定。

### 7. 指定评价标准

如果你让模型生成内容，最好告诉它什么叫“好”。

例如：

```text
好的答案应该满足：
1. 初学者能看懂。
2. 有具体例子。
3. 不使用空泛概念。
4. 能和实际项目联系起来。
```

这会让模型知道要朝哪个方向优化。

## Prompt 的常用模板

### 概念解释类

```text
请解释「主题」。

要求：
1. 先用一句话给出结论。
2. 用通俗语言解释。
3. 举一个生活例子或项目例子。
4. 拆掉常见误区。
5. 最后用自己的话总结。

读者背景：
我是前端开发者，正在学习 AI 工程。
```

### 对比分析类

```text
请对比 A 和 B。

从以下维度分析：
1. 本质区别
2. 解决的问题
3. 使用场景
4. 优缺点
5. 在真实项目中如何选择

输出：
先给结论，再用表格对比，最后给选择建议。
```

### 代码审查类

```text
请审查下面的代码。

重点关注：
1. 是否有 bug。
2. 是否有边界情况没处理。
3. 是否有类型或异步问题。
4. 是否符合当前项目风格。

输出格式：
- 问题位置
- 问题原因
- 修改建议
- 风险等级
```

### RAG 问答类

```text
你是一个知识库问答助手。

规则：
1. 只能基于给定资料回答。
2. 如果资料不足，请明确说明资料不足。
3. 不要编造来源。
4. 回答后列出引用片段。

用户问题：
{question}

检索资料：
{context}
```

## 从更高层面理解 Prompt

如果只把 Prompt 理解成“提示词技巧”，很容易停留在表面。

更高层面看，Prompt 至少有四种身份。

### 1. Prompt 是人机协作协议

Prompt 定义了人类和模型之间如何协作。

```text
人类负责提出目标、边界和判断标准。
模型负责理解、生成、推理和执行部分步骤。
```

Prompt 写得越清楚，协作成本越低。

### 2. Prompt 是自然语言接口

传统程序的接口是函数：

```ts
search(keyword: string): Result[]
```

大模型的接口可以是自然语言：

```text
请从这段文本中提取人物、时间、地点和事件。
```

所以 Prompt 可以看成一种新的 API：

```text
Natural Language API
```

它不像代码接口那么严格，但更灵活。  
也正因为灵活，所以需要格式约束、示例和校验。

### 3. Prompt 是上下文工程

模型看到的只有输入上下文。  
它不知道你的项目、文件、数据库和真实意图，除非你把这些信息放进上下文。

所以工程中的 Prompt 不只是手写的一段话，还包括：

- 系统规则；
- 用户问题；
- 历史对话；
- 检索到的资料；
- 工具列表；
- 输出格式；
- 安全边界。

这就是为什么现在越来越多人说：

```text
Prompt Engineering 正在变成 Context Engineering。
```

真正重要的是：把正确的信息，在正确的时机，以正确的结构，放到模型面前。

### 4. Prompt 是软件系统的一部分

在真实项目中，Prompt 不应该散落在代码里随便写。

它应该像代码一样被管理：

- 有版本；
- 有模板；
- 有变量；
- 有测试集；
- 有评估指标；
- 有回滚机制。

例如一个 RAG 项目里，Prompt 改一句话，可能会影响：

- 回答准确率；
- 引用来源；
- 拒答率；
- 输出格式；
- token 成本；
- 响应延迟。

所以 Prompt 不是文案，而是系统行为的一部分。

## 好 Prompt 的判断标准

一个好 Prompt 不只是“看起来专业”，而是能稳定产生好结果。

可以用这几个标准判断：

| 标准     | 问题                   |
| -------- | ---------------------- |
| 明确性   | 模型是否知道要做什么   |
| 约束性   | 模型是否知道不能做什么 |
| 可执行性 | 任务是否能被一步步完成 |
| 可验证性 | 输出是否能被检查对错   |
| 稳定性   | 多次运行是否结果一致   |
| 可复用性 | 换一个输入是否仍然有效 |
| 可维护性 | 后续是否容易修改和调试 |

如果一个 Prompt 每次都要靠运气，那它还不是一个可靠的工程方案。

## 常见误区

### 误区一：Prompt 越长越好

不是。  
Prompt 越长，模型越可能忽略重点，也会增加成本。

好的 Prompt 应该是：

```text
信息足够，但不啰嗦。
```

### 误区二：只要角色设定够强，答案就会变好

“你是一个顶级专家”只能提供方向，不能替代任务说明。

更重要的是：

- 具体任务；
- 背景资料；
- 输出格式；
- 判断标准。

### 误区三：Prompt 可以彻底解决幻觉

Prompt 可以降低幻觉，但不能彻底消除幻觉。

如果任务需要事实准确，应该结合：

- RAG；
- 引用来源；
- 工具调用；
- 结果校验；
- 人工审核。

### 误区四：Prompt 写完就不用改了

Prompt 需要像代码一样迭代。

每次发现输出不稳定，都应该问：

```text
是任务没说清？
是资料不够？
是格式约束不够？
是示例不够？
还是模型能力不足？
```

## 用自己的话总结

Prompt 的本质不是“咒语”，而是：

```text
把人的意图组织成模型可执行的上下文。
```

早期 Prompt 是机器提示人输入。  
大模型时代，Prompt 变成了人提示模型完成任务。  
再往后，Prompt 不只是提示词，而是上下文工程、任务协议和 AI 系统设计的一部分。

写好 Prompt 的关键，不是学一堆固定话术，而是想清楚：

```text
我要模型完成什么任务？
模型需要哪些信息？
模型应该遵守哪些边界？
输出结果如何被使用和验证？
```

能回答清楚这些问题，Prompt 就不再是玄学，而是一种可以设计、测试和优化的工程能力。
