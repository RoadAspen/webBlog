# Prompt

> Prompt 通常翻译成“提示词”或“提示”。  
> 在大模型应用里，它不是咒语，而是人类和模型之间的任务接口。

## 先说结论

Prompt 可以理解为：

```text
把任务、背景、规则、示例和输出要求，
组织成模型能理解的输入。
```

从工程角度看，Prompt 更像：

```text
需求说明书 + 上下文 + 执行规则 + 输出协议
```

写 Prompt 的核心，不是把话说得漂亮，而是让模型知道：

- 要做什么；
- 基于什么资料做；
- 不能做什么；
- 按什么格式输出；
- 不确定时怎么处理。

## Prompt 什么时候出现的

Prompt 这个词并不是大模型时代才出现。

在传统计算机里，Prompt 指“系统提示用户输入”。比如：

```text
C:\>
请输入用户名：
```

那时是：

```text
机器提示人输入。
```

到了大模型时代，方向反过来了：

```text
人提示模型完成任务。
```

可以按三个阶段理解：

| 阶段 | Prompt 的含义 | 典型场景 |
| --- | --- | --- |
| 传统计算机时代 | 系统提示用户输入 | 命令行、表单 |
| 早期 NLP 时代 | 用模板引导模型完成任务 | 分类、填空、问答 |
| 大模型时代 | 用自然语言描述复杂任务 | ChatGPT、Agent、RAG |

GPT-3 之后，研究者发现模型可以通过上下文里的任务说明和少量示例临时学会任务，这叫：

```text
In-context Learning
```

ChatGPT 普及后，Prompt 从研究概念变成普通用户也能感知的能力。

## Prompt 的演进

Prompt 的演进，本质上是人类和模型协作方式的演进。

### 1. 关键词式

像搜索引擎一样输入：

```text
RAG 是什么
JavaScript 闭包
大模型厂商
```

简单，但任务不清楚，模型容易自由发挥。

### 2. 指令式

给模型明确任务：

```text
请用通俗语言解释 RAG 是什么，并举一个例子。
```

它说明了任务、风格和输出要求，比关键词式更稳定。

### 3. 角色式

给模型设定回答视角：

```text
你是一名资深 AI 工程师，请用适合前端开发者理解的方式解释 RAG。
```

角色设定能帮助模型确定语气、深度和关注点，但不能替代任务、背景和输出格式。

### 4. 结构化

复杂任务最好结构化：

```text
背景：
我正在学习 AI 工程。

任务：
解释 Prompt 的含义。

要求：
1. 先说结论。
2. 举例说明。
3. 拆掉常见误区。

输出格式：
Markdown。
```

结构化 Prompt 更容易复用和调试。

### 5. Few-shot

Few-shot 是给模型几个示例，让它模仿格式和判断标准：

```text
示例：
输入：明天下午三点提醒我开会
输出：{"task":"开会","time":"明天下午三点"}

现在处理：
输入：今晚八点提醒我复习 Prompt
输出：
```

适合分类、信息抽取、格式转换。

### 6. 工具调用和系统级 Prompt

进入 Agent / Tool Calling 阶段后，Prompt 不只是让模型回答，还要指导模型何时调用工具。

在真实产品里，Prompt 往往是一套上下文：

```text
System Prompt
+ 用户输入
+ 历史对话
+ 检索资料
+ 工具说明
+ 输出格式
+ 安全规则
```

所以 Prompt 正在从“提示词技巧”变成“上下文工程”。

## 如何写好 Prompt

可以记住一个公式：

```text
角色 + 背景 + 任务 + 约束 + 示例 + 输出格式 + 失败处理
```

不一定每次都全部写，但复杂任务最好尽量明确。

### 1. 任务要具体

不要只说：

```text
帮我看看这个。
```

更好的写法：

```text
请检查这段代码是否有潜在 bug，重点关注异步请求、空值处理和类型错误。
```

### 2. 背景要足够

模型不知道你的真实处境，必要背景要直接告诉它：

```text
我是前端开发者，正在学习 AI 工程。
请用我能理解的方式解释 Embedding，不要使用太多数学公式。
```

### 3. 输出格式要明确

如果结果要被程序处理，最好直接要求结构化输出：

```text
请只输出 JSON：

{
  "title": string,
  "tags": string[],
  "summary": string
}
```

### 4. 约束要写清楚

尤其是事实类、知识库类任务：

```text
如果资料中没有答案，请说“资料不足，无法确定”，不要根据常识补充。
```

### 5. 复杂任务要拆步骤

不要把大任务塞进一句话。更好的方式是：

```text
请按步骤完成：
1. 先列出关键知识点。
2. 再按优先级排序。
3. 每个知识点说明为什么重要。
4. 最后给出第一周学习计划。
```

## 常用模板

### RAG 问答类

```text
你是一个知识库问答助手。

规则：
1. 只能基于给定资料回答。
2. 资料不足时明确说明。
3. 不要编造来源。
4. 回答后列出引用片段。

用户问题：
{question}

检索资料：
{context}
```

## 更高层理解 Prompt

Prompt 不只是“提问技巧”，它至少有三层含义。

### 1. Prompt 是人机协作协议

人类负责提出目标、边界和判断标准；模型负责理解、生成、推理和执行部分步骤。

Prompt 写得越清楚，协作成本越低。

### 2. Prompt 是自然语言接口

传统程序接口像这样：

```ts
search(keyword: string): Result[]
```

大模型接口可以是：

```text
请从这段文本中提取人物、时间、地点和事件。
```

它更灵活，但也更需要格式约束和结果校验。

### 3. Prompt 是系统行为的一部分

在 AI 应用里，Prompt 改一句话，可能影响：

- 回答准确率；
- 输出格式；
- 拒答率；
- token 成本；
- 响应延迟。

所以 Prompt 应该像代码一样被管理：有模板、有版本、有测试、有回滚。

## 常见误区

### 误区一：Prompt 越长越好

不是。Prompt 太长会增加成本，也可能让模型忽略重点。  
好的 Prompt 是“信息足够，但不啰嗦”。

### 误区二：角色设定能解决一切

“你是顶级专家”只能提供方向，不能替代任务说明、背景资料和输出格式。

### 误区三：Prompt 可以彻底解决幻觉

Prompt 可以降低幻觉，但不能消除幻觉。事实类任务还需要 RAG、引用来源、工具调用和结果校验。

## 用自己的话总结

Prompt 的本质不是咒语，而是：

```text
把人的意图组织成模型可执行的上下文。
```

早期 Prompt 是机器提示人输入。  
大模型时代，Prompt 变成人提示模型完成任务。  
再往后，Prompt 不只是提示词，而是上下文工程和 AI 系统设计的一部分。
