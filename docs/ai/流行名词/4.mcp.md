# MCP（Model Context Protocol）

> MCP 全称 Model Context Protocol，中文常译作"模型上下文协议"。  
> 它是一个开放协议，定义了 LLM 如何与外部工具、数据源进行标准化通信。

## 一、先说结论

MCP 的本质可以一句话概括：

```text
MCP 是 LLM 的"USB-C 接口"——它规定了一套通用标准，
让任何模型都能用同一种方式对接任何工具和数据源，
而不需要为每一对"模型 + 工具"组合单独写胶水代码。
```

类比：

```text
没有 MCP 之前：每个模型 × 每个工具 = N × M 种对接方式（像每种手机都有专属充电口）
有了 MCP 之后：模型 ← MCP → 工具       （像大家都用 USB-C，一根线全搞定）
```

你不是在学一个新框架，而是在理解一个**通信协议**。它不替代 Function Calling，而是让 Function Calling 的实现变得标准化、可复用。

## 二、MCP 的核心能力

### 它解决了什么问题

在 MCP 出现之前，让 LLM 调用外部工具有几个痛点：

| 痛点                 | 具体表现                                                                      |
| -------------------- | ----------------------------------------------------------------------------- |
| 每个工具都要单独集成 | 接 Google Drive 写一套代码，接 GitHub 又写一套，接数据库再写一套              |
| 模型和工具强绑定     | 换一个模型（如从 Claude 换到 GPT），所有工具集成的代码可能都要改              |
| 工具生态碎片化       | 社区写了一个"搜索本地文件"的工具，你大概率没法直接用在自己的项目里            |
| 上下文注入不统一     | 有的工具返回 JSON，有的返回纯文本，有的返回二进制——模型收到的信息格式五花八门 |

MCP 的解法很直接：**把"模型怎么调用工具"这件事抽象成一个标准协议**。工具开发者按协议写 Server，模型厂商按协议对接 Client，两边互不关心对方内部细节。

### 如果不理解它，容易产生什么误解

很多人第一次听到 MCP 会以为它是：

- ❌ "又一个 Agent 框架" —— 不是。MCP 是协议，不是框架。
- ❌ "Anthropic 私有的东西" —— 不是。它是开放的，其他厂商也在跟进。
- ❌ "Function Calling 的替代品" —— 不是。它和 Function Calling 是不同层级的东西（详见第五节）。

### 它和 AI 应用开发的关系

如果你在做 RAG、Agent 或任何需要 LLM 和外部世界交互的系统，MCP 影响的是：

```text
你怎么让模型获取"外部上下文"和"执行外部动作"。
```

在你的 RagFlow 项目里，MCP 可能对应的是：检索模块如何暴露接口、工具模块如何注册、模型如何统一调度它们。

## 三、举一个直观例子

### 场景：让 LLM 帮你查 GitHub Issue 并创建文件

**没有 MCP 时，你需要做的事：**

```text
1. 写代码调用 GitHub API，拿到 issues 数据
2. 把数据塞进 Prompt 上下文
3. 模型分析完后，再写代码解析模型的输出
4. 根据解析结果，写代码调用文件系统 API 创建文件
```

每一步都是你手写胶水代码。换个数据源（比如从 GitHub 换成 Jira），第 1 步和第 4 步基本要重写。

**有 MCP 时，事情变成：**

```text
1. 启动一个 GitHub MCP Server（别人写好的，直接 npm install 就行）
2. 启动一个 Filesystem MCP Server（也是现成的）
3. 你的 MCP Client 把两个 Server 的能力列表一起发给模型
4. 模型自己决定：先调 GitHub 工具拿 issues，再调文件系统工具写文件
```

你的代码量从"写一堆胶水"变成了"配置两个 Server"。

用 TypeScript 表达这个对比：

```ts
// ===== 没有 MCP：手写胶水代码 =====

// 每种工具要单独实现
async function getGitHubIssues(repo: string) {
	const res = await fetch(`https://api.github.com/repos/${repo}/issues`, {
		headers: { Authorization: `Bearer ${GITHUB_TOKEN}` },
	});
	return res.json();
}

async function createFile(path: string, content: string) {
	await fs.writeFile(path, content);
}

// 还要自己把工具描述注入 Prompt
const prompt = `
可用工具：
- getGitHubIssues(repo: string): 获取指定仓库的 issues
- createFile(path: string, content: string): 创建文件

用户目标：帮我总结 ${repo} 的 issues 并保存到 summary.md
`;
// ... 然后手动解析模型输出、手动调用工具、手动拼接结果 ...

// ===== 有 MCP：配置式对接 =====

// 启动现成的 MCP Server（通常一行命令或几行配置）
// npx @anthropic/create-mcp-server github --token $GITHUB_TOKEN
// npx @anthropic/create-mcp-server filesystem --path /workspace

// Client 端自动发现所有 Server 提供的工具
const tools = await mcpClient.listTools();
// tools = [
//   { name: "search_issues", server: "github", ... },
//   { name: "write_file", server: "filesystem", ... },
//   { name: "read_file", server: "filesystem", ... },
//   ...
// ]

// 把工具列表交给模型，模型自己决定调用哪个
// 调用结果通过 MCP 协议标准化返回，无需手动解析
```

核心差别：MCP 把"怎么调用"和"怎么返回"标准化了，你只关心"用哪个工具"。

## 四、本质是什么

### 架构

MCP 定义了三层角色：

```text
┌──────────────────────────────────────────┐
│              MCP Host（宿主应用）          │
│  比如：Claude Desktop、VS Code、你的 App   │
│                                            │
│  ┌──────────────────────────────────────┐ │
│  │       MCP Client（协议客户端）        │ │
│  │  负责：发现工具、发送请求、接收结果    │ │
│  │  一个 Host 可以连接多个 Server        │ │
│  └──────────┬───────────────┬───────────┘ │
│             │               │              │
└─────────────┼───────────────┼──────────────┘
              │               │
    ┌─────────▼─────┐  ┌──────▼────────┐
    │ MCP Server A  │  │ MCP Server B  │
    │ (文件系统)     │  │ (数据库)       │
    │               │  │               │
    │ • 读文件      │  │ • 执行 SQL    │
    │ • 写文件      │  │ • 查表结构    │
    │ • 列目录      │  │ • 导出数据    │
    └───────────────┘  └───────────────┘
```

### 协议的核心构成

MCP 协议主要定义了三类通信：

| 通信类型                | 做什么                               | 举例                          |
| ----------------------- | ------------------------------------ | ----------------------------- |
| **Tools（工具）**       | 模型调用外部能力，执行动作           | 发邮件、创建文件、查询数据库  |
| **Resources（资源）**   | 模型读取外部数据，获取上下文         | 读文档、拿日志、获取当前时间  |
| **Prompts（提示模板）** | 预定义的 Prompt 模板，用户可快速触发 | `/review-pr` `/summarize-doc` |

再加上一套生命周期管理：Server 如何注册、Client 如何发现、连接如何保活。

### 通信方式

MCP 支持两种传输方式：

| 传输方式                             | 适用场景     | 特点                                                       |
| ------------------------------------ | ------------ | ---------------------------------------------------------- |
| **stdio**（标准输入输出）            | 本地进程通信 | 简单，Server 作为子进程启动，通过 stdin/stdout 传 JSON-RPC |
| **HTTP + SSE**（Server-Sent Events） | 远程服务通信 | Server 可以跑在另一台机器上，适合生产环境和服务化部署      |

底层消息格式是 **JSON-RPC 2.0**，这意味着：

```text
MCP 本身不绑定任何编程语言、任何模型、任何传输层。
它只是一套约定："请求长这样，响应长那样"。
```

### 一次完整的工具调用流程

```text
1. Client 启动 → 连接所有已配置的 MCP Server
2. Client 向每个 Server 请求能力列表：你有哪些工具/资源/Prompt？
3. Client 把所有 Server 的能力汇总，连同用户问题一起发给 LLM
4. LLM 决定调用某个工具，返回 { toolName, arguments }
5. Client 找到对应的 MCP Server，通过 JSON-RPC 发出调用请求
6. Server 执行工具 → 返回标准化结果
7. Client 把结果注入对话上下文 → LLM 继续推理
8. 重复步骤 4-7，直到 LLM 给出最终回答
```

用伪代码表达：

```ts
// MCP 的一次工具调用本质就是一次 JSON-RPC 往返
const request = {
	jsonrpc: '2.0',
	method: 'tools/call',
	params: {
		name: 'search_issues',
		arguments: { repo: 'my-org/my-project', label: 'bug' },
	},
	id: 1,
};

const response = {
	jsonrpc: '2.0',
	result: {
		content: [{ type: 'text', text: '#42: 登录页面崩溃\n#58: 图表不显示' }],
	},
	id: 1,
};
```

这就是 MCP 的本质：**一个基于 JSON-RPC 的、专门为 LLM 和外部世界通信设计的标准协议。**

## 五、和相近概念的区别

### MCP vs Function Calling

这是一个最容易混淆的点。先说结论：

```text
Function Calling 是"模型层面的能力"：模型可以输出"我想调这个函数"。
MCP 是"协议层面的标准"：定义了工具怎么被发现、描述、调用和返回结果。
```

| 维度         | Function Calling                                           | MCP                              |
| ------------ | ---------------------------------------------------------- | -------------------------------- |
| 层级         | 模型能力                                                   | 通信协议                         |
| 定义者       | 每个模型厂商各自定义                                       | Anthropic 提出，开源协议         |
| 工具描述格式 | 每个厂商不同（OpenAI 用 JSON Schema，Claude 用 XML，等等） | 统一的 JSON Schema               |
| 工具发现     | 你手动把函数定义传给模型                                   | Server 自动上报，Client 自动发现 |
| 可复用性     | 换模型可能要改工具定义格式                                 | 一次写 Server，到处复用          |
| 类比         | 模型"会点菜"                                               | 餐厅的"标准菜单和出餐流程"       |

更直观的理解：

```text
Function Calling = 模型的大脑有能力决定"我要调一个函数"
MCP            = 定义了一套标准，让"函数是什么、怎么调、返回什么"
                 这件事统一起来，不再每个厂商各搞一套
```

它们是互补关系，不是替代关系。MCP 让 Function Calling 的实现从"每个项目手写"变成"按协议配置"。

### MCP vs REST API

| 维度       | REST API                                 | MCP                                |
| ---------- | ---------------------------------------- | ---------------------------------- |
| 设计目标   | 通用的 Web 服务通信                      | 专门为 LLM 和工具通信设计          |
| 接口描述   | 靠文档或 OpenAPI Spec，模型不易理解      | 自带结构化的工具描述，模型直接可读 |
| 数据格式   | 五花八门（JSON / XML / 纯文本 / 二进制） | 统一（JSON-RPC + 结构化内容类型）  |
| 发现机制   | 没有标准（要人工看文档或 OpenAPI）       | 内置：Client 连接即自动发现        |
| 上下文注入 | 需要手动把 API 返回拼进 Prompt           | 工具返回自动注入对话上下文         |

一句话：**REST API 是给人设计的，MCP 是给 LLM 设计的。** 你可以把已有的 REST API 包装成 MCP Server，让 LLM 能直接调用它。

### MCP vs Agent Framework（如 LangChain、CrewAI）

| 维度   | Agent Framework                           | MCP                      |
| ------ | ----------------------------------------- | ------------------------ |
| 层级   | 应用框架                                  | 通信协议                 |
| 做什么 | 定义 Agent 的编排逻辑、记忆管理、任务调度 | 定义模型和工具的通信标准 |
| 关系   | 框架**可以使用** MCP 作为工具接入标准     | MCP 不定义任何编排逻辑   |

可以这样理解：

```text
Agent Framework（LangChain 等）
  ├── 任务编排：先做什么、后做什么
  ├── 记忆管理：记住哪些对话
  ├── 推理循环：Plan → Act → Observe → Reflect
  └── 工具接入 ← 这里可以用 MCP！← MCP Server A / B / C
```

## 六、对工程实践的影响

### 对 Prompt 设计的影响

当你使用 MCP 时，工具的发现和描述是自动的，不再需要手动在 Prompt 里写"可用工具列表"。但你需要更关注：

- **系统级 Prompt 里怎么约束工具的权限边界**（哪些操作允许、哪些要确认）
- **输出格式约束**：当模型同时有多个 MCP Server 可用时，要明确优先级

### 对 RAG 检索的影响

MCP 的 Resources 概念天然适合 RAG：

```text
传统 RAG：手动拼接检索结果 → 注入 Prompt
MCP RAG：知识库作为一个 MCP Resource Server → 模型按需拉取
```

这不是替代 RAG，而是让检索变成一个标准化的"上下文源"，模型可以主动决定"我需要查什么"，而不只是被动接收你塞进去的资料。

### 对你 RagFlow 项目的启发

在你的 RagFlow 项目中，如果引入 MCP 思想，可能会变成：

| 模块        | MCP 对应                            |
| ----------- | ----------------------------------- |
| 文档检索    | MCP Resource Server（暴露检索能力） |
| 工具调用    | MCP Tool Server（暴露执行能力）     |
| Prompt 模板 | MCP Prompt（暴露常用任务模板）      |
| 模型路由    | MCP Client（统一发现和调度）        |

这样做的好处：以后你想加一个新的数据源（比如接 Slack、接数据库），只需要新增一个 MCP Server，不需要改核心代码。

### 对成本、稳定性的影响

| 维度         | 影响                                                                                                                   |
| ------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **成本**     | 工具调用是额外一轮 LLM 推理，每次调工具都消耗 token。MCP 不改变成本结构，但让成本更可控（Server 的返回可以裁剪和截断） |
| **延迟**     | 加了 Server 通信的额外开销。本地 stdio 模式几乎无延迟，远程 HTTP 模式取决于网络                                        |
| **稳定性**   | 标准化降低了集成出错的概率。但 Server 挂了，模型就少了一类能力                                                         |
| **幻觉治理** | 工具返回的标准化内容比自由文本更容易被模型正确解读，间接降低幻觉                                                       |

### 选型建议

```text
什么时候值得引入 MCP：
✅ 你的系统需要对接 ≥3 个外部工具/数据源
✅ 你想让工具定义和服务逻辑可复用
✅ 你可能需要切换不同模型厂商
✅ 你在设计一个需要长期维护的 AI 应用

什么时候先不引入：
❌ 只有一个工具，简单 Function Calling 就够了
❌ 还在原型阶段，快速验证比标准化更重要
❌ 团队对协议层还不熟悉，引入会增加认知负担
```

## 七、常见误区

### 误区一：MCP 是 Anthropic 的私有协议

**错误理解：**
MCP 是 Claude 专用的，其他模型不能用。

**正确理解：**
MCP 是 Anthropic 提出并开源的开放协议，底层是 JSON-RPC 2.0。任何模型厂商只要按协议实现 Client，就能对接所有 MCP Server。已经有社区实现让 GPT、Gemini、本地模型通过 MCP 调用工具。

---

### 误区二：MCP 替代了 Function Calling

**错误理解：**
有了 MCP 就不需要 Function Calling 了。

**正确理解：**
MCP 和 Function Calling 是不同层级的东西。Function Calling 是模型的输出能力（"我要调一个函数"），MCP 是工具系统的接入标准（"这些函数怎么统一管理和调用"）。MCP 做的是把 Function Calling 的**工具端**标准化，而不是替代模型端的决策能力。

---

### 误区三：MCP Server 一定要自己写

**错误理解：**
每个工具都要自己开发一个 MCP Server，门槛很高。

**正确理解：**
社区和厂商已经在维护大量现成的 MCP Server。GitHub、Slack、Postgres、Google Drive、文件系统、浏览器自动化等常见场景都有开箱即用的 Server。你的主要工作是**配置和组合**，不是从零开发。

---

### 误区四：用了 MCP 就不需要 Agent 框架了

**错误理解：**
MCP 能管理工具，所以不需要 Agent 编排了。

**正确理解：**
MCP 只解决"怎么接入工具"，不解决"Agent 的思考和执行逻辑"。Agent 框架决定什么时候调工具、调哪个、调完怎么继续——这些仍然是 Agent 框架或你自己代码的责任。MCP 让你的 Agent 接入工具更优雅，但不会替你写 Agent 逻辑。

## 八、用自己的话总结

MCP 不是框架、不是模型、不是新概念——它就是一个**标准化协议**。

```text
类比：HTTP 标准化了"浏览器怎么和服务器通信"，
     MCP 标准化了"LLM 怎么和工具通信"。
```

如果要用一句话记住它：

```text
MCP = LLM 世界的 USB-C / HTTP：
      让任何模型用同一套方式接入任何工具和数据源。
```

对前端开发者来说，你可以把它理解为：

```text
MCP 之于 LLM 工具调用，
就像 RESTful API 规范之于前后端通信。

以前是每个后端自己定 API 格式（有的用 REST、有的用 RPC、有的随便写），
前端对接 N 个后端要学 N 种风格；
后来大家统一用 RESTful，前端只需要知道"GET /users"是查、"POST /users"是增。

MCP 在 LLM 领域做的事，就是这场"统一化"。
```

最后记住一个关键关系链：

```text
LLM（大脑，负责思考）
  ↕ 通过 Function Calling 表达"我想做什么"
  ↕ 通过 MCP 标准化"工具怎么接入、怎么调、怎么返回"
Tool（手脚，负责执行）
```

理解了这个关系，你就不会被"MCP 又是一个新名词"绕晕——它是一次值得关注的工程标准化，而不是又一个需要从头学的范式革命。
