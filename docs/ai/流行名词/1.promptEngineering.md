# Prompt Engineering

> Prompt Engineering 通常翻译成“提示词工程”。  
> 它关注的不是“写一句神奇提示词”，而是如何系统地设计 Prompt，让模型稳定完成任务。

## 先说结论

Prompt Engineering 可以理解为：

```text
设计、测试、优化 Prompt 的方法规范。
```

如果说 Prompt 是一次具体输入，那么 Prompt Engineering 就是设计这类输入的方法论。

更直白地说：

```text
普通 Prompt：我怎么问模型。
Prompt Engineering：我如何设计一套可靠的提问方式，让模型稳定完成任务。
```

## 它和 Prompt 的区别

| 概念                | 含义                           | 例子                                                 |
| ------------------- | ------------------------------ | ---------------------------------------------------- |
| Prompt              | 给模型的一次具体提示           | “请解释 RAG 是什么”                                  |
| Prompt Template     | 可复用的提示模板               | “请解释 {concept}，要求先说结论、再举例”             |
| Prompt Engineering  | 设计、测试、优化 Prompt 的过程 | 调整任务、示例、格式、约束，让输出更稳定             |
| Context Engineering | 更大范围的上下文设计           | 组织系统规则、用户问题、历史对话、RAG 资料、工具说明 |

Prompt Engineering 解决的不是“怎么把话说得好听”，而是“怎么让模型更可靠”。

## 为什么需要 Prompt Engineering

大模型不是传统函数。

传统函数通常是：

```text
输入确定 -> 输出确定
```

大模型更像：

```text
给它一段上下文 -> 它基于概率生成最可能的后续内容
```

这会带来几个问题：

- 同一个问题可能有不同回答；
- 任务描述不清时，模型会自己脑补；
- 没有边界时，模型容易编造；
- 格式不清时，输出不可控；
- 复杂任务容易漏步骤。

Prompt Engineering 的作用，就是把模糊需求变成清晰的模型输入：

```text
把“我想要什么”
翻译成
“模型应该如何执行”
```

## Prompt Engineering 包含什么

### 1. 任务定义

先明确模型到底要完成什么。

不好的写法：

```text
帮我优化一下。
```

更好的写法：

```text
请优化下面这段文章的表达，让它更适合技术博客读者阅读。
要求保留原意，不要增加新观点。
```

### 2. 背景补充

模型不知道你的真实上下文，需要补充必要背景：

```text
我是前端开发者，正在学习 AI 工程。
请用我已有的前端经验类比解释 Agent。
```

### 3. 约束设计

告诉模型不能做什么：

```text
如果资料中没有答案，请直接说“资料不足”，不要根据常识补充。
```

### 4. 输出格式设计

让模型按固定结构输出，方便阅读或程序处理：

```text
请只输出 JSON：

{
  "title": string,
  "summary": string,
  "tags": string[]
}
```

### 5. 示例设计

示例能告诉模型“什么是我想要的结果”：

```text
输入：Prompt 是什么？
输出：Prompt 是人类给模型的任务说明，用来控制模型理解什么、怎么做、怎么输出。
```

### 6. 评估和迭代

Prompt 写完后不能只看一次结果。  
应该准备几组测试输入，观察：

- 是否正确完成任务；
- 输出格式是否稳定；
- 有没有幻觉；
- 有没有漏掉边界情况；
- 换一个模型是否仍然可用。

这一步才真正体现“工程”两个字。

## 它和写作文案的区别

Prompt Engineering 看起来像写文字，但本质更接近设计接口。

写作文案关注：

```text
表达是否优美
读者是否喜欢
语气是否自然
```

Prompt Engineering 关注：

```text
模型是否理解
任务是否可执行
输出是否稳定
结果是否可验证
失败是否可处理
```

所以好的 Prompt 不一定华丽，但一定清楚、可执行、可测试。

## Prompt Engineering 的目标

它的目标不是写出万能提示词，而是让模型输出更可控、更稳定、更符合任务需求。

| 目标     | 含义                         |
| -------- | ---------------------------- |
| 准确性   | 回答更接近事实和任务要求     |
| 稳定性   | 多次运行结果风格和格式更一致 |
| 可控性   | 能控制语气、长度、结构、边界 |
| 可解析性 | 输出能被程序继续处理         |
| 可复用性 | 同一模板能处理同类任务       |
| 可评估性 | 能判断输出好不好             |
| 可维护性 | Prompt 变化后容易测试和回滚  |

## Prompt Engineering 的局限

Prompt Engineering 很重要，但它不是万能的。

它不能完全解决：

- 模型本身能力不足；
- 资料缺失；
- 训练知识过时；
- 工具结果错误；
- 长上下文遗忘；
- 高风险场景中的事实校验。

真实系统里，它通常要和其他工程手段配合：

- RAG：补充外部知识；
- Tool Calling：调用真实工具；
- Structured Output：约束输出格式；
- Guardrails：做安全和格式校验；
- Evaluation：用测试集评估效果；
- Human Review：高风险结果人工确认。

## 用自己的话总结

Prompt Engineering 是把模型从“会聊天”推向“能稳定完成任务”的第一层工程化手段。

它可以是手工方法，也可以由程序动态组装和执行。  
程序不是必须的，但在真实 AI 应用里，Prompt 往往会被模板、变量、RAG 资料和工具说明自动拼接出来。
