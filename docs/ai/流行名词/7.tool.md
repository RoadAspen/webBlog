# Tool（工具）

> Tool 在大模型语境里不是指你用的 VS Code 或 Postman 这类软件，而是指**模型可以调用的外部函数或能力**。  
> 它是 LLM 从"只能说话"变成"能做事"的关键桥梁。

## 一、先说结论

Tool 的本质可以这样理解：

```text
Tool = 一个带"说明书"的函数，模型读了说明书后，
       自己决定什么时候调用它、传什么参数，
       然后根据返回结果继续推理。
```

类比：

```text
没有 Tool 的 LLM：一个只能口头给你建议的顾问
有 Tool 的 LLM：  一个可以亲自去查资料、算数字、发邮件的助手
```

Tool 不是一个新技术，而是一种**约定**——你用模型能理解的格式描述一个函数（名称、用途、参数类型），模型就能在需要时"请求调用"它。真正执行这个函数的还是你的代码，模型只负责"决策"。

## 二、tool 的核心能力

### 它解决了什么问题

LLM 有两大天然短板：

| 短板                 | 表现                                             | Tool 怎么解决                                     |
| -------------------- | ------------------------------------------------ | ------------------------------------------------- |
| **无法获取实时信息** | 训练数据有截止日期，不知道今天的天气、股价、新闻 | 给一个 `search_web` Tool，模型需要时就调          |
| **无法执行精确操作** | 不擅长精确计算、不擅长操作外部系统               | 给 `calculator`、`send_email`、`query_db` 等 Tool |
| **幻觉**             | 没有真实数据时会编造                             | Tool 返回的是真实数据，不是模型生成的             |
| **无法持久化**       | 聊完就没了，不能写文件、改数据库                 | 给 `write_file`、`update_record` 等 Tool          |

本质上是承认：**LLM 擅长推理，但不擅长所有事。** Tool 让模型把不擅长的事"外包"出去。

### 如果不理解它，容易产生什么误解

- ❌ "Tool 是模型自带的能力" —— 不是。Tool 是你给的外部函数，模型只负责决策调用哪个。
- ❌ "Tool 调用是模型在执行代码" —— 不是。模型只是输出"我想调 `search`，参数是 `xxx`"，你的程序收到后自己去调。
- ❌ "Tool 和 Function Calling 是同一个东西" —— 不完全。Tool 是你定义的函数，Function Calling 是模型输出调用指令的能力。详见第五节。

### 它在整个 AI 技术栈里的位置

```text
Agent（任务执行系统）
  └── 需要 Tool 才能做事

MCP（模型上下文协议）
  └── 标准化 Tool 的接入方式

RAG（检索增强生成）
  └── 检索本身就可以封装成一个 Tool

Prompt（提示词）
  └── 需要把 Tool 的描述注入 Prompt，模型才知道有哪些工具可用
```

可以说：**Tool 是 AI 应用从"对话式"走向"行动式"的基石。**

## 三、举一个直观例子

### 场景：做一个能查天气的聊天机器人

**需求**：用户问"北京今天天气怎么样？"，模型要给出真实天气。

模型自己不知道天气，你需要给它一个 Tool。

#### Step 1：定义 Tool（写说明书）

```ts
// 这是你定义的 Tool —— 本质就是一个带"说明书"的异步函数
const weatherTool = {
	// 标识
	name: 'get_weather',

	// 说明书：用自然语言告诉模型这工具是干嘛的
	description: '查询指定城市当天的天气情况，返回温度、湿度和天气描述',

	// 参数定义：用 JSON Schema 告诉模型需要传什么参数
	parameters: {
		type: 'object',
		properties: {
			city: {
				type: 'string',
				description: "城市名称，例如 '北京'、'上海'",
			},
		},
		required: ['city'],
	},

	// 实际执行函数：模型"请求调用"后，由你的代码执行
	async execute(params: { city: string }) {
		const response = await fetch(
			`https://api.weather.com/v1/current?city=${encodeURIComponent(
				params.city,
			)}`,
		);
		return response.json();
		// 返回例如：{ temperature: 28, humidity: "65%", description: "晴" }
	},
};
```

#### Step 2：把 Tool 说明书注入 Prompt

```ts
// 把 Tool 信息转换成模型能读的格式，放入 System Prompt
const systemPrompt = `
你是一个天气助手。你可以使用以下工具来回答用户的问题。

可用工具：
- 名称：${weatherTool.name}
- 描述：${weatherTool.description}
- 参数：${JSON.stringify(weatherTool.parameters, null, 2)}

当你需要查询天气时，请用以下 JSON 格式表示工具调用：
{
  "tool": "get_weather",
  "params": { "city": "城市名" }
}
`;
```

#### Step 3：执行完整调用循环

```ts
async function chat(userMessage: string) {
	// 1. 把用户消息 + System Prompt 发给模型
	const response = await callLLM(systemPrompt, userMessage);

	// 2. 检查模型是否想调 Tool
	if (response.toolCall) {
		// 3. 找到对应的 Tool 并执行
		const tool = findTool(response.toolCall.name); // → weatherTool
		const result = await tool.execute(response.toolCall.params);

		// 4. 把 Tool 执行结果塞回对话，让模型继续推理
		const finalResponse = await callLLM(
			systemPrompt,
			`${userMessage}\n工具返回结果：${JSON.stringify(
				result,
			)}\n请用自然语言回复用户。`,
		);

		return finalResponse.content;
	}

	// 模型不需要 Tool，直接返回文本
	return response.content;
}

// 用户问："北京今天天气怎么样？"
chat('北京今天天气怎么样？');
// → 模型输出 toolCall: { name: "get_weather", params: { city: "北京" } }
// → 你的代码执行 weatherTool.execute({ city: "北京" })
// → 得到真实天气数据
// → 模型最终回复："北京今天晴天，温度 28°C，湿度 65%。"
```

#### 用开发类比

如果你熟悉前后端通信，Tool 机制和这个很像：

```text
前端组件           →  后端 API
===========================================
Tool 的 parameters  →  API 的请求 body schema
Tool 的 execute()   →  API handler 的实现
Tool 的返回值        →  API 的 response
模型的 toolCall     →  前端的 fetch() 调用
```

区别只在于：前端是你手写 `fetch()` 代码，而 Tool 调用是**模型替你决定**什么时候发请求、传什么参数。

## 四、本质是什么

### Tool 的四个要素

一个标准的 Tool 必须包含：

| 要素                              | 作用                                        | 如果缺失会怎样                           |
| --------------------------------- | ------------------------------------------- | ---------------------------------------- |
| **name**（名称）                  | 唯一标识，模型用它来指定调用哪个            | 模型不知道该怎么指代                     |
| **description**（描述）           | 告诉模型这个工具什么时候用、做什么          | 模型不知道什么时候该调它，可能误用或不用 |
| **parameters**（参数定义）        | JSON Schema，告诉模型该传哪些参数、什么类型 | 模型可能乱填参数，导致调用失败           |
| **execute / handler**（执行函数） | 真正干活的代码                              | Tool 只是一个空壳，无法实际执行          |

### 一次 Tool 调用的完整流程

```text
用户输入
  │
  ▼
┌─────────────────────────────────────┐
│ 1. 组装上下文                       │
│    用户消息 + System Prompt          │
│    + Tool 说明书（name + desc +      │
│      parameters schema）             │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ 2. 模型推理                         │
│    - 判断：需要调 Tool 吗？          │
│    - 如果需要，选哪个 Tool？         │
│    - 传什么参数？                    │
│    - 输出：工具调用指令（structured）│
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ 3. 你的程序解析工具调用指令          │
│    - 提取 toolName                  │
│    - 校验 params 是否符合 schema     │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ 4. 执行 Tool                        │
│    findTool(name).execute(params)   │
│    → 返回结果（文本/JSON/二进制）    │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ 5. 把 Tool 结果注入上下文            │
│    role: "tool"                     │
│    content: 工具返回结果             │
│    → 再次发给模型                    │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│ 6. 模型继续推理                     │
│    - 根据 Tool 结果生成最终回答      │
│    - 或者再调另一个 Tool             │
│    - 循环直到任务完成                │
└──────────────┬──────────────────────┘
               ▼
            最终回复给用户
```

### Tool 参数为什么用 JSON Schema

```ts
// 不好的参数描述 —— 太模糊
const badTool = {
	parameters: 'string', // ❌ 模型不知道要传什么
};

// 好的参数描述 —— 用 JSON Schema，精确到类型和字段
const goodTool = {
	parameters: {
		type: 'object',
		properties: {
			city: {
				type: 'string',
				description: "城市名，支持中英文，例如 '北京' 或 'Beijing'",
			},
			date: {
				type: 'string',
				description: '查询日期，格式 YYYY-MM-DD。不填则默认今天',
			},
		},
		required: ['city'], // 必填字段
	},
};
```

JSON Schema 在这里的作用不是校验（虽然你也可以用它校验），而是**让模型精确理解每个参数的含义和约束**。它就像 TypeScript 的类型定义之于前端开发者——让调用方知道传什么、怎么传。

## 五、和相近概念的区别

### Tool vs Function Calling

这是最常被搞混的一组概念：

```text
Tool            = 你定义的、实实在在的函数（有 name、参数、执行逻辑）
Function Calling = 模型的能力：能看懂 Tool 的说明书，并输出"我要调这个 Tool"
```

| 维度     | Tool           | Function Calling           |
| -------- | -------------- | -------------------------- |
| 谁定义的 | 开发者（你）   | 模型厂商（内置能力）       |
| 是什么   | 一个具体的函数 | 一种模型输出格式           |
| 存哪里   | 你的代码里     | 模型参数/推理逻辑里        |
| 能执行吗 | 能，它是真代码 | 不能，它只是一种"指令格式" |
| 类比     | 餐厅后厨的菜   | 顾客点菜的能力             |

一句话记忆：

```text
Tool 是"手"，Function Calling 是"脑子知道要伸手"。
```

### Tool vs API

| 维度     | 普通 API                       | Tool                                      |
| -------- | ------------------------------ | ----------------------------------------- |
| 调用者   | 人（写代码）                   | 模型（自动决策）                          |
| 接口文档 | 写给人看的（Swagger / 文档页） | 写给模型看的（description + JSON Schema） |
| 调用时机 | 程序员预先编码决定             | 模型在推理时动态决定                      |
| 错误处理 | HTTP 状态码 + 错误信息         | 需要翻译成模型能理解的自然语言            |

实际上，一个 Tool 的 `execute` 函数内部经常就是在调用现有 API：

```ts
const weatherTool = {
	name: 'get_weather',
	description: '...', // ← 写给模型看的"接口文档"
	parameters: {
		/* ... */
	}, // ← 写给模型看的"参数说明"
	async execute(params) {
		// ↓ 这里调的是传统的 REST API
		return fetch(`https://api.weather.com?city=${params.city}`);
	},
};
```

所以 Tool ≈ **传统 API 外面包了一层"模型可读的说明书"**。

### Tool vs MCP Server

| 维度   | Tool                     | MCP Server                                |
| ------ | ------------------------ | ----------------------------------------- |
| 粒度   | 单个工具                 | 工具的集合（一个 Server 可提供多个 Tool） |
| 标准化 | 各厂商格式不同           | 统一协议，跨厂商复用                      |
| 部署   | 通常嵌入在你的应用代码里 | 独立进程，通过网络或 stdio 通信           |

Tool 是 MCP Server 的核心组成单元。一个 MCP Server 本质上就是"按协议封装的一组 Tool + Resource + Prompt"。

### Tool vs Plugin（ChatGPT Plugins）

| 维度     | Tool                          | Plugin                       |
| -------- | ----------------------------- | ---------------------------- |
| 开放性   | 你自己定义的，完全可控        | 平台方定义的，受限于上架规范 |
| 适用场景 | 你自己的应用                  | ChatGPT 等封闭平台           |
| 协议     | 各模型厂商格式（或 MCP 统一） | ChatGPT 专有格式             |

Plugin 本质上是 ChatGPT 平台上的一种"托管式 Tool"。你现在学 Tool，以后再看 Plugin，会发现它就是套了一层平台规范的 Tool。

## 六、对工程实践的影响

### Tool 设计原则

```
好的 Tool = 单一职责 + 清晰描述 + 精确参数 + 容错返回
```

| 原则             | 做法                                     | 反面例子                          |
| ---------------- | ---------------------------------------- | --------------------------------- |
| **单一职责**     | 一个 Tool 只做一件事                     | `do_everything` 这种万能工具      |
| **描述即文档**   | description 写清楚何时用、何时不用       | `"query database"` 太模糊         |
| **参数越少越好** | 必填参数 ≤ 3 个，总共 ≤ 5 个             | 10 个参数，模型大概率填错         |
| **返回值要友好** | 返回结构化的、模型易读的信息             | 返回 5000 行 HTML，模型根本看不懂 |
| **错误要有提示** | 失败时返回可读的错误信息，不是直接抛异常 | `throw new Error("err")`          |

### Tool 数量控制

这个经验来自实践：

```text
Tool 太少（1-2 个）：模型不够用，可能硬套不合适的 Tool
Tool 刚好（3-8 个）：模型能准确选择
Tool 太多（>15 个）：模型开始"选择困难"，容易选错
```

如果你的系统需要很多 Tool，考虑分组：

```ts
// ❌ 一次性把 30 个 Tool 全部暴露
const tools = [tool1, tool2, ..., tool30];

// ✅ 按场景分组，只给模型当前需要的
const toolsByScene = {
  coding: [readFile, writeFile, searchCode, runTest],
  research: [webSearch, fetchPage, summarize],
  office: [sendEmail, createDoc, scheduleMeeting],
};
```

### 对 Prompt 设计的影响

一旦引入 Tool，你的 System Prompt 就需要额外考虑：

```text
1. 工具使用规则：什么时候该调 Tool，什么时候直接回答
2. 工具选择优先级：多个 Tool 都可以用时，优先用哪个
3. 参数补全策略：参数缺了，是问用户还是用默认值
4. 失败处理：Tool 调用失败时，是重试还是告诉用户
5. 并行调用：是否允许同时调用多个无关 Tool
```

一个典型的带 Tool 的 System Prompt 增强：

```text
# 工具使用规则
- 用户问天气、新闻等实时信息时，必须调用对应 Tool，不要凭记忆回答。
- 如果 Tool 返回错误，告诉用户具体什么出错了，不要假装成功。
- 多个 Tool 可以同时调用时，一次请求里并行调用，提高效率。
- 如果参数不完整，优先使用合理默认值；如果无法默认，再问用户。
```

这比纯对话 Prompt 复杂很多——因为你现在不只是在描述"怎么回答"，还在描述"怎么行动"。

### 对幻觉治理的影响

Tool 是一把双刃剑：

```text
✅ Tool 返回真实数据 → 模型基于真实数据回答 → 降低幻觉
❌ Tool 返回了错误数据 → 模型信以为真 → 产生新的幻觉来源
```

所以 Tool 的返回内容也需要谨慎设计：只返回必要信息，不要夹带不可靠的推测。

### 在 RagFlow 项目中的对应

在你的 RagFlow 项目里，Tool 可能对应这些场景：

| 场景         | 可能的 Tool                         |
| ------------ | ----------------------------------- |
| 检索文档     | `search_docs(query, topK)`          |
| 重排序结果   | `rerank(results, query)`            |
| 读取完整文档 | `read_document(docId)`              |
| 切换知识库   | `switch_knowledge_base(kbName)`     |
| 导出结果     | `export_to_markdown(content, path)` |

这些 Tool 都可以用同一个模式定义和注册，让模型的推理能力和你的业务逻辑解耦。

## 七、常见误区

### 误区一：Tool 是模型自带的功能

**错误理解：**
模型内置了各种工具，你只需要跟它说"帮我查天气"它就能做到。

**正确理解：**
模型本身什么工具都没有。它只能输出文本。Tool 是你（开发者）提供的函数，模型做的只是输出一个"调用指令"，你的代码收到指令后才真正执行。ChatGPT 能查天气是因为 OpenAI 在背后接了天气 API，不是 ChatGPT 自己会。

---

### 误区二：Tool 越多越好

**错误理解：**
给模型配几十个 Tool，它就能做任何事，越全能越好。

**正确理解：**
Tool 多了会降低选择准确率。模型需要在有限的上下文窗口里阅读所有 Tool 的描述，Tool 太多会分散注意力、增加选错的概率。好的做法是**按场景给刚好够用的 Tool**。

---

### 误区三：Tool 的 description 和 parameters 随便写写就行

**错误理解：**
模型很聪明，只要给它函数名，它就能猜出怎么用。

**正确理解：**
Tool 的 description 和 parameters schema 是模型唯一的信息来源。写得不清楚，模型调 Tool 就会出错。这和你写 API 文档一样——文档模糊，调用方就会乱用。description 不是注释，是**给模型看的说明书**，需要认真写。

---

### 误区四：Tool 调用是安全的，因为最终还是人在控制

**错误理解：**
模型只是发指令，真正执行的是我的代码，所以不会有安全问题。

**正确理解：**
危险在于**模型可以决定参数值**。比如你给了一个 `delete_file(path)` Tool，模型的参数里的 `path` 可能被恶意构造（prompt injection），也可能纯粹是模型"想错了"。所以 Tool 的执行函数里必须有**参数校验、权限检查、日志记录**，高风险操作要有**人工确认**。

## 八、用自己的话总结

Tool 不是魔法，它就是一个约定：

```text
Tool = 函数 + JSON Schema 说明书 + 调用循环
```

用前端开发的话来记：

```text
Tool 之于 LLM，
就像 API 之于前端应用。

前端调用 API 获取数据、执行操作；
LLM 调用 Tool 做同样的事。
区别只在于：前端是你写死代码调 API，
Tool 是模型自己决定调哪个、什么时候调。
```

核心记忆句：

```text
Tool 是 LLM 的"手和脚"——它让模型不只停留在"说"，
而是进入"做"的世界。
但 Tool 本身只是函数，模型只负责"决定调用"，
真正执行的始终是你的代码。
```

和 Agent、MCP 的关系链：

```text
Tool  <──  单个能做的事（函数）
  │
  ▼
MCP   <──  标准化 Tool 的接入协议（让 Tool 可复用）
  │
  ▼
Agent <──  把 Tool + 目标 + 循环组合成自主任务系统
```

理解这个递进关系，你就不会再把 Tool、Function Calling、MCP、Agent 这四个概念混在一起了。
