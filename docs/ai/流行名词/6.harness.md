# Harness（Agent 运行时）

> Harness 原意是"马具"或"安全带"，在软件工程里指**控制、管理和监控某个系统运行的外层框架**。  
> 在 AI 语境下，Harness 特指：**把大模型变成一个能稳定运行的 Agent 所需的全部运行时基础设施**。

## 一、先说结论

Harness 的本质用一句话：

```text
Agent = Model（模型） + Harness（运行时外壳）

模型负责"思考"，Harness 负责"思考之外的一切"。
```

类比：

```text
模型 ≈ CPU（中央处理器，负责计算）
Harness ≈ 操作系统（管理内存、调度进程、处理中断、提供安全边界）
```

你只给 CPU 通电，它什么都做不了。必须配上操作系统，才能跑应用。同样的道理：你只给 LLM 一个 Prompt，它只能生成文本。必须配上 Harness，才能变成能稳定执行任务的 Agent。

**Harness 不是某个特定工具，而是一层架构概念**——它是模型和真实世界之间的"中间层"。

## 二、为什么需要理解它

### AI 工程的范式迁移

2023 到 2026 年，AI 应用开发的焦点经历了三次转移：

```text
2023：Prompt Engineering（提示词工程）
      → 核心问题：怎么让模型输出更好的结果？

2024-2025：Context Engineering（上下文工程）
      → 核心问题：怎么给模型提供最相关的信息？

2026：Harness Engineering（运行时工程）
      → 核心问题：怎么让模型稳定、安全、可观测地持续运行？
```

如果你只关注 Prompt 和模型选型，而不理解 Harness，就像你只关注发动机马力，却不关心刹车、方向盘和仪表盘——车能跑，但上路一定出事故。

### 它解决了什么问题

当你从"调用一次 LLM"升级到"让 LLM 自主完成一个任务"时，会冒出大量新问题：

| 问题 | 没有 Harness 时的做法 | Harness 的做法 |
|---|---|---|
| 模型调了危险命令怎么办？ | 靠 Prompt 里说"不要做危险的事" | 执行前拦截，策略引擎判断是否允许 |
| 任务跑到一半崩溃了？ | 从头再来 | 状态持久化，从断点恢复 |
| 怎么知道 Agent 每一步做了什么？ | 翻聊天记录 | 结构化日志、链路追踪、审计回放 |
| 模型陷入循环出不来？ | 手动关掉 | 超时检测、步数限制、循环检测 |
| 多个用户同时用，状态乱了？ | 自己管理 Session | 内置租户隔离和会话管理 |
| 第三方工具返回了恶意内容？ | 不知道，直接塞给模型 | 沙箱隔离、输出校验、内容过滤 |

Harness 的本质是：**把所有"出了事怎么办"的问题，提前用工程手段解决掉。**

## 三、举一个直观例子

### 场景：做一个能帮你发邮件的 Agent

**不用 Harness（裸调模型 + 手写循环）：**

```ts
// 你手写了一个简单的 Agent 循环
async function runAgent(userGoal: string) {
  const tools = [searchTool, emailTool, fileTool];

  for (let step = 0; step < 10; step++) {
    const response = await callLLM(userGoal, tools);

    if (response.wantToCallTool) {
      // ⚠️ 直接执行！没有权限检查
      const result = await executeTool(response.toolName, response.params);
      // ⚠️ 如果这里崩溃了，前面的状态全丢
      // ⚠️ 没有日志，不知道发生了什么
      // ⚠️ 如果工具返回了恶意内容，直接塞给模型
      continue;
    }

    return response.finalAnswer;
  }
  // ⚠️ 循环结束没有处理——静默失败
}
```

这个代码能跑，但它经不起真实世界的折腾。十个问题里它一个都处理不了。

**用 Harness（模型 + 运行时基础设施）：**

```ts
// Harness 提供了完整的运行时外壳
const agent = createAgent({
  model: "claude-sonnet-4-6",

  // 1. 工具注册 + 权限分级
  tools: [
    { name: "search_email", permission: "read", handler: searchEmail },
    { name: "send_email", permission: "write", requireApproval: true, handler: sendEmail },
    { name: "delete_email", permission: "dangerous", requireApproval: "always", handler: deleteEmail },
  ],

  // 2. 策略引擎：执行前的拦截规则
  policies: {
    // 任何"发送"操作都要确认收件人
    beforeAction: async (action, context) => {
      if (action.tool === "send_email" && !action.params.to.endsWith("@mycompany.com")) {
        return { allow: false, reason: "不允许发送到外部邮箱，需要人工审批" };
      }
      return { allow: true };
    },
  },

  // 3. 状态持久化
  state: {
    backend: "postgres",       // 存到数据库，不怕崩溃
    snapshotInterval: 5,       // 每 5 步保存快照
  },

  // 4. 可观测性
  observability: {
    tracing: true,              // 链路追踪
    logLevel: "structured",    // 结构化日志
  },

  // 5. 安全边界
  limits: {
    maxSteps: 20,               // 步数上限
    maxWallTime: "5min",        // 时间上限
    maxCost: "$2.00",           // 成本上限
  },
});

// 运行 Agent——Harness 替你处理所有边缘情况
const result = await agent.run("帮我整理今天的邮件并回复重要事项");
// Harness 自动处理：权限检查、状态保存、异常恢复、日志记录、成本控制...
```

### 用前端开发类比

```text
Harness 之于 Agent，
就像 React/Vue 之于 Web 应用。

你可以用原生 JS 写一个网页（裸调 DOM API），
也可以选择用 React（有状态管理、组件生命周期、虚拟 DOM、DevTools）。

前者在小项目里够用，但一旦交互复杂、状态变多、多人协作，
你就需要一个框架来管理复杂度。

Harness 就是 Agent 世界的"前端框架"——
它不替代模型（就像 React 不替代 JS 引擎），
而是提供了一套工程化的运行时基础设施。
```

## 四、本质是什么

### 八层架构模型

业界逐渐形成了一个共识——一个完整的 Harness 包含八个层次：

```text
┌─────────────────────────────────────────┐
│ 1. 输入 & 触发层                         │
│    API 调用、消息、定时任务、Webhook      │
├─────────────────────────────────────────┤
│ 2. 上下文工程层                           │
│    检索、压缩、信息组装、Prompt 渲染      │
├─────────────────────────────────────────┤
│ 3. 编排 & 规划层                          │
│    任务分解、多 Agent 协作、DAG 执行      │
├─────────────────────────────────────────┤
│ 4. 推理核心层  ← 模型在这里               │
│    记忆系统 + LLM 调用 + 工具调度          │
├─────────────────────────────────────────┤
│ 5. 策略 & 护栏层                          │
│    执行前拦截、权限校验、合规检查          │
├─────────────────────────────────────────┤
│ 6. 状态 & 会话层                          │
│    持久化、崩溃恢复、快照/还原             │
├─────────────────────────────────────────┤
│ 7. 动作 & 输出层                          │
│    工具执行、沙箱隔离、结果格式化          │
├─────────────────────────────────────────┤
│ 8. 可观测性层（贯穿全部）                  │
│    链路追踪、评估、审计、回放              │
└─────────────────────────────────────────┘
```

你不用每次都实现全部八层，但你要知道：当你写的 Agent 出问题时，大概率是某几层被省略了。

### 核心公式

```text
Agent = Model + Harness

展开：
Agent = LLM（推理）
      + Tools（能力）
      + Memory（记忆）
      + Loop（循环）
      + Policies（策略护栏）        ← 这五行合在一起
      + State（状态持久化）           ← 就是 Harness
      + Observability（可观测性）
      + Sandbox（沙箱）
```

### Harness 不是框架，是"环境"

一个重要区分：

```text
Agent Framework（LangChain, CrewAI）
  → 告诉你"怎么编排 Agent 的逻辑"
  → 你照着框架写的代码里包含了 Harness 的部分功能

Harness（概念层）
  → 描述的是"Agent 运行时需要哪些能力"
  → 不管你用什么框架，这些能力你都需要

类比：
Framework = 装修方案（告诉你怎么布置房间）
Harness    = 房子的水电煤网（不管什么风格，都得通）
```

## 五、和相近概念的区别

### Harness vs Agent Framework

| 维度 | Harness | Agent Framework |
|---|---|---|
| 层级 | 架构概念 / 能力集合 | 具体的代码库 |
| 关注点 | Agent **运行时需要什么** | Agent **逻辑怎么写** |
| 包含关系 | Harness 包含 Framework 提供的部分能力 | Framework 是实现 Harness 的一种方式 |
| 例子 | "Agent 需要权限控制、状态持久化、可观测性" | LangChain、CrewAI 等 |

你用 LangChain 搭建 Agent 时，LangChain 提供了部分 Harness 能力（如工具调度、记忆管理），但策略护栏、状态持久化、可观测性通常需要你自己补上。

### Harness vs MCP

```text
MCP 是 Harness 中"工具接入"那一层的标准化协议。

Harness 八层中的第 7 层（动作 & 输出层）可以用 MCP 来实现工具的标准接入。
但 Harness 还包括权限、状态、编排、可观测性等 MCP 不覆盖的部分。
```

### Harness vs 操作系统

| 维度 | 操作系统 | Harness |
|---|---|---|
| 管理者 | 管理硬件资源（CPU、内存、磁盘） | 管理 Agent 资源（模型调用、工具执行、状态、安全） |
| 提供 | 进程隔离、内存管理、中断处理 | 权限隔离、上下文管理、异常恢复 |
| 类比 | CPU 的运行时 | LLM 的运行时 |

Harness 经常被称为"Agent OS"不是夸张——它在 Agent 技术栈里扮演的角色，确实和操作系统在计算机体系里扮演的角色一样：**它是模型和真实世界之间的那层管理软件**。

## 六、对工程实践的影响

### 设计 AI 应用时的思维转变

理解了 Harness 之后，你设计 AI 应用的方式会发生变化：

```text
之前："我选哪个模型？Prompt 怎么写？"
之后："我的 Agent 需要哪些运行时能力？权限怎么分级？状态存哪里？怎么监控？"
```

具体到每个环节：

| 环节 | 之前的关注点 | 之后的关注点 |
|---|---|---|
| 工具集成 | 能不能调通 | 权限怎么分级？失败怎么降级？ |
| 对话历史 | 塞进 Prompt 就行 | 记忆怎么分层？短期 vs 长期怎么管理？ |
| 错误处理 | try-catch | 状态能不能从断点恢复？能不能回滚已执行的操作？ |
| 部署 | 起一个服务 | 崩溃怎么自动恢复？多实例状态怎么同步？ |
| 监控 | 看日志文件 | 每个 Tool 调用的延迟/成功率/成本？用户体验的端到端追踪？ |

### 最小可行 Harness 清单

即使是原型项目，也建议至少覆盖这几层：

```text
✅ 工具执行前的一层权限校验（哪怕只是一张"允许列表"）
✅ 最大步数限制（防止无限循环烧钱）
✅ 每一步的结构化日志（不用复杂的可观测性平台，先写到文件里）
✅ 工具执行的 try-catch + 有意义的错误信息（不要静默吞掉）
```

这四点只需要几十行代码，但能帮你避开 80% 的 Agent 开发常见坑。

### 在 RagFlow 项目中的对应

如果把 RagFlow 看作一个 Agent，它的 Harness 可能长这样：

| Harness 层 | RagFlow 对应模块 |
|---|---|
| 输入 & 触发 | API / 消息队列接收查询请求 |
| 上下文工程 | 文档检索、重排序、Chunk 拼接 |
| 编排 & 规划 | 查询改写、多路召回、结果融合 |
| 推理核心 | LLM 调用 + 检索结果注入 |
| 策略 & 护栏 | 知识库范围限制、敏感词过滤、来源校验 |
| 状态 & 会话 | 对话历史存储、用户偏好记忆 |
| 动作 & 输出 | 检索工具、导出工具、反馈收集 |
| 可观测性 | 检索命中率、LLM 延迟、用户满意度追踪 |

## 七、常见误区

### 误区一：Harness 就是一个 Agent 框架

**错误理解：**
Harness 就是 LangChain、CrewAI 这类东西，用框架就等于有了 Harness。

**正确理解：**
框架提供了部分 Harness 能力，但 Harness 是一个更广的概念——它还包括权限系统、状态持久化、可观测性、沙箱隔离等基础设施。用了框架不代表你的 Agent 就有完整的运行时保障，这些基础设施往往需要单独搭建。

---

### 误区二：Harness 是可有可无的"高级货"

**错误理解：**
我写个简单的 Agent 脚本，调几次 LLM、调几个工具，不需要什么 Harness，那是大厂才用的。

**正确理解：**
你的 `while` 循环、`try-catch`、`maxSteps` 限制、console.log——这些已经是雏形的 Harness 了。区别只在于你有没有**系统地**做这些事。即使是原型，有意识地加几层基础保障（步数限制、权限检查、结构化日志），也能大幅减少调试时间。

---

### 误区三：Prompt 写得够好，就不需要 Harness

**错误理解：**
只要 Prompt 里约束得够清楚（"不要删除文件""不确定就问用户"），模型就会遵守，不需要额外的运行时控制。

**正确理解：**
Prompt 是建议，不是规则。Prompt injection（恶意指令注入）可以让模型完全绕过 Prompt 里的约束。真正的安全控制必须在执行层——在代码里拦截，不能只在 Prompt 里声明。Prompt 是第一层防线，Harness 的护栏是最后一道防线。

---

### 误区四：Harness 里最重要的是工具调用

**错误理解：**
Agent 运行时最核心的就是怎么调度工具，其他的（状态、日志、权限）都是次要的。

**正确理解：**
工具调度只是 Harness 八层中的两层。真正让 Agent 从"demo"变成"产品"的，恰恰是那些"次要"的东西——崩溃恢复、权限控制、可观测性。一个只会调工具但没有状态持久化的 Agent，在生产环境活不过一天。

## 八、用自己的话总结

Harness 不是一个新产品或新框架，而是 AI 工程在 2026 年形成的一种**共识**：

```text
把大模型变成可靠 Agent 的关键，
不在于 Prompt 写得多好、模型选得多强，
而在于模型外面那层"运行时外壳"够不够扎实。
```

这个共识的形成过程很像前端工程化的历史：

```text
2010 年的前端：手写 HTML + CSS + 少量 jQuery，能跑就行
2020 年的前端：Webpack + TypeScript + React + 测试 + CI/CD
               → 不是语言变了，是工程标准变了

2023 年的 AI：调 API + 写 Prompt，能回答就行
2026 年的 AI：Model + Harness（状态 + 权限 + 可观测 + 容错）
               → 不是模型变了，是运行时标准变了
```

核心记忆句：

```text
Agent = Model + Harness
模型是大脑，Harness 是身体——大脑再聪明，
没有身体（权限边界、状态管理、错误恢复、监控日志），
也做不成任何事。
```

## 附录：Harness vs OpenClaw 对比

你之前学过的 OpenClaw 和 Harness 是什么关系？放在一起比较会更清楚：

| 维度 | Harness | OpenClaw |
|---|---|---|
| **是什么** | 一个**架构概念**：Agent 运行时所需的能力集合 | 一个**具体产品**：开源的 AI Agent 应用 |
| **层级** | 概念层 / 设计模式 | 产品层 / 实现 |
| **关系** | Harness 描述了"一个好的 Agent 运行时应该有哪些能力" | OpenClaw 是 Harness 概念的一个**具体实例** |
| **谁来实现** | 你自己搭（或用框架拼） | 装好即用，内置了一套 Harness 实现 |
| **灵活性** | 完全可控，但需要自己写很多代码 | 开箱即用，但受限于 OpenClaw 的架构选择 |

### OpenClaw 实现了 Harness 的哪些层

```text
Harness 八层          OpenClaw 的对应实现
──────────────────────────────────────────
1. 输入 & 触发    ✅  50+ 消息平台接入（微信/Slack/Discord/Web...）
2. 上下文工程      ✅  分层记忆 + MCP Resources
3. 编排 & 规划     ✅  任务调度中心（有限状态机）
4. 推理核心        ✅  模型推理引擎（本地 + 云端）
5. 策略 & 护栏     ⚠️  有，但不够——这是它最大的争议点
6. 状态 & 会话     ✅  本地存储 / 内存（缺乏生产级持久化）
7. 动作 & 输出     ✅  Shell 执行 + 文件系统 + 5700+ 技能模块
8. 可观测性        ⚠️  有基本日志，但缺少企业级追踪和审计
```

OpenClaw 几乎把 Harness 八层都实现了一遍，但它在**第 5 层（策略 & 护栏）**上的薄弱，正是导致那 1.5 万台设备被公网暴露、1200+ 恶意技能被注入的根本原因。

### 关键差异：安全哲学

| 维度 | Harness 的理念 | OpenClaw 的实践 |
|---|---|---|
| 权限模型 | 最小权限：不给超出任务需要的权限 | 最大能力：默认给你操作系统级权限 |
| 执行前拦截 | 策略引擎在每次 Tool 调用前校验 | 主要靠 Prompt 约束（"建议"不是"规则"） |
| 沙箱隔离 | 第三方工具在隔离环境中运行 | 技能模块直接访问宿主机 |
| 人工确认 | 高风险操作默认需要确认 | 需要用户自己配置（很多人不配） |
| 设计哲学 | "先设边界，再逐步开放" | "先全开放，再打补丁" |

### 一句话总结两者的关系

```text
Harness 是"Agent 运行时应该长什么样"的设计蓝图。
OpenClaw 是"一个激进的 Agent 产品实验"——它证明了
蓝图里"护栏层"的重要性，用的是反面教材的方式。
```

换句话说：学 Harness 让你知道**怎么正确地搭建 Agent 运行时**；学 OpenClaw 让你知道**搭错了会有什么后果**。两个都懂，你才算真正理解了 AI Agent 工程的全貌。
