# Skills

> Skill 通常翻译成“技能”。  
> 在 AI 应用里，它不是新模型，也不是工具本身，而是一份可复用的“任务说明书”。

## 一、先说结论

Skill 可以理解为：

```text
把一类任务的做法，封装成可复用的结构化指令。
```

更直白一点：

```text
Prompt 是一次性的嘱咐。
Skill 是可以反复使用的做事方法。
```

它的核心价值不是让模型突然变聪明，而是让模型在某类任务上更稳定、更专业、更像“按流程办事”。

一个简化公式：

```text
Skill = 适用场景 + 角色设定 + 工作流程 + 输出约束 + 可用工具
```

## 二、为什么需要 Skill

直接写 Prompt 有两个常见问题：

1. **重复**：同类任务每次都要重新交代规则。
2. **不稳定**：同样的需求，不同时间输出质量不一样。

比如你经常让 AI 写学习文档，每次都要说：

```text
先说结论。
用中文。
多举例。
不要堆术语。
最后总结。
```

这些规则如果每次都手写，就很容易漏。  
Skill 的作用就是把这些经过验证的规则固化下来。

可以类比成前端里的组件：

```text
裸 Prompt：每次手写一段按钮样式。
Skill：把按钮封装成组件，以后直接复用。
```

## 三、举一个直观例子

假设你想让 AI 稳定生成 AI 学习文档。

普通 Prompt 可能是：

```text
帮我写一篇关于 Agent 的文章。
```

结果可能有时像百科，有时像教程，有时又太散。

如果做成 Skill，它会变成一套固定规则：

```text
当用户要学习 AI 概念时：
1. 先用一句话给结论。
2. 解释为什么需要理解它。
3. 举一个直观例子。
4. 说明它的本质。
5. 对比容易混淆的概念。
6. 说明工程实践影响。
7. 最后用自己的话总结。

输出要求：
- 中文；
- Markdown；
- 少术语；
- 多例子；
- 适合前端开发者理解。
```

这就是一个 Skill 的雏形。

它不是代码，但它像代码一样被复用。  
它不是 Tool，但它可以指导模型什么时候用 Tool。  
它不是 Agent，但它可以成为 Agent 的一部分能力。

## 四、Skill 的本质是什么

从更底层看，Skill 是对模型上下文的组织。

LLM 本质上是基于上下文生成内容的模型。  
Skill 做的事情，就是提前把高质量的任务规则放进上下文，让模型输出朝固定方向收敛。

所以 Skill 的本质可以说是：

```text
一种可复用的上下文模块。
```

一个 Skill 通常包含这些部分：

| 组成部分 | 作用                   | 例子                                 |
| -------- | ---------------------- | ------------------------------------ |
| 适用场景 | 什么时候触发           | 用户要求写 AI 学习文档               |
| 角色设定 | 模型应该以什么身份工作 | 你是技术写作者                       |
| 工作流程 | 按什么步骤完成任务     | 结论 -> 例子 -> 本质 -> 误区 -> 总结 |
| 输出约束 | 结果长什么样           | Markdown、中文、表格、代码块         |
| 可用工具 | 可以借助什么能力       | 读文件、搜索、运行命令               |
| 边界限制 | 什么不能做             | 不编造、不乱改文件、不越权操作       |

这也是 Skill 和普通 Prompt 最大的区别：  
Skill 不只是“说一句话”，而是把一类任务的上下文结构固定下来。

## 五、和 Prompt、Tool、Agent 的区别

这几个概念很容易混在一起。

| 概念   | 是什么                 | 类比                   |
| ------ | ---------------------- | ---------------------- |
| Prompt | 一次具体输入           | 一句话请求             |
| Skill  | 可复用任务方法         | 操作手册 / 组件        |
| Tool   | 可执行功能             | 函数 / API             |
| Agent  | 能规划和执行任务的系统 | 会调用组件和工具的应用 |

### Skill vs Prompt

Prompt 是一次性的。

```text
帮我 review 这段代码。
```

Skill 是可复用的。

```text
当做 code review 时：
1. 先找 bug。
2. 再看安全风险。
3. 再看性能问题。
4. 最后按严重程度输出表格。
```

可以理解为：

```text
Skill = Prompt 的工程化封装。
```

### Skill vs Tool

Tool 是能真正执行动作的函数或接口。

```text
读取文件
搜索代码
调用数据库
运行测试
```

Skill 是指导模型如何完成任务的流程。

```text
先读文件 -> 再找问题 -> 再输出报告
```

一句话区分：

```text
Tool 是手。
Skill 是手艺。
```

### Skill vs Agent

Agent 是一个能围绕目标持续行动的系统。  
Skill 是 Agent 可以加载的一种能力模块。

```text
Agent 负责决定做什么。
Skill 提供某类任务怎么做。
Tool 负责真正执行动作。
```

关系可以这样看：

```text
Agent
  ├─ Skill：写技术文档
  ├─ Skill：代码审查
  ├─ Skill：需求拆解
  └─ Tools：读文件、搜索、执行命令、调用 API
```

## 六、Skill 对工程实践的影响

理解 Skill 之后，AI 应用设计会从“写一段 Prompt”变成“沉淀一类能力”。

### 1. 把高频任务模块化

适合做成 Skill 的任务通常有几个特点：

- 经常重复；
- 有稳定流程；
- 有明确输出格式；
- 需要特定领域经验；
- 不希望每次重新解释规则。

比如：

- 写技术文档；
- 做代码审查；
- 生成 PR 描述；
- 分析错误日志；
- 做 RAG 问答；
- 生成测试用例。

### 2. 让 AI 输出更稳定

Skill 会固定任务流程和输出格式。  
这能减少模型自由发挥，让结果更像“按规范生产”。

### 3. 让团队经验可复用

一个好的 Skill，本质上是在沉淀团队经验。

比如代码审查 Skill 里可以写：

```text
优先关注：
1. 真实 bug；
2. 安全风险；
3. 破坏兼容性的改动；
4. 缺失测试。
```

这样新人、AI、不同项目都能复用同一套标准。

### 4. 更容易评估和迭代

Skill 是固定文件或固定配置，就可以像代码一样：

- 改版本；
- 做测试；
- 比较输出质量；
- 回滚到旧版本。

这比临时写 Prompt 更工程化。

## 七、如何设计一个 Skill

可以按这个模板思考：

```text
Skill 名称：
这个 Skill 解决什么问题？

适用场景：
用户在什么情况下应该使用它？

输入：
它需要哪些信息？

工作流程：
模型应该按什么步骤完成？

输出格式：
最终结果应该长什么样？

可用工具：
是否需要读文件、搜索、执行命令、调用 API？

边界：
什么事情不能做？

质量标准：
什么样的输出才算好？
```

一个简化例子：

```text
Skill 名称：
AI 概念文档写作

适用场景：
用户想学习某个 AI 概念，并希望生成 Markdown 笔记。

工作流程：
1. 先说结论。
2. 解释为什么重要。
3. 举一个直观例子。
4. 拆解本质。
5. 对比相近概念。
6. 说明工程实践影响。
7. 总结成一句话。

输出要求：
中文、Markdown、少术语、多例子。
```

## 八、常见误区

### 误区一：Skill 是新模型能力

错误理解：Skill 会让模型本身变强。

正确理解：Skill 不改模型，只是给模型提供更好的上下文和流程。

### 误区二：Skill 就是 Tool

错误理解：Skill 可以直接执行动作。

正确理解：Tool 才负责执行，Skill 负责指导怎么做。

### 误区三：Skill 越多越好

错误理解：给 Agent 加越多 Skill，它越强。

正确理解：Skill 太多会增加选择成本。好的 Skill 应该按场景拆分，按需加载。

### 误区四：有了 Skill 就不用人判断

错误理解：Skill 固化了流程，所以结果一定可靠。

正确理解：Skill 提高稳定性，但不保证正确。涉及代码修改、数据操作、对外发送内容时，仍然需要人工确认。

## 九、用自己的话总结

Skill 就是把“怎么让 AI 做好一类事”的经验，写成一份可以反复使用的说明书。

一句话记忆：

```text
Skill 是 Prompt 的工程化封装，
把临时嘱咐变成可复用的能力模块。
```

最终公式：

```text
Skill = 场景 × 流程 × 约束 × 输出标准
```

对 AI 工程来说，Skill 的意义在于：  
它让我们不只是“会问 AI”，而是开始把 AI 的工作方式沉淀成可维护、可复用、可迭代的系统能力。
