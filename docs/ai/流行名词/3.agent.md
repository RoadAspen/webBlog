# Agent

> Agent 常被翻译成“智能体”。  
> 它不是一个新的大模型，而是一种“让大模型带着目标、记忆和工具去完成任务”的应用模式。

## 先说结论

Agent 可以理解为：

```text
LLM + 目标 + 工具 + 记忆 + 执行循环
```

普通聊天模型更像是：

```text
你问一句 -> 模型答一句
```

Agent 更像是：

```text
你给一个目标
-> 模型思考要做什么
-> 选择工具
-> 执行工具
-> 观察结果
-> 再决定下一步
-> 直到任务完成
```

所以 Agent 的关键不在于**模型更聪明**，而在于它不只是生成文本，还能围绕目标持续行动。

## Agent 到底是什么

一个最小的 Agent 通常包含 5 个部分：

| 组成部分 | 含义     | 例子                                             |
| -------- | -------- | ------------------------------------------------ |
| Goal     | 目标     | “帮我整理一份某个技术方案的对比报告”             |
| LLM      | 决策大脑 | 判断下一步该搜索、读取文件、写代码还是总结       |
| Tools    | 工具     | 搜索网页、读取文件、调用数据库、执行代码、发请求 |
| Memory   | 记忆     | 当前任务历史、用户偏好、已经查到的信息           |
| Loop     | 执行循环 | 计划 -> 执行 -> 观察 -> 再计划                   |

如果没有工具，Agent 只能“想”和“说”。
如果没有循环，Agent 只能做一步。
如果没有记忆，Agent 很难处理长任务。
如果没有目标，Agent 就只是普通聊天。

## 怎么理解 Agent

可以把 Agent 想成一个实习生，但这个实习生有几个特点：

- 它会读你的目标。
- 它会拆任务。
- 它会决定什么时候用工具。
- 它会根据工具返回的结果继续调整。
- 它可能犯错，所以需要限制权限、检查结果和设置停止条件。

例如你说：

```text
帮我分析这个项目为什么启动失败，并尝试修复。
```

普通 LLM 可能会直接给你一堆建议。

Agent 则可能这样做：

1. 查看项目目录。
2. 读取 package.json。
3. 判断启动命令。
4. 执行 npm run dev。
5. 观察报错。
6. 搜索报错对应的文件。
7. 修改代码。
8. 再次运行。
9. 总结修复结果。

这就是 Agent 和普通问答最大的区别：它会进入“做事循环”。

## Agent 的几个具体例子

### 例子一：代码修复 Agent

目标：

```text
修复项目中的一个 bug。
```

可能的执行流程：

1. 读取错误日志。
2. 查找相关代码。
3. 分析 bug 原因。
4. 修改代码。
5. 运行测试。
6. 如果测试失败，继续分析。
7. 如果测试通过，输出修复说明。

常用工具：

- 读取文件；
- 搜索代码；
- 修改文件；
- 执行测试命令；
- 查看 git diff。

### 例子二：资料研究 Agent

目标：

```text
帮我调研几个大模型厂商的能力差异。
```

可能的执行流程：

1. 搜索权威资料。
2. 打开官方文档。
3. 提取模型名称、价格、上下文长度、工具能力。
4. 做表格对比。
5. 标注资料来源。
6. 总结选型建议。

常用工具：

- Web Search；
- 浏览网页；
- 信息抽取；
- 表格生成。

### 例子三：办公自动化 Agent

目标：

```text
根据会议纪要生成任务清单，并发给对应负责人。
```

可能的执行流程：

1. 读取会议纪要。
2. 提取任务、负责人、截止时间。
3. 判断信息是否缺失。
4. 生成结构化任务列表。
5. 调用任务系统 API 创建任务。
6. 发送通知。

常用工具：

- 文档读取；
- JSON 结构化输出；
- 企业微信 / 飞书 / Slack API；
- 数据库或任务系统。

## Agent 的核心循环

Agent 最经典的执行结构可以写成：

```text
while 任务未完成:
  观察当前状态
  思考下一步
  选择一个工具
  执行工具
  得到结果
  更新记忆
```

也可以抽象成四个词：

```text
Plan -> Act -> Observe -> Reflect
```

中文理解是：

```text
计划 -> 行动 -> 观察结果 -> 反思调整
```

## 简易实现

下面是一个非常简化的 TypeScript 版本，用来理解 Agent 的核心结构。

它没有接入真实大模型，只是展示 Agent 应该有哪些模块。

```ts
type ToolResult = {
	ok: boolean;
	content: string;
};

type Tool = {
	name: string;
	description: string;
	run: (input: string) => Promise<ToolResult>;
};

type AgentStep = {
	thought: string;
	toolName?: string;
	toolInput?: string;
	finalAnswer?: string;
};

const tools: Tool[] = [
	{
		name: 'searchDocs',
		description: 'Search local documents by keyword.',
		async run(input: string) {
			return {
				ok: true,
				content: `Found related document snippets for: ${input}`,
			};
		},
	},
	{
		name: 'calculator',
		description: 'Calculate a simple math expression.',
		async run(input: string) {
			try {
				// Demo only. Do not use eval in real production code.
				const result = Function(`return (${input})`)();
				return { ok: true, content: String(result) };
			} catch {
				return { ok: false, content: 'Invalid expression.' };
			}
		},
	},
];

async function callLLM(prompt: string): Promise<AgentStep> {
	// 真实项目中，这里应该调用 OpenAI / Claude / Gemini / Qwen 等模型。
	// 为了演示，这里用规则模拟模型决策。
	if (prompt.includes('计算')) {
		return {
			thought: '用户需要计算，我应该调用 calculator 工具。',
			toolName: 'calculator',
			toolInput: '100 * 0.8',
		};
	}

	if (prompt.includes('资料') || prompt.includes('文档')) {
		return {
			thought: '用户需要查资料，我应该调用 searchDocs 工具。',
			toolName: 'searchDocs',
			toolInput: 'Agent 是什么',
		};
	}

	return {
		thought: '已经有足够信息，可以直接回答。',
		finalAnswer: 'Agent 是一个能围绕目标调用工具、持续执行任务的 AI 系统。',
	};
}

async function runAgent(userGoal: string) {
	const memory: string[] = [];
	const maxSteps = 5;

	for (let step = 0; step < maxSteps; step++) {
		const prompt = `
用户目标：
${userGoal}

可用工具：
${tools.map((tool) => `- ${tool.name}: ${tool.description}`).join('\n')}

历史记录：
${memory.join('\n')}

请判断下一步要做什么。
`;

		const decision = await callLLM(prompt);
		memory.push(`Thought: ${decision.thought}`);

		if (decision.finalAnswer) {
			return decision.finalAnswer;
		}

		const tool = tools.find((item) => item.name === decision.toolName);

		if (!tool || !decision.toolInput) {
			return '任务失败：模型选择了不存在的工具或缺少工具参数。';
		}

		const result = await tool.run(decision.toolInput);
		memory.push(`Tool: ${tool.name}`);
		memory.push(`Tool Input: ${decision.toolInput}`);
		memory.push(`Observation: ${result.content}`);

		if (result.ok) {
			return `工具执行完成：${result.content}`;
		}
	}

	return '任务停止：超过最大执行步数。';
}

runAgent('帮我查一下 Agent 是什么').then(console.log);
```

这个简易实现里有几个关键点：

- `tools`：定义 Agent 能用哪些工具。
- `callLLM`：让模型判断下一步。
- `memory`：保存执行历史。
- `maxSteps`：限制最多执行几步，避免无限循环。
- `runAgent`：把“思考 -> 工具调用 -> 观察结果”串起来。

## 真实项目中还需要什么

上面的代码只是玩具版本。真正能用的 Agent 还需要更多工程能力：

| 能力            | 为什么需要                    |
| --------------- | ----------------------------- |
| 工具参数 schema | 防止模型乱填参数              |
| 权限控制        | 防止模型调用危险工具          |
| 执行日志        | 方便追踪每一步做了什么        |
| 错误重试        | 工具失败后可以恢复            |
| 停止条件        | 防止无限循环和成本失控        |
| 人工确认        | 高风险操作前让用户确认        |
| 状态管理        | 长任务需要保存进度            |
| 评估机制        | 判断 Agent 是否真的完成了任务 |

尤其要注意：Agent 越能行动，越需要边界。

例如：

- 读取文件可以允许；
- 修改文件要谨慎；
- 删除文件要非常谨慎；
- 发邮件、转账、下单、执行生产命令，一定要有人类确认。

## 常见误区

### 误区一：Agent 就是更强的大模型

不是。Agent 是应用架构，不是模型本身。

同一个模型，可以被用作普通聊天，也可以被放进 Agent 系统里。

### 误区二：Agent 会自动变聪明

不会。Agent 只是让模型有了工具和循环。  
如果目标不清楚、工具不好用、上下文混乱，它仍然会失败。

### 误区三：工具越多越好

工具太多会增加选择难度。  
好的 Agent 应该给模型提供“刚好够用”的工具，并且每个工具描述清楚、参数明确。

### 误区四：Agent 可以完全无人监管

对低风险任务可以自动执行。  
对高风险任务必须有人类确认。

## 用自己的话总结

Agent 的本质不是“让 AI 像人一样有意识”，而是：

```text
让大模型在一个受控系统里，
围绕目标，
选择工具，
执行动作，
观察结果，
持续推进任务。
```

普通 LLM 更像“会回答问题的大脑”。  
Agent 更像“带着工具箱做事的助手”。

对于 AI 应用开发来说，Agent 是从“问答系统”走向“任务执行系统”的关键一步。
