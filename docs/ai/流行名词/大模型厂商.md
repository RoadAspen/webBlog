# 大模型厂商

> 更新时间：2026-07-16  
> 这篇文档不是严格排行榜，而是一张大模型厂商地图：快速理解国内外主要玩家、主力模型、能力差异和工具生态。

## 先说结论

当前大模型厂商可以粗略分成三类：

1. **闭源前沿模型厂商**：OpenAI、Anthropic、Google DeepMind、xAI 等，优势是综合能力、工具生态、稳定 API 和产品体验。
2. **开源 / 开放权重强厂商**：DeepSeek、Qwen、GLM、Meta Llama、Mistral 等，优势是成本、私有化部署、可控性和二次开发。
3. **应用和平台型厂商**：百度、腾讯、字节、Kimi、MiniMax 等，优势是中文场景、办公 / 搜索 / 多媒体工具、国内合规和云服务集成。

如果是个人学习或做 RagFlow 这类 RAG 项目，最实用的选型思路是：

- **要综合能力和工具链**：优先看 OpenAI / Claude / Gemini。
- **要低成本和中文能力**：优先看 DeepSeek / Qwen / Kimi / GLM。
- **要私有化或本地部署**：优先看 Qwen / DeepSeek / GLM / Llama / Mistral。
- **要国内云平台和企业集成**：看百度千帆、阿里云百炼、腾讯混元、火山方舟。

## 怎么理解“模型强不强”

不要只看一个总榜。大模型能力至少要拆成几层：

1. **基础语言能力**：能不能读懂问题，表达是否自然。
2. **推理能力**：能不能处理多步骤逻辑、数学、规划和复杂代码。
3. **工具使用能力**：能不能稳定输出函数参数，能不能根据工具结果继续完成任务。
4. **上下文能力**：能不能处理长文档，是否会漏掉中间信息。
5. **工程稳定性**：API 是否稳定、延迟是否可控、价格是否能接受、错误是否好处理。
6. **生态能力**：有没有 SDK、文档、云服务、监控、权限、评测、企业支持。

对一个 AI 应用开发者来说，“最强模型”不一定是“最适合模型”。例如：

- 做高质量复杂代码：Claude / OpenAI / Gemini / Qwen-Coder / GLM / DeepSeek 都值得实测。
- 做中文长文档问答：Kimi / Qwen / DeepSeek / GLM / 百度千帆会很有竞争力。
- 做低成本 RAG：DeepSeek、Qwen、GLM、豆包、Gemini Flash 类模型更适合作为默认模型。
- 做企业内网知识库：开放权重模型加私有化部署，往往比闭源 API 更容易通过合规。

## 如何评估大模型的“智力能力”

这里说的“智力”不是人类 IQ，而是模型在一组任务上的综合能力。更准确地说，是看模型能不能理解问题、推理、调用知识、遵守指令、处理复杂上下文，并且稳定地产生可用结果。

### 1. 看知识和考试能力

这类评估会让模型回答学科题、考试题、专业题。

常见指标：

- **MMLU / MMLU-Pro**：多学科选择题，覆盖数学、历史、法律、医学、计算机等。
- **C-Eval / CMMLU**：中文多学科能力评测，更适合观察中文知识和中国语境。
- **GPQA**：偏研究生 / 博士级别的科学问题，常用来测高难专业知识和推理。
- **Humanity's Last Exam**：更难的综合知识评测，用来拉开前沿模型差距。

适合判断：

- 模型知识面是否广；
- 专业问题是否靠谱；
- 中文学科题表现是否稳定。

局限：

- 选择题容易被训练数据污染；
- 会做题不等于能做好真实项目；
- 单个分数不能代表所有能力。

### 2. 看数学和逻辑推理能力

这类评估看模型能不能进行多步推导，而不是只靠记忆。

常见指标：

- **GSM8K**：小学到初中难度的数学应用题。
- **MATH / AIME / OlympiadBench**：更高难度数学、竞赛数学和奥赛类题目。
- **ARC-AGI**：抽象图形推理，关注模型能不能归纳规则。

适合判断：

- 模型是否能拆解多步骤问题；
- reasoning / thinking 模式是否真的有效；
- 模型面对陌生题时是否会乱猜。

局限：

- 数学强不代表产品体验强；
- 有些题库已经被大量模型训练或对齐过；
- 最好看解题过程和最终答案，而不是只看准确率。

### 3. 看代码能力

代码能力是大模型能力差距最容易被真实验证的方向之一，因为代码可以运行、测试、复现。

常见指标：

- **HumanEval / MBPP**：短函数代码生成。
- **SWE-bench**：在真实 GitHub issue 上修复 bug，更接近真实软件工程。
- **LiveCodeBench**：持续更新的代码题，降低题库污染。
- **Aider / Terminal-Bench / WebDev Arena**：更偏真实开发、命令行操作或网页生成。

适合判断：

- 模型能不能读懂需求；
- 能不能跨文件理解项目；
- 能不能写出可运行、可测试、可维护的代码；
- Agent 类 coding 工具是否真的能完成任务。

局限：

- 短题榜单不能代表真实工程能力；
- 代码模型要看上下文长度、工具使用、测试修复能力；
- 前端、后端、算法、数据分析的能力可能差异很大。

### 4. 看指令遵循和稳定输出

很多应用不需要模型会奥数，而是需要它听话、稳定、格式正确。

常见指标：

- **IFEval**：检查模型是否严格遵守明确指令。
- **JSON / Structured Output 测试**：看模型能不能稳定输出指定 schema。
- **Tool Calling 测试**：看模型能不能正确选择工具、填参数、使用工具结果。

适合判断：

- 信息抽取是否稳定；
- 后端接口能不能接住模型输出；
- Agent 工作流是否容易失控。

局限：

- 榜单上的指令遵循，不一定等于你的业务格式一定稳定；
- 生产环境仍然需要 schema 校验、重试和兜底逻辑。

### 5. 看长上下文能力

长上下文不是只看“最大支持多少 token”，而是看模型在长文档中能不能找到关键事实、保持一致性，并且不忽略中间信息。

常见指标：

- **LongBench / LongBench v2**：长文本理解、摘要、问答、推理。
- **Needle-in-a-Haystack**：在长上下文里找一条隐藏信息。
- **InfiniteBench / LOFT**：更长、更复杂的上下文任务。

适合判断：

- 长文档问答；
- 合同 / 论文 / 日志分析；
- 多轮对话记忆和资料定位。

局限：

- “针在草堆”测试太简单，不能代表真实文档理解；
- 上下文越长，成本、延迟和遗忘风险越高；
- 真实知识库系统通常仍然需要 RAG。

### 6. 看多模态能力

多模态模型不仅要会看图，还要能理解图表、截图、表格、视频和音频。

常见指标：

- **MMMU / MMMU-Pro**：多学科视觉理解。
- **MathVista**：图像中的数学和图表推理。
- **MMBench / Open VLM Leaderboard**：视觉语言模型综合评测。
- **VideoMMMU**：视频理解和跨帧推理。

适合判断：

- 图片问答；
- UI 截图理解；
- 图表分析；
- 视频内容理解；
- OCR + 结构化信息抽取。

局限：

- 看图能力和 OCR 能力不是一回事；
- 图像理解强，不代表视频、音频、文档解析都强。

### 7. 看事实性、安全性和拒答能力

真实应用里，模型不能只追求“会回答”，还要知道什么时候不能答、什么时候要引用来源。

常见指标：

- **TruthfulQA**：测试模型是否会复述常见错误观念。
- **SimpleQA**：短事实问答，关注事实准确性。
- **Safety / Jailbreak Benchmarks**：测试越狱、防滥用和安全边界。
- **RAG Faithfulness / Groundedness**：测试回答是否基于给定资料。

适合判断：

- 客服、医疗、法律、金融等高风险场景；
- 知识库问答；
- 是否容易幻觉；
- 是否会把不确定内容说得很肯定。

局限：

- 安全策略会影响可用性；
- 拒答太多会显得“笨”，拒答太少又容易出错；
- 需要结合具体业务设置边界。

## 怎么正确看榜单

榜单很有用，但不能迷信。看榜单时可以按下面几个问题过滤：

1. **评测任务是什么**：聊天、代码、数学、长上下文、多模态，测的不是同一种能力。
2. **评分方式是什么**：人工投票、自动判题、LLM-as-a-Judge、专家打分，误差来源不同。
3. **数据是否可能污染**：老 benchmark 很可能被训练进模型，分数会虚高。
4. **是否只看平均分**：平均分高的模型，未必在你的任务上最好。
5. **是否考虑成本和延迟**：生产系统不是比赛，价格、速度、稳定性很关键。
6. **是否区分模型版本**：同一个模型名可能有 thinking / non-thinking、不同上下文、不同供应商版本。
7. **是否能复现**：如果榜单不给 prompt、样本、评测脚本，只能作为弱参考。

## 自己做模型评测的简单方法

真正选型时，最好准备一份自己的小评测集。规模不用大，但要贴近业务。

可以按这个结构准备：

| 评测类型  | 准备样本                          | 看什么                               |
| --------- | --------------------------------- | ------------------------------------ |
| 普通问答  | 10 个常见问题                     | 回答是否清楚、准确、自然             |
| 中文表达  | 10 个中文改写 / 总结任务          | 中文语感、结构、是否啰嗦             |
| RAG 问答  | 20 个基于文档的问题               | 是否命中资料、是否引用来源、是否幻觉 |
| 代码任务  | 5 个真实项目小需求                | 能否读懂现有代码、改动是否正确       |
| JSON 输出 | 20 条信息抽取任务                 | 格式是否稳定、字段是否缺失           |
| 工具调用  | 10 个需要查库 / 搜索 / 计算的问题 | 是否选对工具、参数是否正确           |
| 拒答测试  | 10 个资料不足或越权问题           | 是否会乱编、是否能说明不知道         |

最后不要只记录“对 / 错”，最好记录这几个指标：

- **准确率**：答案是否正确。
- **稳定性**：重复多次是否一致。
- **幻觉率**：是否编造资料。
- **格式成功率**：JSON / schema 是否可解析。
- **延迟**：首 token 时间和完整响应时间。
- **成本**：每 1000 次请求大约多少钱。
- **可维护性**：错误时是否容易重试和兜底。

用自己的话说：榜单解决的是“先看谁值得试”，自己的评测解决的是“谁适合我的项目”。
