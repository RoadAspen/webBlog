# AI 工程学习体系：从 0 到 RagFlow

这一系列文章用于记录个人从 0 学习 AI 工程、并落地 RAG 知识库项目的全过程。

目标不是只记住概念，而是形成一套可复用的 AI 应用开发能力：

```text
理解原理 -> 输出文章 -> 落地功能 -> 复盘验证 -> 形成体系
```

> 开始时间：2026-04-15  
> 项目方向：RAG 个人知识库系统  
> 技术栈：Next.js + NestJS + PostgreSQL + pgvector + LLM API  
> 学习方式：理论学习、项目实践、知识输出同步推进

---

## 1. 总体定位

本计划是一条面向前端开发者的 AI 工程实践路线。

背景假设：

- 有前端开发经验，熟悉 JavaScript / TypeScript。
- 接触过 Node.js、数据库、Python 或 Java。
- AI 基础较弱，之前没有系统接触过向量检索、知识库、RAG。
- 希望通过真实项目理解 AI 应用开发，而不是只停留在 API 调用。

最终目标：

- 搭建一个可运行的 RAG 个人知识库 MVP。
- 系统理解 LLM、Embedding、向量检索、RAG、Prompt、幻觉治理和 AI 应用架构。
- 输出一组能沉淀个人理解的技术文章。
- 形成一个可以放入作品集、简历和面试表达的项目。

## 2. 执行原则

每个阶段只关注三件事：

```text
项目进度
完成标准

```

每个知识主题都同时代表两件事：

```text
我要学会什么
我要写出哪篇文章
```

执行节奏：

```text
学习一个知识主题
-> 用自己的话写成文章
-> 在 RagFlow 项目中实现一个相关功能
-> 用项目结果验证理解是否正确
```

优先级说明：

- P0：当前阶段必须掌握，并且直接服务 RAG MVP。
- P1：重要能力，MVP 完成后继续增强。

### 阶段 3：Embedding 语义搜索 Demo

周期：第 3 周

项目进度：

1. 调用 Embedding API，将文本转成向量。
2. 在 PostgreSQL + pgvector 中存储向量。
3. 输入查询语句，返回最相似的文本片段。
4. 先完成语义搜索 Demo，不接入 LLM 生成。

完成标准：

- 输入一句自然语言查询，可以召回语义相关文本。
- 能解释相似度、距离、TopK 的含义。

输出文档：

- Embedding 的本质：把文本放进语义空间。
- 为什么语义相似可以被计算。
- Embedding + 向量数据库原理。
- TopK、相似度阈值和召回结果过滤。

---

### 阶段 4：文档入库 Pipeline

周期：第 4-5 周

项目进度：

1. 创建知识库。
2. 上传 txt / md 文档。
3. 后端解析文档文本。
4. 按策略切分 Chunk。
5. 为每个 Chunk 生成 Embedding。
6. 将 Chunk 和向量写入数据库。
7. 前端展示文档状态和 Chunk 列表。

完成标准：

- 上传文档后可以自动完成解析、切片、向量化。
- 可以在前端看到文档的处理状态。
- 可以查看每个文档被切成了哪些 Chunk。
- 召回测试可以基于已入库文档返回相关片段。

输出文档：

- 文档入库 Pipeline：上传、解析、切片、向量化。
- 文本切分 Chunking 策略。
- Embedding + 向量数据库原理。
- AI 系统和传统系统的核心差异。

---

### 阶段 5：RAG 问答闭环

周期：第 6-7 周

项目进度：

1. 实现 RagService。
2. 用户问题转 Embedding。
3. 在指定知识库内检索相关 Chunk。
4. 将检索结果拼入 Prompt。
5. 调用 LLM 生成回答。
6. 返回 answer + sources。
7. 支持知识库问答和自由对话两种模式。

完成标准：

- 用户可以选择知识库后提问。
- AI 能基于知识库内容回答。
- 回答可以返回引用来源。
- 当资料不足时，系统不会强行编造答案。

输出文档：

- RAG 整体架构：从 0 到 1 理解检索增强生成。
- Prompt 模板化与复用。
- 上下文窗口机制。
- 幻觉产生的根本原因。

### 阶段 6：引用来源与拒答

周期：第 8 周

项目进度：

1. 回答附带引用来源。

4. 资料不足时明确拒答。
5. Prompt 中约束模型只能基于给定资料回答。

完成标准：

- 每条知识库回答都可以追溯来源。
- 低相关问题会提示知识库中没有可靠依据。
- 用户可以看到答案是基于哪些原文片段生成的。

输出文档：

- 引用来源：为什么 RAG 回答必须可追溯。
- 资料不足拒答：如何避免模型强行编造。

## 5. 进阶计划：生产级 RAG 深化

项目进度：

输出文档：

- PDF 文档解析为什么比 TXT 难。

### 第 14 周：混合检索

项目进度：

- 增加关键词检索。
- 实现向量检索 + 关键词检索的简单融合。

完成标准：

- 同一个问题可以同时看到向量检索和关键词检索结果。

输出文档：

- 混合检索为什么比纯向量检索更稳。

完成标准：

- 能看到高级召回策略对测试集的影响。
- 能说清楚它提升的是召回、排序还是生成质量。

输出文档：

- Rerank 在 RAG 中解决什么问题。
- HyDE 检索策略是什么。
- Parent-Child Chunking 如何提升召回质量。

### 第 16 周：评估体系深化

项目进度：

完成标准：

## 7. 单篇文章写作模板

每篇文章建议使用统一结构：

```text
1. 我为什么要学这个
2. 这个概念解决什么问题
3. 用初学者能理解的话解释它
4. 它在 RagFlow 项目中对应哪个功能
5. 我在实现中遇到了什么问题
6. 最后用自己的话总结
```

示例结构：

```markdown
# 标题

## 1. 问题背景

## 2. 核心概念

## 3. 一个简单例子

## 4. 在 RagFlow 中如何落地

## 5. 踩坑与理解

## 6. 总结
```
