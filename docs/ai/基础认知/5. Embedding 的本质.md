# Embedding 的本质：构建语义空间

## 一、先说结论

当我们说"Embedding"(向量化)的时候,本质上是在做一件事情:

> 把文字、图片、代码 等非结构化信息, 转换成一串数字坐标,让计算机能够判断内容之间的语义关系.

简单来说: Embedding 是为了让计算机可以理解内容之间的**相似性**的一种方式

## 为什么需要 Embedding?

在计算机的世界中只有 0 和 1, 它不能直接理解文字、图片等非结构化数据。

eg:

```
我喜欢苹果手机

apple phone is great!
```

这两句话虽然表达的意思相似, 但是对计算机来说,他们就只是两串不同的字符串。

因此, 需要有一种方式:

> 把这些文字转换成可以表达**语义关系**的数字

这就是 Embedding 存在的意义

## Embedding 到底做了什么?

输入一句话:

> "我喜欢苹果手机"

经过 Embedding 模型:

```
我喜欢苹果手机
    ↓
Embedding Model
    ↓
[0.1, 0.9, 0.3, 0.7, ...] // 向量

```

这串数字就是**向量(Vector)**

Embedding 做了这样一件事 :

> 接收一串文字,经过 Embedding 之后, 最终输出了一串**多维数字**, 这一串数字就是向量。

## 向量到底表示了什么?

从数学这个科目来讲,我们都学习过向量, **向量是一个有方向和长度的箭头**, 通常我们接触到的是向量都是二维或者三维的,比如 (2, 3) 或者 (1, 2, 3), 我们可以构建一个二维或者三维的坐标系来表示这些向量。

但是 Embedding 输出的向量通常是 100 维、300 维甚至上千维的, 这样的向量无法用坐标系来表示, 但我们可以抽象理解为它是一个**高维空间中的一个点**。

在 Embedding 中, 这些数字被称为**维度(Dimension)**

很多人会问:
第 1 个数字代表什么?
第 100 个数字代表什么意思?

答案是: **单个数字不表示任何具体含义**

真正重要的是:

> **向量与向量之间的距离**

一个向量代表了一个词的语义,无数个向量构成了一个**语义空间**

## 什么是语义空间

Embedding 模型在训练过程中，会不断调整每个词、每句话在向量空间中的位置。相似的语义靠得更近，不同的语义分得更远——这就形成了一个**语义空间**。

> **语义空间是如何形成的？为什么用余弦而不是欧氏距离来衡量相似度？**
>
> 详见第六篇：[为什么"语义相似"可以被计算](6.为什么语义相似可以被计算.md)

## 整个流程串起来

向量存储:

```
输入文本
↓
Embedding 模型
↓
生成向量
↓
存入向量数据库
```

向量搜索

```
输入文本
↓
Embedding 模型
↓
计算向量距离
↓
找到语义最相近的内容
```

## 实际技术栈参考

本项目的 Embedding 方案：

- **Embedding 模型**：智谱 GLM embedding-3 — 负责将文本转换为向量
- **向量存储**：PostgreSQL pgvector 扩展 — 负责向量存储与相似度检索

pgvector 支持多种距离算法（COSINE / L2 / INNER PRODUCT），可以直接在 SQL 里跑向量检索，不需要引入额外的向量数据库。

## 一个非常关键的理解:

很多人以为 Embedding 输出的是

- 关键词
- 标签
- 分类

其实都不是, Embedding 输出的是**在语义空间中的一串能代表语义的坐标点数字**, 比较的是不同向量之间的夹角大小。

```
用户提问：

请假需要提前多久？

即使知识库里写的是：

员工请假需要提前一天提交申请。

Embedding 仍然能够判断：

这两个内容表达的是相近的意思。
```

基于这样的能力,支撑起了 RAG 知识库和大模型 等技术的语义理解能力。

## 为什么需要向量搜索

- 传统关键词匹配无法处理语义相似性问题
- 向量搜索可以理解语义相似度
- 通过计算向量夹角，找出最接近的语义内容
- 这样就能实现更智能的问答系统

## Embedding 和 LLM 的关系

一个常见的误解是把 Embedding 模型和生成模型（GPT、GLM 等）混为一谈。它们是**两套完全不同的模型**：

|                 | Embedding 模型           | 生成模型（LLM）                    |
| --------------- | ------------------------ | ---------------------------------- |
| 做什么          | 把文本变成向量           | 根据上下文生成文本                 |
| 输出            | 一组固定维度的数字       | 一个 Token 接一个 Token 地生成文字 |
| 能不能聊天      | ❌ 不会回答问题          | ✅ 可以对话                        |
| 在 RAG 里的角色 | 负责"找"（检索相关文档） | 负责"说"（基于文档生成回答）       |

> **一句话：Embedding 模型负责编码语义，LLM 负责生成文字。RAG 系统需要两者配合。**

更深一层的差异：

1. 传统 Embedding 模型（如 GLM embedding-3）：最终只保留一个向量来表示整段文本。
2. LLM 内部的 Embedding 层：每个 Token 都有自己的向量，而且这些向量会随着上下文不断变化（这是 Transformer 做的事，详见原理机制模块）。

## 总结

Embedding 的本质是：

> 把文本转换成向量，并通过向量之间的距离来表示语义相似度。

它不会回答问题，也不会生成文本。
它最核心的作用就是：

> 让计算机能够"理解"哪些内容意思相近，从而找到最相关的信息。
