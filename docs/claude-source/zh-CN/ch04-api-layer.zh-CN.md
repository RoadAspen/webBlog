# 第 4 章：与 Claude 交谈 — API 层

第 3 章规定了状态所在的位置以及两层如何通信。现在我们跟踪使用该状态时会发生什么：系统需要与语言模型对话。 Claude Code 中的一切——引导序列、状态系统、权限框架——都是为了服务于这一刻而存在的。

该层比系统的任何其他部分处理更多的故障模式。它必须通过单个透明接口通过四个云提供商。它必须构建system prompt，并在字节级上了解服务器的Prompt Cache如何工作，因为单个错误放置的部分可能会破坏价值 50,000 多个令牌的缓存。它必须通过主动故障检测来传输响应，因为 TCP 连接会悄无声息地终止。而且它必须保持会话稳定的不变量，以便会话中对功能标志的更改不会导致看不见的性能悬崖。

让我们从头到尾跟踪单个 API 调用。

## 多提供商客户端工厂

`getAnthropicClient()` 函数是所有模型通信的单一工厂。它返回一个为部署目标提供者配置的 Anthropic SDK 客户端：

调度完全由环境变量驱动，以固定的优先级顺序进行评估。所有四个特定于提供商的 SDK 类都通过“asknown as Anthropic”强制转换为“Anthropic”。源代码中的评论令人耳目一新：“我们一直在返回类型上撒谎。”这种有意的类型擦除意味着每个消费者都会看到统一的界面。代码库的其余部分永远不会在提供者上分支。

每个提供者 SDK 都是动态导入的——“AnthropicBedrock”、“AnthropicFoundry”、“AnthropicVertex”是具有自己的依赖树的重型模块。动态导入确保未使用的提供程序永远不会加载。

提供程序选择在启动时确定并存储在引导程序“STATE”中。query loop从不检查哪个提供程序处于活动状态。从 Direct API 切换到 Bedrock 是配置更改，而不是代码更改。

## buildFetch 包装器

每个出站提取都会被包装以注入“x-client-request-id”标头 - 每个请求生成的 UUID。当请求超时时，服务器绝不会为响应分配请求 ID。如果没有客户端 ID，API 团队就无法将超时与服务器端日志关联起来。这个标题弥补了这一差距。它仅发送到第一方 Anthropic 端点​​ - 第三方提供商可能会拒绝未知标头。

## 系统快速构建

system prompt符是整个系统中对缓存最敏感的工件。 Claude 的 API 提供服务器端Prompt Cache：可以缓存跨请求的相同prompt前缀，从而节省延迟和成本。 200K 令牌对话可能有 50-70K 令牌与上一轮相同。破坏该缓存会迫使服务器重新处理所有缓存。

边界之前的所有内容在会话、用户和组织之间都是相同的——它获得最高层的服务器端缓存。之后的所有内容都包含用户特定的内容并下降到每个会话缓存。

## 2^N 问题

`prompts.ts` 中的注释解释了为什么条件部分必须位于边界之后：

这里的每个条件都是一个运行时位，否则会乘以 Blake2b 前缀哈希变体 (2^N)。

边界之前的每个布尔条件都会使唯一全局缓存条目的数量加倍。三个条件创建 8 个变体；五个创建 32。静态部分故意是无条件的。编译时功能标志（由捆绑器解析）在边界之前是可接受的。运行时检查（这是俳句吗？用户是否有自动模式？）必须进行。

这种约束是不可见的，除非你违反它。一位善意的工程师在边界之前添加一个用户设置门控部分可能会默默地分割全局缓存，并使机群的即时处理成本加倍。

## 流媒体

## 基于 SDK 抽象的原始 SSE

流实现使用原始 Stream<BetaRawMessageStreamEvent> 而不是 SDK 的更高级别 BetaMessageStream。原因：“BetaMessageStream”在每个“input_json_delta”事件上调用“partialParse()”。对于具有大量 JSON 输入的工具调用（具有数百行的文件编辑），这会在每个块上从头开始重新解析不断增长的 JSON 字符串 — O(n^2) 行为。 Claude Code 本身处理工具输入累积，因此部分解析纯粹是浪费。

## 闲置看门狗

TCP 连接可能会在没有通知的情况下终止。服务器可能会崩溃，负载均衡器可能会默默地断开连接，或者公司代理可能会超时。 SDK 的请求超时仅涵盖初始获取 - 一旦 HTTP 200 到达，超时就会得到满足。如果流体停止，则没有任何东西可以捕获它。

看门狗：一个“setTimeout”，在每个接收到的块上重置。如果 90 秒内没有块到达，则流将中止，系统将退回到非流重试。 45 秒处发出警告。当看门狗触发时，它会使用客户端请求 ID 记录事件以进行关联。

## 非流式回退

当流式传输在响应中失败（网络错误、停顿、截断）时，系统会回退到同步“messages.create()”调用。这可以处理代理失败，其中代理返回带有非 SSE 正文的 HTTP 200，或者中途截断 SSE 流。

当流工具执行处于活动状态时，可以禁用回退，因为回退将重新执行整个请求并可能运行工具两次。

## Prompt Cache系统

## 三层

## Prompt Cache在三个级别上运行：

临时缓存（默认）：每个会话缓存具有服务器定义的 TTL（约 5 分钟）。所有用户都会得到这个。

1 小时 TTL：符合条件的用户可以获得延长缓存。资格由订阅状态决定，并锁存于引导状态 - 第 3 章中的“promptCache1hEligible”粘性锁存确保会话中超额翻转不会更改 TTL。

全局范围：system prompt缓存条目得到跨会话、跨组织共享。对于所有 Claude Code 用户来说，prompt的静态部分都是相同的，因此单个缓存副本可以为所有人服务。当存在 MCP 工具时，全局作用域被禁用，因为 MCP 工具定义是特定于用户的，并且会将缓存分段为数百万个唯一前缀。

## 粘性闩锁的实际应用

在请求构造期间，此处评估了第 3 章中的五个粘性锁存器。每个闩锁都以“null”开始，一旦设置为“true”，会话中将保持“true”。锁存器块上方的注释很精确：“动态 beta 标头的粘性锁存器。每个标头一旦首次发送，就会在会话的其余部分继续发送，因此会话中切换不会更改服务器端缓存密钥并破坏约 50-70K 令牌。”

请参阅第 3 章第 3.1 节，了解锁存器模式、五个特定锁存器的完整说明，以及为什么始终发送所有标头不是正确的解决方案。

## 查询模型生成器

`queryModel()` 函数是一个异步生成器（约 700 行），用于协调整个 API 调用生命周期。它产生 `StreamEvent`、`AssistantMessage` 和 `SystemAPIErrorMessage` 对象。

## 请求组装遵循仔细排序的顺序：

- 终止开关检查 - 最昂贵型号层的安全阀

- Beta 接头组件 — 特定于型号，采用粘性闩锁

- 工具模式构建 - 通过“Promise.all()”并行，延迟工具被排除直到发现

- 消息标准化 - 修复孤立的 tool_use/tool_result 不匹配，剥离多余的介质，删除陈旧的块

- system prompt区块构建——动态边界分割，分配缓存范围

- 重试包装流 - 处理 529（过载）、模型回退、思考降级、OAuth 刷新

## 产出代币上限

默认输出上限为 8,000 个代币，而不是典型的 32K 或 64K。生产数据显示，p99 产量为 4,911 个代币——标准限制超额储备 8-16 倍。当响应达到上限（<1% 的请求）时，它会以 64K 进行一次干净的重试。这在机队规模上节省了大量成本。

## 错误处理和重试

`withRetry()` 函数本身就是一个异步生成器，它会生成 `SystemAPIErrorMessage` 事件，以便 UI 可以显示重试状态。重试策略：

- 529（过载）：等待并重试，可选择降级快速模式

- 模型回退：主要模型失败，尝试回退（例如，Opus 到 Sonnet）

- 思维降级：上下文窗口溢出触发思维预算减少

- OAuth 401：刷新令牌并重试一次

生成器模式意味着重试进度（“服务器过载，5 秒后重试……”）显示为事件流的自然部分，而不是侧通道通知。

## 应用此

将Prompt Cache视为架构约束，而不是功能切换。 大多数法学硕士申请都会“打开”缓存。 Claude Code 将其视为一种设计约束，影响prompt排序、节记忆、标题锁定和配置管理。结构良好的prompt（缓存命中 50K 令牌）和结构不良的prompt（每轮都进行完全重新处理）之间的差异是系统中最大的成本杠杆。

使用危险的命名约定来命名昂贵的逃生舱口。 当代码库有一个很容易意外违反的不变量时，用响亮的前缀命名逃生舱口会起到三件事：使违规行为在代码审查中可见，强制记录（所需的原因参数），并产生对安全默认值的心理摩擦。这可以推广到缓存之外的任何具有隐形成本的操作。

使用看门狗构建流，而不仅仅是超时。 SDK 的请求超时满足 HTTP 200，但响应正文可能会在任何时候停止到达。在每个块上重置的“setTimeout”捕获了这一点。非流回退处理代理故障模式（带有非 SSE 正文的 HTTP 200、中流截断），这些模式在企业环境中比您预期的更常见。

使重试策略基于产量，而不是基于异常。 通过使重试包装器成为生成状态事件的异步生成器，调用者将重试进度显示为事件流的自然部分。模型回退模式（Opus 失败，尝试 Sonnet）对于生产弹性特别有用。

将快速路径与完整管道分开。 并非每个 API 调用都需要工具搜索、顾问集成、思考预算和流基础设施。 Claude Code 的“queryHaiku()”函数为内部操作（压缩、分类）提供了一条简化的路径，跳过了所有代理问题。具有简化界面的独立功能可防止意外的复杂性泄漏。
