# 第 14 章：输入和交互

## 原始字节，有意义的行动

在 Claude Code 中，当您按下 Ctrl+X，然后按下 Ctrl+K 时，终端会发送两个字节序列，两个字节序列之间的间隔可能为 200 毫秒。第一个是“0x18”（ASCII CAN）。第二个是“0x0B”（ASCII VT）。这些字节都不包含除“控制字符”之外的任何固有含义。输入系统必须识别出这两个字节在超时窗口内按顺序到达，构成了弦“ctrl+x ctrl+k”，它映射到动作“chat:killAgents”，该动作终止所有正在运行的sub-agent。

在原始字节和被杀死的代理之间，激活了六个系统：一个标记器分割转义序列，一个解析器将它们分类为五个终端协议，一个键绑定解析器将序列与上下文特定的绑定进行匹配，一个和弦状态机管理多键序列，一个处理程序执行操作，React 将结果状态更新批处理到单个渲染中。

困难不在于这些系统中的任何一个。它处于终端多样性的组合爆炸中。 iTerm2 发送 Kitty 键盘协议序列。 macOS 终端发送旧版 VT220 序列。 Ghostty 通过 SSH 发送 xtermmodifyOtherKeys。 tmux 可以根据其配置来吃掉、转换或传递其中任何一个。 Windows 终端对于 VT 模式有其自身的怪癖。输入系统必须从所有这些对象中生成正确的“ParsedKey”对象，因为用户不必知道他们的终端使用哪种键盘协议。

本章追溯了从原始字节到跨该领域的有意义的操作的路径。

设计理念是渐进增强和优雅退化。在支持 Kitty 键盘协议的现代终端上，Claude Code 可以获得完整的修饰符检测（Ctrl+Shift+A 与 Ctrl+A 不同）、超级键报告（Cmd 快捷键）和明确的键识别。在通过 SSH 的旧终端上，它会回退到最佳可用协议，失去一些修饰符区别，但保持核心功能完整。用户永远不会看到有关其终端不受支持的错误消息。他们可能无法使用“ctrl+shift+f”进行全局搜索，但“ctrl+r”进行历史搜索可以在任何地方使用。

## 关键解析管道

输入以字节块的形式到达标准输入。管道分阶段处理它们：

分词器是基础。终端输入是混合可打印字符、控制代码和多字节转义序列的字节流，没有显式框架。来自标准输入的单个“read()”可能会返回“\x1b[1;5A”（Ctrl+向上箭头），或者可能在一次读取中返回“\x1b”，在下一次读取中返回“[1;5A”，具体取决于从 PTY 到达字节的速度。标记生成器维护一个状态机，用于缓冲部分转义序列并发出完整的标记。

不完整序列问题是根本性的。当分词器看到一个单独的“\x1b”时，它无法知道这是 Escape 键还是 CSI 序列的开始。它缓冲字节并启动 50ms 定时器。如果没有继续到达，缓冲区将被刷新，并且“\x1b”将成为 Escape 按键。但在刷新之前，标记生成器会检查“stdin.可读长度”——如果字节在内核缓冲区中等待，则计时器会重新准备而不是刷新。这可以处理事件循环被阻塞超过 50 毫秒且连续字节已被缓冲但尚未读取的情况。

对于粘贴操作，超时时间延长至 500 毫秒。粘贴的文本可能很大并且以多个块的形式到达。

来自单个“read()”的所有解析键都在一个“reconciler.discreteUpdates()”调用中处理。这会批量进行 React 状态更新，以便粘贴 100 个字符会产生一次重新渲染，而不是 100 个。批处理至关重要：如果没有它，粘贴中的每个字符都会触发完整的协调周期 - 状态更新、协调、提交、Yoga 布局、渲染、比较、写入。如果每个周期 5 毫秒，则处理 100 个字符的粘贴需要 500 毫秒。通过批处理，相同的粘贴需要 5 毫秒一个周期。

## 标准输入管理

“App”组件通过引用计数管理原始模式。当任何组件需要原始输入（prompt、对话框、vim 模式）时，它会调用“setRawMode(true)”，从而递增计数器。当它不再需要原始输入时，它会调用“setRawMode(false)”，并递减。仅当计数器达到零时，原始模式才会被禁用。这可以防止终端应用程序中的常见错误：组件 A 启用原始模式，组件 B 启用原始模式，组件 A 禁用原始模式，然后组件 B 的输入突然中断，因为原始模式已全局禁用。

## 首次启用原始模式时，应用程序：

- 停止早期输入捕获（在 React 安装之前收集击键的引导阶段机制）

- 将标准输入置于原始模式（无行缓冲、无回显、无信号处理）

- 附加一个用于异步输入处理的“可读”监听器

- 启用括号粘贴（因此粘贴的文本是可识别的）

- 启用焦点报告（以便应用程序知道终端窗口何时获得/失去焦点）

- 启用扩展按键报告（Kitty 键盘协议 + xtermmodifyOtherKeys）

禁用后，所有这些都会以相反的顺序反转。仔细的排序可以防止转义序列泄漏 - 在禁用原始模式之前禁用扩展密钥报告可确保终端在应用程序停止解析 Kitty 编码序列后不会继续发送它们。

`onExit` 信号处理程序（通过 `signal-exit` 包）确保即使在意外终止时也会进行清理。如果进程收到 SIGTERM 或 SIGINT，处理程序将禁用原始模式，恢复终端状态，退出备用屏幕（如果处于活动状态），并在进程退出之前重新显示光标。如果没有这种清理，崩溃的 Claude Code 会话将使终端处于原始模式，没有光标，也没有回显——用户需要盲目地键入“reset”来恢复终端。

## 多协议支持

终端对于如何对键盘输入进行编码并不一致。像 Kitty 这样的现代终端模拟器会发送带有完整修饰符信息的结构化序列。通过 SSH 的传统终端发送不明确的字节序列，需要上下文来解释。 Claude Code 的解析器同时处理五种不同的协议，因为用户的终端可能是其中任何一种。

CSI u（Kitty 键盘协议）是现代标准。格式：`ESC [ 代码点 [;修饰语] u`。示例：`ESC[13;2u` 是 Shift+Enter，`ESC[27u` 是不带修饰符的 Escape。代码点明确地标识了密钥 - Escape-the-key 和 Escape-as-sequence-prefix 之间没有歧义。修饰词将 shift、alt、ctrl 和 super (Cmd) 编码为单独的位。 Claude Code 在启动时通过“ENABLE_KITTY_KEYBOARD”转义序列在支持该协议的终端上启用此协议，并在退出时通过“DISABLE_KITTY_KEYBOARD”禁用它。该协议是通过查询/响应握手来检测的：应用程序发送“CSI？” u`，终端响应“CSI?” flags u`，其中“flags”表示支持的协议级别。

xtermmodifyOtherKeys 是像 Ghostty over SSH 这样的终端的后备方案，其中不协商 Kitty 协议。格式：`ESC [ 27;修饰符；键码~`。请注意，参数顺序与 CSI u 相反 - 修饰符位于键码之前，然后是键码。这是解析器错误的常见来源。该协议通过 `CSI > 4 启用； 2 m`，当未检测到终端的 TERM 标识时由 Ghostty、tmux 和 xterm 发出（常见于 SSH，其中不转发“TERM_PROGRAM”）。

传统的终端序列涵盖了其他所有内容：通过“ESC O”和“ESC [”序列的功能键、箭头键、小键盘、Home/End/Insert/Delete，以及 40 多年终端演变中积累的 VT100/VT220/xterm 变体的完整动物园。解析器使用两个正则表达式来匹配它们：“FN_KEY_RE”用于“ESC O/N/[/[[”前缀模式（匹配功能键、箭头键及其修改变体），“META_KEY_CODE_RE”用于元键代码（“ESC”后跟单个字母数字，传统的 Alt+键编码）。

遗留序列的挑战是模糊性。 `ESC [ 1; 2 R` 可以是 Shift+F3 或光标位置报告，具体取决于上下文。解析器通过私有标记检查解决了这个问题：光标位置报告使用 `CSI ？排; col R`（带有“？”私有标记），而修改后的功能键使用“CSI params R”（不带它）。这种歧义消除就是 Claude Code 请求 DECXCPR（扩展光标位置报告）而不是标准 CPR 的原因 — 扩展形式是明确的。

终端识别又增加了一层复杂性。启动时，Claude Code 发送“XTVERSION”查询（“CSI > 0 q”）来发现终端的名称和版本。响应（`DCS > | name ST`）在 SSH 连接后仍然存在——与 `TERM_PROGRAM` 不同，后者是一个不通过 SSH 传播的环境变量。了解终端身份允许解析器处理特定于终端的怪癖。例如，xterm.js（由 VS Code 的集成终端使用）具有与本机 xterm 不同的转义序列行为，并且标识字符串 (`xterm.js(X.Y.Z)`) 允许解析器解释这些差异。

SGR 鼠标事件使用格式 `ESC [ < 按钮;坳；行 M/m`，其中“M”表示按下，“m”表示释放。按钮代码对操作进行编码：0/1/2 表示左/中/右键单击，64/65 表示滚轮向上/向下（0x40 与滚轮位或），32+ 表示拖动（0x20 与运动位或）。滚轮事件被转换为“ParsedKey”对象，以便它们流经键绑定系统；单击和拖动事件成为路由到选择处理程序的“ParsedMouse”对象。

带括号的粘贴将粘贴的内容包裹在“ESC [200~”和“ESC [201~”标记之间。标记之间的所有内容都会变成带有“isPasted: true”的单个“ParsedKey”，无论粘贴的文本可能包含什么转义序列。这可以防止粘贴的代码被解释为命令——当用户粘贴包含“\x03”（即 Ctrl+C 作为原始字节）的代码片段时，这是一个关键的安全功能。

## 解析器的输出类型形成一个干净的可区分联合：

` 分机
` 类型 ParsedKey = {
种类：“钥匙”；
名称：字符串； // 'return'、'escape'、'a'、'f1' 等
ctrl ：布尔值； 元：布尔值； 移位：布尔值；
选项：布尔值； 超级：布尔值；
序列：字符串； // 用于调试的原始转义序列
isPasted: 布尔值; // 括号内粘贴
}

类型 ParsedMouse = {
种类：'鼠标'；
按钮：数字； // SGR按钮代码
动作：“按”| '发布';
列：编号； 行：编号； // 1 索引的终端坐标
}

类型解析响应 = {
kind: '响应';
响应：终端响应； // 路由到 TerminalQuierer
} `
`

“kind”判别式确保下游代码显式处理每种输入类型。按键不会被意外地处理为鼠标事件；终端响应不会被意外地解释为按键。 `ParsedKey` 类型还携带用于调试的原始 `sequence` 字符串 - 当用户报告“按 Ctrl+Shift+A 没有任何反应”时，调试日志可以准确显示终端发送的字节序列，从而可以诊断问题是否出在终端的编码、解析器的识别或键绑定的配置中。

“ParsedKey”上的“isPasted”标志对于安全性至关重要。启用括号粘贴后，终端会将粘​​贴的内容封装在标记序列中。解析器在生成的按键事件上设置“isPasted: true”，并且按键绑定解析器会跳过粘贴按键的按键绑定匹配。如果没有这个，粘贴包含“\x03”（Ctrl+C 作为原始字节）或转义序列的文本将触发应用程序命令。使用它，粘贴的内容将被视为文字文本输入，无论其字节内容如何。

解析器还识别终端响应——终端本身发送的序列来回答查询。其中包括设备属性（DA1、DA2）、光标位置报告、Kitty 键盘标志响应、XTVERSION（终端标识）和 DECRPM（模式状态）。这些被路由到“TerminalQuerier”而不是输入处理程序：

` 分机
` 输入终端响应 =
| { 类型： 'decrpm'; 模式：数字； 状态：号码}
| { 类型：'da1'； 参数：数字[]}
| { 类型：'da2'； 参数：数字[]}
| { 类型： 'kittyKeyboard'; 标志：数字}
| { 类型：'光标位置'； 行：编号； 列：数字 }
| { 类型：'osc'； 代码：号码； 数据：字符串}
| { 类型： 'xtversion'; 版本：字符串} `
`

修饰符解码遵循 XTerm 约定：修饰词为“1 + (shift? 1: 0) + (alt? 2: 0) + (ctrl? 4: 0) + (super? 8: 0)”。 “ParsedKey”中的“meta”字段映射到 Alt/Option（位 2）。 “super”字段是不同的（位 8，macOS 上的 Cmd）。这种区别很重要，因为 Cmd 快捷方式是由操作系统保留的，无法被终端应用程序捕获 - 除非终端使用 Kitty 协议，该协议会报告其他协议默默吞下的超级修改密钥。

当间隔后 5 秒内没有输入到达时，stdin-gap 检测器会触发终端模式重新断言。这可以处理 tmux 重新连接和笔记本电脑唤醒场景，其中终端的键盘模式可能已被多路复用器或操作系统重置。当重新断言触发时，它会重新发送“ENABLE_KITTY_KEYBOARD”、“ENABLE_MODIFY_OTHER_KEYS”、括号内的粘贴和焦点报告序列。如果没有这个，从 tmux 会话分离并重新连接会默默地将键盘协议降级为旧模式，从而破坏会话其余部分的修饰符检测。

## 终端 I/O 层

解析器下面有一个结构化终端 I/O 子系统，位于“ink/termio/”中：

- csi.ts — CSI（控制序列介绍者）序列：光标移动、擦除、滚动区域、括号内粘贴启用/禁用、焦点事件启用/禁用、Kitty 键盘协议启用/禁用

- dec.ts — DEC 私有模式序列：备用屏幕缓冲区 (1049)、鼠标跟踪模式 (1000/1002/1003)、光标可见性、括号内粘贴 (2004)、焦点事件 (1004)

- osc.ts — 操作系统命令：剪贴板访问 (OSC 52)、选项卡状态、iTerm2 进度指示器、tmux/屏幕多路复用器包装（需要遍历多路复用器边界的序列的 DCS 直通）

- sgr.ts — 选择图形呈现：ANSI 样式代码系统（颜色、粗体、斜体、下划线、反色）

- tokenize.ts — 用于转义序列边界检测的状态分词器

多路复用器的包装值得一提。当 Claude Code 在 tmux 内部运行时，某些转义序列（如 Kitty 键盘协议协商）必须传递到外部终端。 tmux 使用 DCS 直通（`ESC P... ST`）来转发它不理解的序列。 `osc.ts` 中的 `wrapForMultiplexer` 函数检测多路复用器环境并适当地包装序列。如果没有这个，Ki​​tty 键盘模式将在 tmux 内默默地失败，并且用户永远不会知道为什么他们的 Ctrl+Shift 绑定停止工作。

## 事件系统

`ink/events/` 目录实现了一个与浏览器兼容的事件系统，具有七种事件类型：`KeyboardEvent`、`ClickEvent`、`FocusEvent`、`InputEvent`、`TerminalFocusEvent` 和基本 `TerminalEvent`。每个都带有“target”、“currentTarget”、“eventPhase”，并支持“stopPropagation()”、“stopImmediatePropagation()”和“preventDefault()”。

包装“ParsedKey”的“InputEvent”的存在是为了向后兼容旧的“EventEmitter”路径，较旧的组件可能仍在使用该路径。新组件使用带有捕获/冒泡阶段的 DOM 样式键盘事件调度。两条路径都从同一个已解析的键触发，因此它们始终是一致的 - 到达 stdin 的键会生成一个“ParsedKey”，它会生成一个“InputEvent”（用于传统侦听器）和一个“KeyboardEvent”（用于 DOM 样式调度）。这种双路径设计允许从 EventEmitter 模式增量迁移到 DOM 事件模式，而不会破坏现有组件。

## 键绑定系统

键绑定系统将三个经常纠缠在一起的问题分开：什么键触发什么操作（绑定）、触发操作时会发生什么（处理程序）以及哪些绑定现在处于活动状态（上下文）。

特定于平台的绑定在定义时处理。图像粘贴在 macOS/Linux 上为“ctrl+v”，但在 Windows 上为“alt+v”（其中“ctrl+v”是系统粘贴）。在支持 VT 模式的终端上，模式循环是“shift+tab”，但在不支持 VT 模式的 Windows 终端上，模式循环是“meta+m”。有条件地包含功能标记的绑定（快速搜索、语音模式、终端面板）。

用户可以通过“~/.claude/keybindings.json”覆盖任何绑定。解析器接受修饰符别名（`ctrl`/`control`、`alt`/`opt`/`option`、`cmd`/`command`/`super`/`win`）、键别名（`esc` -> `escape`、`return` -> `enter`）、和弦符号（空格分隔的步骤，如 `ctrl+k ctrl+s`）以及用于取消绑定默认键的 null 操作。空操作与不定义绑定不同 - 它显式阻止触发默认绑定，这对于想要回收密钥供其终端使用的用户来说非常重要。

当密钥到达时，解析器从当前活动上下文（由 React 组件状态确定）构建上下文列表，对其进行重复数据删除并保留优先级顺序，并搜索匹配的绑定。最后匹配的绑定获胜——这就是用户覆盖优先于默认值的方式。每次击键都会重建上下文列表（它很便宜：数组串联和最多 16 个字符串的重复数据删除），因此上下文更改会立即生效，无需任何订阅或侦听器机制。

上下文设计处理一个棘手的交互模式：嵌套模式。当运行任务期间出现权限对话框时，“确认”和“任务”上下文都可能处于活动状态。 `Confirmation` 上下文优先（它稍后在组件树中注册），因此 `y` 触发“approve”而不是任何任务级绑定。当对话框关闭时，“确认”上下文将停用，“任务”绑定将恢复。这种堆叠行为自然地从上下文列表的优先级排序中出现——不需要特殊的模式处理代码。

## 预留的快捷方式

并不是所有的事情都能反弹。系统强制实施三层预订：

不可重新绑定（硬编码行为）：“ctrl+c”（中断/退出）、“ctrl+d”（退出）、“ctrl+m”（与所有终端中的 Enter 相同 - 重新绑定会破坏 Enter）。

终端保留（警告）：`ctrl+z` (SIGTSTP)、`ctrl+\` (SIGQUIT)。这些在技术上是可以绑定的，但终端会在应用程序在大多数配置中看到它们之前拦截它们。

macOS 保留（错误）：`cmd+c`、`cmd+v`、`cmd+x`、`cmd+q`、`cmd+w`、`cmd+tab`、`cmd+space`。操作系统在它们到达终端之前拦截它们。绑定它们将创建一个永远不会触发的快捷方式。

## 解决流程

## 当密钥到达时，解析路径为：

- 构建上下文列表：组件的已注册活动上下文加上全局、重复数据删除并保留优先级

- 针对合并的绑定表调用 `resolveKeyWithChordState(input, key, contexts)`

- 在“match”上：清除任何挂起的和弦，在事件上调用处理程序“stopImmediatePropagation()”

- 在“chord_started”上：保存待处理的击键，停止传播，启动和弦超时

- 关于“chord_cancelled”：清除待处理的和弦，让事件失败

- 在“unbound”上：清除和弦 - 这是显式解除绑定（用户将操作设置为“null”），因此传播停止但没有处理程序运行

- 在“none”上：落入其他处理程序

“最后获胜”解决策略意味着，如果默认绑定和用户绑定都在“Chat”上下文中定义了“ctrl+k”，则用户的绑定优先。这是在匹配时通过按定义顺序迭代绑定并保留最后一个匹配来评估的，而不是在加载时构建覆盖映射。优点：特定于上下文的覆盖可以自然地组合。用户可以覆盖“聊天”中的“输入”，而不影响“确认”中的“输入”。

## 和弦支持

“ctrl+x ctrl+k”绑定是一个和弦：两个击键一起形成一个动作。解析器通过状态机对此进行管理。

## 当钥匙到达时：

- 解析器将其附加到任何待处理的和弦前缀

- 它检查任何绑定的和弦是否以此前缀开头。如果是这样，它返回 `chord_started` 并保存挂起的击键

- 如果完整和弦与绑定完全匹配，则返回“match”并清除挂起状态

- 如果和弦前缀不匹配，则返回“chord_cancelled”

“ChordInterceptor”组件在和弦等待状态期间拦截所有输入。它有 1000 毫秒的超时时间 — 如果第二次击键未在一秒内到达，则和弦将被取消，第一次击键将被丢弃。 `KeybindingContext` 提供了一个 `pendingChordRef` 用于同步访问挂起状态，避免了 React 状态更新延迟，这种延迟可能导致第二个击键在第一个击键的状态更新完成之前被处理。

和弦设计避免了遮蔽阅读行编辑键。如果没有和弦，“杀死代理”的键绑定可能是“ctrl+k”——但这就是 readline 的“杀死到行尾”，这是用户在终端文本输入中所期望的。通过使用 ctrl+x 作为前缀（匹配 readline 自己的和弦前缀约定），系统获得与单键编辑快捷键不冲突的绑定命名空间。

该实现处理了大多数和弦系统忽略的边缘情况：当用户按下“ctrl+x”但随后键入不属于任何和弦的字符时会发生什么？如果不小心处理，该角色就会被吞掉——和弦拦截器消耗了输入，和弦被取消，角色就消失了。在这种情况下，Claude Code 的“ChordInterceptor”返回“chord_cancelled”，这会导致待处理的输入被丢弃，但允许不匹配的字符进入正常的输入处理。性格并没有丢失；仅丢弃和弦前缀。这符合用户对 Emacs 风格和弦前缀的期望。

## Vim 模式

## 状态机

vim 实现是一个具有详尽类型检查的纯状态机。类型是文档：

` 分机
` 导出类型 VimState =
| { 模式：'插入'； 插入文本：字符串}
| { 模式：'正常'； 命令：命令状态}

导出类型 CommandState =
| { 类型：'空闲' }
| { 类型：'计数'； 数字：字符串}
| { 类型：'操作员'； op：操作员； 计数：数量}
| { 类型：'操作员计数'； op：操作员； 计数：数量； 数字：字符串}
| { 类型: 'operatorFind'; op：操作员； 计数：数量； 查找：查找类型}
| { 类型： 'operatorTextObj'; op：操作员； 计数：数量； 范围：TextObjScope }

| { 类型： '查找'; 查找：查找类型； 计数：数量}
| { 类型：'g'； 计数：数量}
| { 类型： 'operatorG'; op：操作员； 计数：数量}
| { 类型： '替换'; 计数：数量}
| { 类型：'缩进'； 目录：'>'| '<'; 计数：数量 } `
`

这是一个有 12 个变体的受歧视联盟。 TypeScript 的详尽检查确保“CommandState.type”上的每个“switch”语句都能处理所有 12 种情况。向联合添加新状态会导致每个不完整的切换产生编译错误。状态机不能有死状态或丢失转换——类型系统禁止它。

请注意每个状态如何准确地携带下一个转换所需的数据。 “operator”状态知道哪个运算符（“op”）和前面的计数。 `operatorCount` 状态添加数字累加器（`digits`）。 `operatorTextObj` 状态添加范围（`inner` 或 `around`）。没有一个国家携带它不需要的数据。这不仅仅是良好的品味 - 它可以防止处理程序从先前命令读取过时数据的整个类别的错误。如果您处于“find”状态，则会有一个“FindType”和一个“count”。您没有操作员，因为没有待处理的操作员。该类型使得不可能的状态无法表示。

## 状态图讲述了这个故事：

从“idle”状态，按“d”进入“operator”状态。在“operator”中，按“w”会通过“w”动作执行“delete”。再次按“d”（“dd”）会触发行删除。按“2”进入“operatorCount”，因此“d2w”变成“删除接下来的2个单词”。按 `i` 输入 `operatorTextObj`，因此 `di"` 变为“删除引号内”。每个中间状态都准确地承载着下一个转换所需的上下文——不多也不少。

## 作为纯函数的转换

`transition()` 函数将当前状态类型分派给 10 个处理函数之一。每个都返回一个 `TransitionResult`：

` 分机
` 类型转换结果 = {
下一个？：命令状态； // 新状态（省略 = 保持当前状态）
执行?: () => void; // 副作用（省略 = 尚未执行任何操作）
} `
`

返回副作用，而不执行。转换函数是纯粹的——给定一个状态和一个键，它返回下一个状态，并且可选地返回一个执行该操作的闭包。调用者决定何时运行效果。这使得状态机可以轻松测试：向其提供状态和键，对返回的状态进行断言，忽略闭包。这也意味着转换函数不依赖于编辑器状态、光标位置或缓冲区内容。这些细节在创建时由闭包捕获，而不是在转换时由状态机消耗。

`fromIdle` 处理程序是入口点，涵盖了完整的 vim 词汇：

- 计数前缀：‘1-9’进入‘计数’状态，累加位数。 `0` 是特殊的 - 它是“行的开头”运动，而不是计数数字，除非数字已经累积

- 运算符：“d”、“c”、“y”进入“运算符”状态，等待运动或文本对象定义范围

- Find: `f`、`F`、`t`、`T` 进入 `find` 状态，等待查找字符

- G 前缀：`g` 进入复合命令的 `g` 状态（`gg`、`gj`、`gk`）

- Replace: `r` 进入 `replace` 状态，等待替换字符

- 缩进：`>`、`<`进入`缩进`状态（对于`>>`和`<<`）

- 简单动作：`h/j/k/l/w/b/e/W/B/E/0/^/$` 立即执行，移动光标

- 立即命令：`x`（删除字符）、`~`（切换大小写）、`J`（连接行）、`p/P`（粘贴）、`D/C/Y`（操作符快捷键）、`G`（转到末尾）、`.`（点重复）、`;/,`（查找重复）、`u`（撤消）、`i/I/a/A/o/O`（进入插入模式）

## 动作、操作符和文本对象

动作是将按键映射到光标位置的纯函数。 `resolveMotion(key,cursor,count)`应用运动`count`次，如果光标停止移动则短路（不能向左移动超过第0列）。这种短路对于行尾的“3w”很重要——它在最后一个单词处停止，而不是换行或出错。

## 动作根据它们与操作员交互的方式进行分类：

- 独占（默认）— 目标处的字符不包含在范围内。 `dw` 删除直到但不包括下一个单词的第一个字符

- 包含 (`e`、`E`、`$`) — 包含目标处的字符。 `de` 删除当前单词的最后一个字符

- Linewise (`j`、`k`、`G`、`gg`、`gj`、`gk`) — 与运算符一起使用时，范围扩展到覆盖整行。 `dj` 删除当前行和下面一行，而不仅仅是两个光标位置之间的字符

运算符应用于一个范围。 `delete` 删除文本并将其保存到寄存器中。 `change` 删除文本并进入插入模式。 `yank` 不加修改地复制到寄存器。 `cw`/`cW` 特殊情况遵循 vim 约定：change-word 转到当前单词的末尾，而不是下一个单词的开头（与 `dw` 不同）。

一种有趣的边缘情况：“[Image #N]”芯片卡扣。当文字运动落在图像参考芯片（在终端中呈现为单个视觉单元）内时，范围会扩展到覆盖整个芯片。这可以防止部分删除用户认为是原子元素的内容 - 您无法删除“[Image #3]”的一半，因为运动系统将整个芯片视为单个单词。

其他命令涵盖了完整的预期 vim 词汇：“x”（删除字符）、“r”（替换字符）、“~”（切换大小写）、“J”（连接行）、“p”/“P”（按行/按字符感知粘贴）、“>>”/“<<”（缩进/减少缩进，带有 2 个空格）、“o”/“O”（在下面/上面打开行并进入插入模式）。

文本对象查找光标周围的边界。他们回答这个问题：“光标所在的‘东西’是什么？”

Word 对象（`iw`、`aw`、`iW`、`aW`）将文本分段为字素，将每个字素分类为单词字符、空格或标点符号，并将选择范围扩展到单词边界。 “i”（内部）变体仅选择单词。 “a”（周围）变体包括周围的空白 - 首选尾随空白，如果位于行尾则回退到前导。大写变体（“W”、“aW”）将任何非空白序列视为单词，忽略标点符号边界。

引用对象（`i"`、`a"`、`i'`、`a'`、`i`、`a`）在当前行查找成对的引号。对按顺序匹配（第一个和第二个报价形成一对，第三个和第四个形成下一对，依此类推）。如果光标位于第一个和第二个引号之间，则表示匹配。 `a` 变体包括引号字符； `i` 变体排除了它们。

括号对象（`ib`/`i(`、`ab`/`a(`、`i[`/`a[`、`iB`/`i{`/`aB`/`a{`、`i<`/`a<`) 对匹配分隔符进行深度跟踪搜索。它们从光标处向外搜索，保持嵌套计数，直到在深度 0 处找到匹配对。这可以正确处理嵌套括号 — `d i (` 内) `foo((bar))` 删除 `bar`，而不是 `(bar)`。

## 持久状态和点重复

vim 模式维护一个在命令间持续存在的“持久状态”——让 vim 感觉像 vim 的“内存”：

` 分机
` 接口持久状态 {
最后更改：记录更改； // 对于点重复
最后查找：{ 类型：查找类型； 字符：字符串}； // 对于;和，
寄存器：字符串； // 复制缓冲区
registerIsLinewise: 布尔值; // 粘贴行为标志
} `
`

每个变异命令将自身记录为“RecordedChange”——一个可区分的联合，涵盖插入、运算符+动作、运算符+textObj、运算符+查找、替换、删除字符、切换大小写、缩进、开行和连接。 `.` 命令从持久状态重播 `lastChange`，使用记录的计数、运算符和动作在当前光标位置重现完全相同的编辑。

查找重复（`;` 和 `,`）使用 `lastFind`。 `;` 命令以相同方向重复上次查找。 `,` 命令翻转方向：`f` 变为 `F`，`t` 变为 `T`，反之亦然。这意味着在“fa”（查找下一个“a”）之后，“;”向前查找下一个“a”，而“,”向后查找下一个“a”——用户不必记住他们正在搜索的方向。

寄存器跟踪拉出和删除的文本。当寄存器内容以“\n”结尾时，它被标记为逐行，这会更改粘贴行为：“p”插入到当前行下方（而不是光标之后），而“P”插入到上方。这种区别对于用户来说是不可见的，但对于 vim 用户经常依赖的“删除一行，将其粘贴到其他地方”工作流程至关重要。

## 虚拟滚动

长时间的Claude Code会话会产生长时间的对话。繁重的调试会话可能会生成 200 多条消息，每条消息都包含 markdown、代码块、工具使用结果和权限记录。如果没有虚拟化，React 将在内存中维护 200 多个组件子树，每个子树都有自己的状态、效果和记忆缓存。 DOM 树将包含数千个节点。瑜伽布局会在每一帧上访问所有这些。终端将无法使用。

“VirtualMessageList”组件通过仅渲染视口中可见的消息以及上方和下方的小缓冲区来解决此问题。在包含数百条消息的对话中，这就是安装 500 个 React 子树（每个子树都有 Markdown 解析、语法突出显示和工具使用块）和安装 15 个之间的区别。

## 该组件维护：

- 每条消息的高度缓存，当终端列计数更改时无效

- 用于转录搜索导航的跳转手柄（跳转到索引、下一个/上一个匹配）

- 使用热缓存支持搜索文本提取（当用户输入“/”时将所有消息预先小写）

- 粘性prompt跟踪 - 当用户滚动离开输入时，他们的最后一个prompt文本将作为上下文显示在顶部

- 消息操作导航 — 基于光标的消息选择，用于倒回功能

`useVirtualScroll` Hook根据 `scrollTop`、`viewportHeight` 和累积消息高度计算要挂载的消息。它在“ScrollBox”上维护滚动钳边界，以防止当爆发“scrollTo”调用超越 React 的异步重新渲染时出现空白屏幕——这是虚拟化列表的一个典型问题，其中滚动位置可能超出 DOM 更新。

虚拟滚动和 Markdown 令牌缓存之间的交互值得注意。当消息滚动出视口时，其 React 子树就会卸载。当用户向后滚动时，子树将重新安装。如果没有缓存，这将意味着重新解析用户滚动经过的每条消息的降价。模块级 LRU 缓存（500 个条目，以内容哈希为键）确保每个唯一消息内容最多发生一次昂贵的“marked.lexer()”调用，无论组件安装和卸载多少次。

`ScrollBox` 组件本身通过 `useImperativeHandle` 提供了一个命令式 API：

- `scrollTo(y)` — 绝对滚动，打破粘性滚动模式

- `scrollBy(dy)` — 累积到 `pendingScrollDelta` 中，由渲染器以上限速率耗尽

- `scrollToElement(el, offset)` — 通过 `scrollAnchor` 将位置读取延迟到渲染时间

- `scrollToBottom()` — 重新启用粘性滚动模式

- `setClampBounds(min, max)` — 约束虚拟滚动窗口

所有滚动突变都直接进入 DOM 节点属性并通过微任务安排渲染，绕过 React 的协调器。 “markScrollActivity()”调用向后台间隔（旋转器、计时器）发出信号以跳过下一个刻度，从而减少活动滚动期间的事件循环争用。这是一种协作调度模式：滚动路径告诉后台工作“我正在进行延迟敏感操作，请让步。”后台间隔在安排下一个刻度之前检查此标志，如果滚动处于活动状态，则延迟一帧。即使多个微调器和计时器在后台运行，结果也能始终保持平滑滚动。

## 应用此：构建上下文感知的键绑定系统

Claude Code 的键绑定架构为任何具有模式输入的应用程序（编辑器、IDE、绘图工具、终端多路复用器）提供了模板。关键见解：

将绑定与处理程序分开。 绑定是数据（哪个键映射到哪个操作名称）。处理程序是代码（操作触发时会发生什么）。将它们分开意味着可以将绑定序列化为 JSON 以供用户自定义，而处理程序保留在拥有相关状态的组件中。用户可以将“ctrl+k”重新绑定到“chat:submit”，而无需触及任何组件代码。

语境是一个一流的概念。 定义根据应用程序状态激活和停用的上下文，而不是单一的键盘映射。当对话框打开时，“确认”上下文将激活，并且其绑定优先于“聊天”绑定。当对话框关闭时，“聊天”绑定将恢复。这消除了分散在事件处理程序中的“if (dialogOpen && key === 'y')”条件汤。

和弦状态作为显式机器。 多键序列（和弦）不是单键绑定的特殊情况 - 它们是一种不同类型的绑定，需要具有超时和取消语义的状态机。明确这一点（使用专用的“ChordInterceptor”组件和“pendingChordRef”）可以防止出现微妙的错误，即和弦的第二次击键被不同的处理程序消耗，因为 React 的状态更新尚未传播。

提早预订，明确警告。 在定义时而不是在解析时识别无法反弹的键（系统快捷键、终端控制字符）。当用户尝试绑定“ctrl+c”时，在配置加载期间显示错误，而不是默默地接受永远不会触发的绑定。这就是有效的键绑定系统和产生神秘错误报告的键绑定系统之间的区别。

终端多样性设计。 Claude Code 的键绑定系统在绑定级别而不是处理程序级别定义了特定于平台的替代方案。图像粘贴是“ctrl+v”或“alt+v”，具体取决于操作系统。模式循环为“shift+tab”或“meta+m”，具体取决于 VT 模式支持。无论哪个键触发每个操作，其处理程序都是相同的。这意味着测试涵盖每个操作的一个代码路径，而不是每个平台键组合的一个代码路径。当出现新的终端怪癖时（例如，Node 24.2.0 之前的 Windows 终端缺少 VT 模式），修复方法是绑定定义中的单个条件，而不是处理程序代码中的一组分散的“if (platform === 'windows')”检查。

提供逃生舱口。 空动作解除绑定机制虽小但很重要。在终端多路复用器内运行 Claude Code 的用户可能会发现“ctrl+t”（切换待办事项）与多路复用器的选项卡切换快捷方式冲突。通过将 `{ "ctrl+t": null }` 添加到 keybindings.json，他们完全禁用了绑定。按键传递到多路复用器。如果没有空解除绑定，用户唯一的选择就是将“ctrl+t”重新绑定到他们不想要的其他操作，或者重新配置他们的多路复用器——这两者都不是一个好的体验。

vim 模式实现又增加了一课：让类型系统强制执行你的状态机。 12 变体“CommandState”联合使得我们不可能忘记 switch 语句中的状态。 “TransitionResult”类型将状态变化与副作用分开，使机器可以作为纯函数进行测试。如果您的应用程序具有模式输入，请将模式表示为可区分联合，并让编译器验证详尽性。定义类型所花费的时间在消除运行时错误方面是值得的。

考虑替代方案：使用可变状态和命令式条件的 vim 实现。 “fromOperator”处理程序将是“if (mode === 'operator' &&endingCount!== null && isDigit(key))”检查的嵌套，每个分支都会改变共享变量。添加新状态（例如，宏记录模式）需要审核每个分支以确保新状态得到处理。对于可区分联合，编译器会进行审核 - 添加新变体的 PR 在每个 switch 语句都处理它之前不会构建。

这是 Claude Code 输入系统的更深层次的教训：在每一层——分词器、解析器、键绑定解析器、vim 状态机——该架构都会尽早将非结构化输入转换为类型化、详尽处理的结构。原始字节在解析器边界成为“ParsedKey”。 `ParsedKey` 成为键绑定边界处的操作名称。操作名称成为组件边界处的类型化处理程序。每次转换都会缩小可能状态的空间，并且每次缩小都是由 TypeScript 的类型系统强制执行的。当击键到达应用程序逻辑时，歧义就消失了。不存在“如果密钥未定义怎么办？”的问题。不存在“如果修饰符组合不可能怎么办？”的问题。这些类型已经禁止这些状态的存在。

两章合在一起讲述一个故事。第 13 章展示了渲染系统如何消除不必要的工作——位块传输未更改的区域、保留重复值、在单元级别进行比较、跟踪损坏边界。第 14 章展示了输入系统如何消除歧义——将五个协议解析为一种类型，根据上下文绑定解析键，将模态状态表示为详尽的联合。渲染系统回答“如何每秒绘制 24,000 个单元 60 次？”输入系统回答“如何在分散的生态系统中将字节流转化为有意义的操作？”这两个答案都遵循相同的原则：将复杂性推向边界，可以一次正确地处理它，以便下游的所有内容都在干净、类型化、边界明确的数据上运行。终端一片混乱。申请是订单。边界代码完成了将一种代码转换为另一种代码的艰苦工作。

## 总结：两种系统，一种设计理念

第 13 章和第 14 章介绍了终端接口的两部分：输出和输入。尽管关注点不同，但这两个系统都遵循相同的架构原则。

实习和间接。 渲染系统将字符、样式和超链接实习到池中，在整个热路径中用整数比较替换字符串比较。输入系统将转义序列实习为解析器边界处的结构化“ParsedKey”对象，在整个处理程序路径中用类型化字段访问替换字节级模式匹配。

分层消除工作。 渲染系统堆叠了五种优化（脏标志、blit、损坏矩形、单元级差异、补丁优化），每一种都消除了一类不必要的计算。输入系统堆叠了三个（标记生成器、协议解析器、键绑定解析器），每个都消除了一类歧义。

纯函数和类型化状态机。 vim 模式是一个带有类型转换的纯状态机。键绑定解析器是从（键、上下文、和弦状态）到解析结果的纯函数。渲染管道是从（DOM树，上一个屏幕）到（新屏幕，补丁）的纯函数。副作用发生在边界处——写入标准输出，分派到 React——而不是在核心逻辑中。

跨环境的优雅降级。 渲染系统适应终端尺寸、替代屏幕支持和同步更新协议可用性。输入系统适应 Kitty 键盘协议、xtermmodifyOtherKeys、传统 VT 序列和多路复用器直通要求。这两个系统都不需要特定的终端来运行；两者在功能更强大的终端上都会变得更好。

这些原则并非特定于终端应用程序。它们适用于任何必须在各种运行时环境中处理高频输入并产生低延迟输出的系统。终端恰好是一个限制足够严格的环境，违反这些原则会立即产生明显的退化——掉帧、吞咽击键、闪烁。这种锐利使其成为一位优秀的老师。

下一章从 UI 层转向协议层：Claude Code 如何实现 MCP——让任何外部服务成为一流工具的通用工具协议。终端 UI 处理用户体验的最后一英里——将数据结构转换为屏幕上的像素，将击键转换为应用程序操作。 MCP 处理第一英里的可扩展性 - 发现、连接和执行位于代理自己的代码库之外的工具。在它们之间，记忆系统（第 11 章）和Skill/Hook 系统（第 12 章）定义了智能和控制层。整个系统的质量上限取决于这四个因素：再多的模型智能也无法弥补滞后的 UI，再多的渲染性能也无法弥补无法达到所需工具的模型。
