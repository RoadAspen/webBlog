# 第 18 章：我们学到了什么

## 五个架构赌注

Claude Code 并不是唯一的智能体系统，也不是第一个。但它做出了五个关键的架构选择，使它在众多智能体框架中显得很不一样。经历了近两千个文件和十七章的拆解之后，这些选择已经值得回头认真检验。

## 赌注 1：用生成器循环取代回调

大多数智能体框架都会给你一条标准管线：定义工具、注册处理器、把编排交给框架。开发者写回调，框架决定什么时候调用它们。

Claude Code 走的是另一条路。`query()` 是一个异步生成器，循环本身掌握在开发者手里。模型流式地产生响应，生成器吐出工具调用，调用方执行后再把结果补回去，随后进入下一轮。每次交互都经过同一个函数、同一条数据流、同一个入口。生成器返回类型里那 10 个终止状态和 7 个延续状态，把系统可能出现的结果编码得很清楚。循环本身，就是系统。

我们打赌，单个生成器函数，即使是增长到 1,700 行的生成器函数，也会比分布式回调图更容易理解。研究来源后，赌注得到了回报。当您想了解会话结束的原因时，您可以查看一个函数。当您想要添加新的终止状态时，您可以将一种变体添加到一个可区分联合中。类型系统强制执行详尽的处理。回调架构会将这种逻辑分散在数十个文件中，并且回调之间的交互将是隐式的，而不是在控制流中可见。

## 赌注 2：基于文件的内存优于数据库

第 11 章详细介绍了该案例，但其架构意义超出了记忆范围。使用纯 Markdown 文件而不是 SQLite、矢量数据库或云服务的决定是对透明度而非功能的赌注。数据库将支持更丰富的查询、更快的查找和事务保证。文件没有提供这些。文件提供的是信任。

在 vim 中打开 `~/.claude/projects/myapp/memory/MEMORY.md` 并准确查看代理记住的内容的用户与必须询问代理“你记得什么？”的用户与系统的关系有着根本不同的关系。并希望答案是完整的。基于文件的设计使得代理的知识状态可以从外部观察，而不仅仅是自我报告。这比查询性能更重要。 LLM 支持的召回系统通过检索智能弥补了存储的简单性——从清单中选择五个相关记忆的 Sonnet 侧查询比嵌入相似性更精确，并且需要零基础设施。

## 赌注 3：自描述工具优于中央协调器

代理框架通常提供一个工具注册表：您在中央配置中描述您的工具，框架将它们呈现给模型。 Claude Code 的工具自我描述。每个“Tool”对象都有自己的名称、描述、输入模式、prompt贡献、并发安全标志和执行逻辑。工具系统的工作不是向模型描述工具，而是让工具描述自己。

这个赌注在可扩展性方面得到了回报。 MCP 工具（第 15 章）通过实现相同的接口成为一等公民。 MCP 服务器中的工具和内置工具对于模型来说是无法区分的。系统不需要单独的“MCP工具适配器”层——包装产生一个标准的“Tool”对象，从那时起，现有的工具管道处理它：权限检查、并发执行、结果预算、Hook拦截。

## 赌注 4：分叉代理以实现缓存共享

第 9 章介绍了分叉机制：sub-agent在其上下文窗口中以父代理的完整对话开始，共享父代理的Prompt Cache。这不是一种方便的优化——这是一种架构赌注，即缓存共享模型值得分叉生命周期管理的复杂性。

另一种选择是生成一个新的代理并提供对话摘要，这种方法更简单，但成本更高。每个新代理都需要支付从头开始处理其上下文的全部成本。分叉代理可以免费获取父级的缓存前缀（输入令牌可享受 90% 的折扣），从而可以经济地生成用于小任务的代理：内存提取、代码审查、验证通过。后台内存提取代理（第 11 章）在每次query loop之后运行，其成本恰恰是微不足道的，因为它共享父级的缓存。如果没有基于分叉的缓存共享，该代理将非常昂贵。

## 赌注 5：对插件的迷恋

大多数可扩展性系统都使用插件——注册功能并在主机进程中运行的代码。 Claude Code 使用Hook——在生命周期点运行的外部进程，并通过退出代码和 stdin/stdout 上的 JSON 进行通信。

可以肯定的是，进程隔离的开销是值得的。插件可能会使主机崩溃。一个Hook会导致它自己的进程崩溃。插件可能会将内存泄漏到主机堆中。Hook的内存随着它的进程而消失。插件需要一个必须进行版本控制和维护的 API 接口。Hook需要 stdin、stdout 和退出代码——这是一个自 1971 年以来一直稳定的协议。

开销是真实存在的：每次调用Hook都要拉起一个进程，代价是几毫秒，而进程内回调则没有这笔成本。第 12 章提到的内部回调快速路径可减少约 70% 的开销，这说明系统很清楚这笔成本的重要性。但对于外部Hook，比如用户脚本、团队检查或企业策略服务，进程隔离换来的安全性仍然值得。企业可以放心部署基于Hook的策略执行逻辑，而不用担心某个格式错误的脚本把开发者会话直接带崩。

## 什么转移，什么不转移

Claude Code中的所有模式并非都具有概括性。有些是规模、资源或其他代理构建者可能不共享的特定限制的结果。

## 转移到任何代理的模式

生成器循环模式。 任何需要流式传输响应、处理工具调用和管理多个终端状态的代理都可以通过显式循环而不是将其隐藏在回调后面来受益。可区分的联合返回类型（准确编码循环停止的原因）是一种消除整个类“代理为什么停止？”的模式。调试会话。

基于文件的内存与 LLM 调用。 具体的实现细节是 Claude Code 的，但简单存储与智能检索相结合的原则适用于任何需要跨会话保存知识的代理。四类分类法（用户、反馈、项目、参考）和可推导性测试（“可以从当前项目状态重新推导出来吗？”）是可重用的设计启发法。

用于远程执行的非对称读/写通道。 当读取是高频流而写入是低频 RPC 时，无论具体的传输协议如何，将它们分开都是正确的。

用于搜索的位图预过滤器。 任何搜索大文件索引的代理都会受益于 26 位字母位图作为预过滤器。每个条目四个字节，每个候选者一次整数比较——成本效益比非常显着。

Prompt Cache稳定性是一个架构问题。 如果您的代理使用带有Prompt Cache的 API，那么首先使用稳定内容、最后使用易失性内容构建prompt并不是一种优化，而是决定您的成本结构的架构决策。

## 特定于 Claude Code 规模的模式

分叉的终端渲染器。 Claude Code 分叉了 Ink 并使用打包类型数组、基于池的实习和单元级比较重新实现了渲染管道，因为它需要在终端中进行 60fps 的流传输。大多数代理都会呈现到 Web 界面或简单的日志输出。仅当终端渲染是您的主要 UI 并且您以高频率进行流式传输时，工程投资才有意义。

50 多个启动分析检查点。 当您拥有数十万用户并且 0.5% 的采样产生具有统计意义的数据时，这才有意义。对于较小的代理，更简单的计时系统就足够了。

八种 MCP 传输类型。 Claude Code 支持 stdio、SSE、HTTP、WebSocket、SDK、两个 IDE 变体和 Claude.ai 代理，因为它必须与每个部署拓扑集成。大多数代理需要 stdio 和 HTTP。

Hook快照安全模型。 在启动时冻结Hook配置并且从不隐式重新读取它是对特定威胁的防御：恶意存储库代码在用户接受信任对话框后修改Hook。当您的代理在具有不受信任的“.claude/”配置的任意存储库中运行时，这一点很重要。仅在受信任环境中运行的代理可以使用更简单的Hook管理。

## 复杂性的代价

近两千个文件。它能买什么，要花多少钱？

文件计数作为复杂性指标具有误导性。其中大部分是测试基础设施、类型定义、配置模式和分叉的 Ink 渲染器。实际的行为复杂性集中在少量高密度文件中：“query.ts”（1,700行，agent loop）、“hooks.ts”（4,900行，生命周期拦截系统）、“REPL.tsx”（5,000行，交互式编排器）以及内存系统的prompt构建函数。

## 复杂性来自三个来源，每个来源都有不同的特征：

协议多样性。 支持五种终端键盘协议、八种 MCP 传输类型、四种远程执行拓扑和七种配置范围本质上是复杂的。每个附加协议都是对代码库的线性添加，而不是指数添加——但总和很大。这种复杂性在布鲁克斯意义上是偶然的：它来自环境（终端碎片、MCP 传输演进、远程部署拓扑），而不是来自正在解决的问题。

性能优化。 基于池的渲染、位图搜索预过滤器、粘性缓存锁存器和推测工具执行都增加了复杂性，以换取可测量的性能提升。这种复杂性是通过测量来证明的——每次优化之前都会分析识别瓶颈的数据。风险在于优化累积和交互的方式使得热路径更难以修改。

行为调整。 内存系统的prompt指令、过时警告、验证协议、“忽略内存”反模式指令——这些都不是代码复杂性。它们非常复杂，并且承担着不同的维护负担。当模型的行为在版本之间发生变化时，通过 evals 仔细调整的prompt指令可能需要重新调整。评估基础设施（在整个代码库中称为案例编号和评估分数）是针对回归的防御措施，但它需要持续的投资。

该系统的维护负担很大。阅读代码库的新工程师不仅必须了解代码路径，还必须了解激发特定prompt短语的评估结果、激发特定安全检查的生产事件以及激发特定优化的性能配置文件。代码注释很详尽——许多包括评估案例编号和之前/之后的测量——但是近两千个文件中的详尽注释本身就是一种阅读负担。

## 代理系统的发展方向

从 Claude Code 的模式中可以看出四种趋势，它们指明了该领域的发展方向。

## MCP 作为通用协议

第 15 章将 Claude Code 描述为最完整的 MCP 客户端之一。重要的不是 Claude Code 的实现，而是 MCP 的存在。用于工具发现和调用的标准化协议意味着为一个代理构建的工具可以与任何代理一起使用。生态系统的影响是显而易见的：Postgres 的 MCP 服务器一旦构建，就会为每个使用 MCP 的代理提供服务。开发者在工具集成方面的投资是可移植的。

对代理构建者的影响：如果您正在定义自定义工具协议，那么您可能会犯一个错误。 MCP 已经足够好了，而且还在变得更好，而且标准协议的生态系统优势会随着时间的推移而复合。构建 MCP 客户端，为规范做出贡献，并让协议通过社区反馈不断发展。

## 多代理协调

Claude Code的sub-agent系统（第8章）、任务协调（第10章）和分叉机制（第9章）是多代理模式的早期实现。它们解决了特定的问题——缓存共享、并行探索、结构化验证——但它们也揭示了根本的挑战：协调开销。

代理之间的每条消息都会消耗令牌。每个分叉共享一个缓存，但添加一个父级最终必须协调的对话分支。任务系统的状态机（排队、运行、完成、失败、取消）是协调机制，它增加了复杂性，但没有增加功能。随着代理人的能力变得越来越强，压力将从“我们如何协调多个代理人？”转变为“我们如何协调多个代理人？”到“我们如何使一名代理人有足够的能力而无需协调？”

目前的证据表明这两种方法将共存。简单的任务将使用单个代理。复杂的任务将使用协调的多代理系统。工程挑战是使协调开销足够低，以便交叉点有利于多智能体进行真正的并行工作，而不仅仅是复杂的任务。

## 持久内存

Claude Code 的内存系统是持久代理内存的第 1 版。基于文件的设计、四种类型的分类法、LLM 支持的召回、过时系统以及用于长时间运行会话的 KAIROS 模式都是针对将显着发展的问题的第一代解决方案。

未来的记忆系统可能会增加结构化检索（当前的系统检索整个文件；未来的系统可能检索特定的事实）、跨项目迁移学习（适用于任何地方的用户偏好，不适用于的项目约定）和协作记忆（第 11 章的团队记忆是第一步，但同步、冲突解决和访问控制是最小的）。

悬而未决的问题是基于文件的方法是否可以扩展。每个项目有 200 个内存，它就可以工作。每个项目有 2,000 个内存时，Sonnet 侧查询的清单变得太大，整合变得太昂贵，并且索引超出了其上限。随着使用量的增长，数据库上的文件架构将面临最严峻的考验。

## 自主运行

KAIROS模式、后台记忆提取代理、自动梦境整合、推测工具执行——这些都是走向自主操作的步骤。代理无需询问即可完成有用的工作：它会记住您忘记告诉它记住的内容，在您睡觉时巩固自己的知识，在当前响应完成之前开始执行下一个工具。

轨迹很清晰。未来的代理将减少被动反应，并更加主动。他们会注意到用户未描述的模式，建议用户未请求的更正，并在没有明确的“/remember”命令的情况下保留自己的知识。Claude Code（Claude Code）的记忆系统及其后台提取安全网和即时设计的“保存什么”启发式方法，是这个未来的原型。

约束就是信任。自主操作要求用户相信代理在无人值守的情况下会做正确的事情。基于文件的内存、可观察的Hook 系统、过时警告、权限对话框——所有这些都存在，因为信任必须是赢得的，而不是假设的。通向更加自主的代理的途径是通过更加透明的代理。

## 闭幕式

十七章。六个核心抽象。中心的生成器循环、向外延伸的工具、随时间向后延伸的内存、守卫周边的Hook、将其全部翻译成屏幕上的字符的渲染引擎，以及将其连接到代码库之外的世界的 MCP。

Claude Code中最深刻的模式不是任何单一的技术。这是将复杂性推向极限的反复决策。渲染系统将复杂性推向池和差异 - 在管道内，一切都是整数比较。输入系统将复杂性推向了分词器和键绑定解析器——在处理程序内部，一切都是键入的操作。内存系统将复杂性推向了写入协议和调用选择器——在对话内部，一切都是上下文。agent loop将复杂性推向终端状态和工具系统——在循环内部，它只是：流、收集、执行、追加、重复。

每个边界都会吸收混乱并输出秩序。原始字节变成“ParsedKey”。 Markdown 文件成为回忆。 MCP JSON-RPC 成为“工具”对象。Hook退出代码成为权限决定。在每个边界的一侧，世界都是混乱的——五种键盘协议、脆弱的 OAuth 服务器、过时的内存、不受信任的存储库Hook。另一方面，世界是有类型的、有界限的、并且经过详尽处理的。

如果您正在构建一个代理系统，那么这是可转移的课程。不是具体的技术——您可能不需要基于池的渲染或 KAIROS 模式或八个 MCP 传输。但原则是：定义你的界限，吸收其中的复杂性，并保持它们之间的一切干净。边界是工程的难点。内部是工程令人愉快的地方。设计令人愉悦的室内空间，并将复杂性预算投入到边缘。
