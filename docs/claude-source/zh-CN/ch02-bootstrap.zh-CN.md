# 第 2 章：快速启动——Bootstrap 管道

如果说第一章为您提供了 Claude Code 的架构图，那么本章将为您提供达到工作状态所需的路线。六个抽象中的每个组件（query loop、工具系统、状态层、Hook、内存）都必须在用户看到光标之前进行初始化。所有这些的预算：300 毫秒。

三百毫秒是人类感知工具即时的阈值。越过它，CLI 就会变得迟缓。错过了很多，开发者就停止使用它。本章中的所有内容都是为了保持在这条线以下。

Bootstrap 必须完成四件事：验证环境、建立安全边界、配置通信层和渲染 UI。它必须在 300 毫秒内完成所有四项操作。架构见解是，这四项工作可以部分重叠、仔细排序并积极修剪，以适应预算，这对于如此复杂的系统来说是不可能的。

关于方法的说明：本章中的时间戳是近似值，源自代码库自己的分析检查点。它们代表了现代硬件上的典型热启动时序。冷启动速度较慢。绝对数量不如相对结构重要：哪些操作重叠，哪些操作阻塞，哪些操作被推迟。

## 管道的形状

启动管道存在于五个文件中，按顺序执行。每个文件都会缩小系统下一步需要执行的操作范围：

每个文件在将控制权传递给下一个文件之前都会执行最少的必要工作。 `cli.tsx` 尝试在导入任何重要内容之前退出。 `main.tsx` 在导入评估期间会引发缓慢的操作作为副作用。 `init.ts` 解析配置并建立信任边界。 `setup.ts` 注册功能。 `replLauncher.ts` 选择正确的入口点并启动 UI。

## 三种并行策略可以加快速度：

- 模块级子流程调度。 Fire keychain 和 MDM 在导入评估期间被视为副作用。子进程在剩余约 135 毫秒的静态导入加载时运行。

- 承诺设置中的并行性。 套接字绑定、Hook快照、命令加载和代理定义加载全部同时运行。

- 渲染后延迟预取。 用户在输入第一条消息之前不需要的所有内容（git 状态、模型功能、AWS 凭证）都会在prompt可见后运行。

第四个策略不太明显，但同样重要：动态导入以推迟模块评估。代码库至少在十几个地方使用了“await import('./module.js')”，以避免在需要时加载代码。 OpenTelemetry (400KB + 700KB gRPC) 仅在遥测初始化时加载。 React 组件仅在渲染时加载。每个动态导入都会用冷路径延迟（首次使用触发模块评估）来换取热路径速度（启动不会为其可能永远不会使用的模块付费）。

## 阶段 0：快速路径调度 (cli.tsx)

进程输入的第一个文件“cli.tsx”有一个任务：确定是否需要完整的引导管道。许多调用——“claude --version”、“claude --help”、“claude mcp list”——只需要一个具体的答案，不需要其他任何东西。加载 React、初始化遥测、读取钥匙串和设置工具系统纯粹是浪费。

模式是：检查“argv”，仅动态导入您需要的处理程序，并在系统加载其余部分之前退出。

` 分机
` // 快速路径模式的伪代码
if (args.length === 1 && args[ 0 ] === '--version' ) {
const { printVersion } =等待导入（'./commands/version.js'）
等待打印版本()
过程。 退出 (0)
} `
`

大约有十几个快速路径，涵盖版本、帮助、配置、MCP 服务器管理和更新检查。具体细节并不重要，模式才重要。每条路径仅动态导入一个模块，调用一个函数，然后退出。代码库的其余部分永远不会加载。

这是在整个引导程序中重复出现的原则的第一个实例：通过更多地了解意图来减少工作量。 argv 数组揭示了用户的意图。如果意图狭窄，执行路径也应该狭窄。

如果没有快速路径匹配，`cli.tsx` 就会进入完整的 `main.tsx` 导入，然后真正的启动开始。

## 第 1 阶段：模块级 I/O (main.tsx)

当导入“main.tsx”时，它的模块级副作用会在评估期间触发 - 在调用文件中的任何函数之前。这是整个引导程序中对性能最关键的技术：

` 分机
` // 这些在导入时运行，而不是在调用时运行
const mdmPromise = 启动MDMSubprocess ()
const keychainPromise = readKeychainCredentials () `
`

当 JavaScript 引擎评估“main.tsx”的其余部分及其传递导入（模块评估约 138 毫秒）时，这两个承诺已经在实施中。 MDM（移动设备管理）子流程检查组织安全策略。钥匙串读取获取存储的凭据。两者都是 I/O 密集型操作，否则会在关键路径上串行化。

见解：模块评估不是空闲时间——而是可以与 I/O 重叠的时间。当“main.tsx”的导出函数首次被调用时，这些承诺通常已经得到解决。

该技术需要抑制相关文件中 ESLint 的顶级等待和副作用模块范围规则。代码库有一个专门针对“process.env”访问模式的自定义 ESLint 规则，该规则允许在模块范围内控制副作用，同时防止其他地方出现不受控制的副作用。

## 第 2 阶段：解析和信任 (init.ts)

`init()` 函数是被记忆的——多次调用它是安全的并且返回相同的结果。这很重要，因为多个入口点（REPL、打印模式、SDK 模式）可能各自调用“init()”，并且记忆保证它只运行一次。

该函数通过 Commander 解析命令行参数，从多个源（全局设置、项目设置、环境变量）加载配置，然后触及管道中最重要的边界。

## 信任边界

在信任边界之前，系统以受限模式运行。之后，即可使用全部功能。边界的存在是因为Claude Code读取环境变量——而环境变量可能会被毒害。

信任边界与用户信任 Claude Code 无关。这是关于Claude Code对环境的信任。恶意“.bashrc”可以设置“LD_PRELOAD”以将代码注入每个子进程。信任对话框确保用户明确同意在可能由其他人配置的目录中进行操作。

该系统有十个不同的信任敏感操作。在用户接受信任对话框之前，仅运行安全操作：TLS 证书配置、主题首选项、遥测选择退出。信任后，系统读取潜在危险的环境变量（PATH、LD_PRELOAD、NODE_OPTIONS），执行git命令，并应用完整的环境配置。

## 预操作Hook

Commander 的“preAction”Hook是架构关键。 Commander 解析命令结构（标志、子命令、位置参数）而不执行任何操作。 `preAction` Hook在解析之后但在匹配的命令处理程序运行之前触发：

` 分机
` 程序。 Hook ( 'preAction', async ( thisCommand ) => {
等待初始化（此命令）
}) `
`

这种分离意味着快速路径命令（在 Commander 加载之前在 `cli.tsx` 中处理）永远不会支付 `init()` 成本。只有需要完整环境的命令才会触发初始化。

## 第 3 阶段：设置 (setup.ts)

`init()` 完成后，`setup()` 会注册系统所需的所有功能：

命令、代理、Hook和插件都尽可能并行注册。设置阶段是系统从“我知道我的配置”过渡到“我拥有我的所有功能”的阶段。设置完成后，每个工具都会被注册，每个Hook都会被连接，并且系统准备好处理用户输入。

安装程序还处理安全Hook快照。Hook配置从磁盘读取一次，冻结到不可变的快照中，并用于会话的其余部分。稍后对磁盘上的 hooks 配置文件的修改将被忽略。这可以防止攻击者在会话开始后修改Hook规则 - 冻结快照是权限决策的唯一真实来源。

## 第 4 阶段：启动 (replLauncher.ts)

七个不同的代码路径汇聚在 `replLauncher.ts` 上：交互式 REPL、打印模式 (`--print`)、SDK 模式、恢复 (`--resume`)、继续 (`--continue`)、管道模式和无头模式。启动器检查“init()”生成的配置并分派到正确的入口点。

## 两个例子说明了这个范围：

交互式 REPL — 标准情况。启动器挂载 React/Ink 组件树，启动终端渲染器，并进入事件循环。用户看到prompt并可以开始输入。

打印模式 (`--print`) — 来自 argv 的单个prompt。启动器创建一个没有 React 树的无头query loop，运行它直至完成，将输出流式传输到 stdout，然后退出。相同的agent loop，不同的呈现。

重要的细节：所有七个路径最终都会调用“query()”——与第 1 章中的相同agent loop。启动路径决定循环的呈现方式（交互式终端、单次、SDK 协议），而不是它的作用。这种融合使得架构可测试和可预测：无论用户如何调用 Claude Code，核心行为都是相同的。

## 绩效预算

阶段 时间 发生的情况 快速路径检查 ~5ms 检查 argv，如果可能的话尽早退出 模块评估 ~138ms 导入树，触发并行 I/O Commander 解析 ~3ms 解析标志和子命令 init() ~14ms 配置解析，信任边界 setup() ~35ms 命令、代理、Hook、插件 启动 + 首次渲染 ~25ms 选择路径、挂载 React、首次绘制 总计 ~240ms 低于 300ms 预算
在现代机器上，总时间约为 240 毫秒——在 300 毫秒的预算下还有 60 毫秒的余量。冷启动（重新启动后首次运行，操作系统缓存为空）可以将模块评估时间推至 200ms 以上，使总时间更接近极限。

## 移民系统

关于 init 期间运行的一个子系统的简要说明：架构迁移。 Claude Code 将配置和会话数据存储在本地文件和目录中。当版本之间的格式发生变化时，迁移会在启动时自动运行。

每个迁移都是一个带有版本号的函数。系统根据最高迁移版本检查当前架构版本，按顺序运行待处理的迁移，并更新版本。迁移是幂等且快速的（在小型本地文件上操作，而不是在数据库上操作）。整个迁移过程通常在 5 毫秒内完成。如果迁移失败，它会记录错误并继续 - 可用性胜过本地配置的严格一致性。

## 初创公司教授的系统设计知识

引导管道是一项缩小范围的研究。每个阶段都会减少可能性的空间：

- 第 0 阶段从“任何 CLI 调用”缩小到“需要完整引导”

- 第 1 阶段从“所有内容都必须加载”缩小为“与 I/O 并行加载”

- 第 2 阶段从“未知环境”缩小到“受信任的、已配置的环境”

- 第三阶段从“无能力”缩小到“完全注册”

- 第四阶段从“七种可能的模式”缩小到“一个具体的启动路径”

当 REPL 呈现时，所有决定都已做出。query loop接收一个完全配置的环境，对于它所处的模式、可用的工具或应用的权限没有任何歧义。 300 毫秒预算不仅仅是一个性能目标，它还是一个强制功能，可以防止引导程序成为延迟初始化系统，在该系统中决策被推迟并分散在整个代码库中。

## 应用此

将 I/O 与初始化重叠。 在模块评估时，在需要它们之前，触发慢速操作（子进程生成、凭证读取、网络检查）。无论如何，JavaScript 引擎都在进行同步工作——将这段时间用于并行 I/O。模式：“const Promise = startSlowThing()”位于文件顶部，“await Promise”位于使用点。

尽早缩小范围。 引导管道的五个文件形成一个漏斗：每个阶段都会消除后续阶段不需要做的工作。快速路径调度是最引人注目的例子，但该原则适用于任何地方。如果您可以在解析时确定代码路径是不必要的，请跳过它。

明确建立信任边界。 如果您的应用程序从它无法控制的环境（环境变量、配置文件、shell 设置）中读取数据，请在“用户同意之前安全读取”和“仅在用户同意后读取”之间划清界限。信任边界可以防止一类攻击，即恶意环境在用户有机会评估应用程序之前就毒害应用程序。

记住你的 init 函数。 使初始化幂等 - 调用它两次会产生相同的结果。当多个入口点可能各自触发初始化时，这消除了排序错误。记忆模式很简单，但消除了一整类双重初始化错误。

在产生之前捕获早期输入。 在事件驱动的系统中，初始化期间到达的用户输入可能会丢失。 Claude Code 在任何异步工作开始之前从 argv 捕获初始prompt，确保如果初始化时间比预期时间长，“claude“修复错误””不会删除prompt。
