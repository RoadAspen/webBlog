# 第 13 章：终端 UI

为什么要构建自定义渲染器？

终端不是浏览器。没有 DOM、没有 CSS 引擎、没有合成器、没有保留模式图形管道。有一个字节流发送到 stdout，还有一个字节流来自 stdin。这两个流之间的所有内容——布局、样式、比较、命中测试、滚动、选择——都必须从头开始发明。

Claude Code 需要一个响应式 UI。它具有prompt输入、流式 Markdown 输出、权限对话框、进度旋转器、可滚动消息列表、搜索突出显示和 vim 模式编辑器。 React 是声明这种组件树的明显选择。但是 React 需要一个主机环境来渲染，而终端不提供这样的环境。

Ink 是标准答案：用于终端的 React 渲染器，基于 Yoga 构建的 Flexbox 布局。 Claude Code 从 Ink 开始，然后将其分叉得面目全非。普通版本为每帧每个单元分配一个 JavaScript 对象 - 在 200x120 终端上，即每 16 毫秒创建和垃圾收集 24,000 个对象。它在字符串级别进行比较，比较 ANSI 编码文本的整行。它没有 blit 优化的概念，没有双缓冲，没有单元级脏跟踪。对于每秒刷新一次的简单 CLI 仪表板来说，这很好。对于在用户滚动浏览包含数百条消息的对话时以 60fps 传输令牌的 LLM 代理来说，这是不可能的。

Claude Code 中保留的是一个自定义渲染引擎，它共享 Ink 的概念 DNA — React 协调器、Yoga 布局、ANSI 输出 — 但重新实现了关键路径：打包类型数组而不是每个单元对象、基于池的字符串驻留而不是每个帧字符串、具有单元级差异的双缓冲渲染，以及将相邻终端写入合并到最小转义序列的优化器。

结果在 200 列终端上以 60fps 运行，同时流式传输来自 Claude 的令牌。为了理解其中的原理，我们需要检查四个层：React 协调的自定义 DOM、将该 DOM 转换为终端输出的渲染管道、基于池的内存管理（使系统在长达数小时的会话中保持活动状态而不会淹没在垃圾收集中），以及将所有这些联系在一起的组件架构。

## 自定义 DOM

React 的协调器需要一些东西来协调。在浏览器中，这就是 DOM。在 Claude Code 的终端中，它是一个自定义的内存树，具有七种元素类型和一种文本节点类型。

## 元素类型直接映射到终端渲染概念：

- `ink-root` — 文档根，每个 Ink 实例一个

- `ink-box` — 一个 Flexbox 容器，相当于一个 `<div>` 的终端

- `ink-text` — 具有用于自动换行的 Yoga 测量功能的文本节点

- `ink-virtual-text` — 另一个文本节点内的嵌套样式文本（在文本上下文内时自动从 `ink-text` 提升）

- `ink-link` — 超链接，通过 OSC 8 转义序列呈现

- `ink-progress` — 进度指示器

- `ink-raw-ansi` — 具有已知尺寸的预渲染 ANSI 内容，用于语法突出显示的代码块

## 每个“DOMElement”都携带渲染管道所需的状态：

“_eventHandlers”与“attributes”的分离是经过深思熟虑的。在 React 中，处理程序标识在每次渲染时都会发生变化（除非手动记忆）。如果处理程序存储为属性，则每次渲染都会将节点标记为脏并触发完全重绘。通过单独存储它们，协调器的“commitUpdate”可以更新处理程序而不会弄脏节点。

“markDirty()”函数是 DOM 突变和渲染管道之间的桥梁。当任何节点的内容发生更改时，“markDirty()”会遍历每个祖先，在每个元素上设置“dirty = true”，并在叶文本节点上调用“yogaNode.markDirty()”。这就是深度嵌套文本节点中的单个字符更改如何安排重新渲染到根的整个路径（但仅限于该路径）。兄弟子树保持干净并且可以从前一帧中删除。

“ink-raw-ansi”元素类型值得特别提及。当代码块已经被语法高亮显示（生成 ANSI 转义序列）时，重新解析这些序列以提取字符和样式将是浪费的。相反，预先突出显示的内容被包装在“ink-raw-ansi”节点中，其中“rawWidth”和“rawHeight”属性告诉 Yoga 确切的尺寸。渲染管道将原始 ANSI 内容直接写入输出缓冲区，而不将其分解为单独样式的字符。这使得语法突出显示的代码块在初始突出显示之后基本上为零成本——UI 中最昂贵的视觉元素也是渲染最便宜的。

“ink-text”节点的测量函数值得理解，因为它在 Yoga 的布局过程中运行，该过程是同步且阻塞的。该函数接收可用宽度，并且必须返回文本的尺寸。它执行自动换行（尊重“wrap”样式属性：“wrap”、“truncate”、“truncate-start”、“truncate-middle”）、考虑字素簇边界（因此它不会跨行分割多代码点表情符号）、正确测量 CJK 双宽字符（每个字符计为 2 列），并从宽度计算中去除 ANSI 转义码（转义序列的视觉宽度为零）。所有这些都必须在每个节点的微秒内完成，因为与 50 个可见文本节点的对话意味着每个布局传递 50 个测量函数调用。

## 反应纤维容器

协调器桥使用“react-reconciler”来创建自定义主机配置。这与 React DOM 和 React Native 使用的 API 相同。主要区别：Claude Code 在“ConcurrentRoot”模式下运行。

` 分机
` createContainer (rootNode, ConcurrentRoot,... ) `
`

ConcurrentRoot 支持 React 的并发功能 - 用于延迟加载语法突出显示的 Suspense、用于流式传输期间非阻塞状态更新的转换。替代方案“LegacyRoot”将强制同步渲染并在大量 Markdown 重新解析期间阻止事件循环。

## 主机配置方法将 React 操作映射到自定义 DOM：

- `createInstance(type, props)` 通过 `createNode()` 创建一个 `DOMElement`，应用初始样式和属性，附加事件处理程序，并捕获 React 组件所有者链以进行调试归因。所有者链存储为“debugOwnerChain”，并由“CLAUDE_CODE_DEBUG_REPAINTS”模式使用，将全屏重置归因于特定组件

- `createTextInstance(text)` 创建一个 `TextNode` — 但前提是我们位于文本上下文中。协调器强制原始字符串必须包含在“<Text>”中。尝试在文本上下文之外创建文本节点会抛出异常，在协调时而不是在渲染时捕获一类错误

- `commitUpdate(node, type, oldProps, newProps)` 通过浅比较来区分新旧 props，然后仅应用更改的内容。样式、属性和事件处理程序都有自己的更新路径。如果没有发生任何变化，diff 函数将返回“undefined”，从而完全避免不必要的 DOM 更改

- `removeChild(parent, child)` 从树中删除节点，递归释放 Yoga 节点（在 `free()` 之前调用 `unsetMeasureFunc()` 以避免访问已释放的 WASM 内存），并通知焦点管理器

- `hideInstance(node)` / `unhideInstance(node)` 切换 `isHidden` 并在 `Display.None` 和 `Display.Flex` 之间切换 Yoga 节点。这是 React 的 Suspense 回退转换机制

- `resetAfterCommit(container)` 是关键Hook：它调用 `rootNode.onComputeLayout()` 来运行 Yoga，然后调用 `rootNode.onRender()` 来安排终端绘制

协调器跟踪每个提交周期的两个性能计数器：Yoga 布局时间 (`lastYogaMs`) 和总提交时间 (`lastCommitMs`)。这些流入 Ink 类报告的“FrameEvent”，从而实现生产中的性能监控。

事件系统反映了浏览器的捕获/气泡模型。 “Dispatcher”类通过三个阶段实现完整的事件传播：捕获（根到目标）、目标处和冒泡（目标到根）。事件类型映射到 React 调度优先级 - 键盘和点击是离散的（最高优先级，立即处理），滚动和调整大小是连续的（可以推迟）。调度程序将所有事件处理包装在“reconciler.discreteUpdates()”中，以实现正确的 React 批处理。

当您在终端中按下某个键时，生成的“KeyboardEvent”将通过自定义 DOM 树分派，从焦点元素向上冒泡到根，就像键盘事件在浏览器 DOM 元素中冒泡一样。路径上的任何处理程序都可以调用“stopPropagation()”或“preventDefault()”，并且语义与浏览器规范相同。

## 渲染管线

## 每一帧都经过七个阶段，每个阶段单独计时：

每个阶段都单独计时并在“FrameEvent.phases”中报告。这种每阶段检测对于诊断性能问题至关重要：当一帧需要 30 毫秒时，您需要知道瓶颈是否是 Yoga 重新测量文本（第 2 阶段）、渲染器遍历大型脏子树（第 3 阶段）或来自慢速终端的 stdout 背压（第 7 阶段）。答案决定修复。

第一阶段：React 提交和 Yoga 布局。 协调器处理状态更新并调用“resetAfterCommit”。这会将根节点的宽度设置为“terminalColumns”并运行“yogaNode.calculateLayout()”。 Yoga 遵循 CSS Flexbox 规范，一次性计算整个 Flexbox 树：它解析所有节点上的 Flex-Grow、Flex-Shrink、填充、边距、间隙、对齐和环绕。结果——“getComputedWidth()”、“getCompulatedHeight()”、“getCompulatedLeft()”、“getCompulatedTop()”——按节点缓存。对于“ink-text”节点，Yoga 在布局期间调用自定义测量函数（“measureTextNode”），该函数通过自动换行和字形测量来计算文本尺寸。这是最昂贵的每节点操作：它必须处理嵌入在文本内容中的 Unicode 字素簇、CJK 双角字符、表情符号序列和 ANSI 转义码。

第 2 阶段：DOM 到屏幕。 渲染器深度优先遍历 DOM 树，将字符和样式写入“Screen”缓冲区。每个字符都成为一个压缩单元。输出是一个完整的框架：终端上的每个单元格都有定义的字符、样式和宽度。

第三阶段：叠加。 文本选择和搜索突出显示就地修改屏幕缓冲区，翻转匹配单元格上的样式 ID。选择应用反视频来创建熟悉的“突出显示文本”外观。搜索突出显示采用了更激进的视觉处理：当前匹配的反色 + 黄色前景 + 粗体 + 下划线，仅其他匹配的反色。这会污染缓冲区——由“prevFrameContaminate”标志跟踪，以便下一帧知道跳过 blit 快速路径。污染是一种有意的权衡：就地修改缓冲区可以避免分配单独的覆盖缓冲区（在 200x120 终端上节省 48KB），代价是清除覆盖后发生一个完全损坏的帧。

第四阶段：差异。 新屏幕与前框屏幕进行了逐个比较。只有改变的单元格才会产生输出。比较是每个单元格的两个整数比较（两个打包的“Int32”字），并且差异遍历损坏矩形而不是全屏。在稳态框架上（仅旋转器滴答作响），这可能会为 24,000 个单元格中的 3 个单元格生成补丁。每个补丁都是一个“{ type: 'stdout', content: string }”对象，其中包含光标移动序列和 ANSI 编码的单元格内容。

第五阶段：优化。 同一行上的相邻补丁将合并为单个写入。消除了多余的光标移动 - 如果补丁 N 在第 10 列结束并且补丁 N+1 在第 11 列开始，则光标已经位于正确的位置，并且不需要移动序列。样式转换通过“StylePool.transition()”缓存进行预序列化，因此从“粗红色”更改为“暗绿色”是单个缓存字符串查找，而不是比较和序列化操作。与原始的每单元输出相比，优化器通常会将字节数减少 30-50%。

第六阶段：写作。 优化的补丁被序列化为 ANSI 转义序列，并在单个“write()”调用中写入标准输出，并在支持它们的终端上封装在同步更新标记 (BSU/ESU) 中。 BSU（开始同步更新，`ESC [? 2026 h`）告诉终端缓冲所有后续输出，ESU（`ESC [? 2026 l`）告诉它刷新。这消除了支持该协议的终端上的可见撕裂——整个帧以原子方式出现。

当启用 CLAUDE_CODE_DEBUG_REPAINTS 时，全屏重置将通过 findOwnerChainAtRow() 归因于其源 React 组件。这相当于 React DevTools 的“高亮更新”的终端——它向您显示哪个组件导致整个屏幕重新绘制，这是渲染管道中可能发生的最昂贵的事情。

blit 优化值得特别关注。当节点不脏并且其位置自上一帧以来没有改变（通过节点缓存检查）时，渲染器直接将单元格从“prevScreen”复制到当前屏幕，而不是重新渲染子树。这使得稳态帧变得极其便宜——在只有旋转器在滴答作响的典型框架上，位块传输覆盖了屏幕的 99%，并且只有旋转器的 3-4 个单元从头开始重新渲染。

## blit 在三种情况下被禁用：

- `prevFrameContaminate` 为 true — 选择覆盖或搜索突出显示就地修改了前框架的屏幕缓冲区，因此不能将这些单元格视为“正确”的先前状态

- 删除了绝对定位的节点 - 绝对定位意味着节点可以在非兄弟单元上绘制，并且需要从实际拥有它们的元素重新渲染这些单元

- 布局偏移——任何节点的缓存位置与其当前计算位置不同，这意味着 blit 会将单元复制到错误的坐标

损坏矩形（“screen.damage”）在渲染期间跟踪所有写入单元的边界框。差异仅检查此矩形内的行，跳过完全未更改的区域。在流消息占据 80-100 行的 120 行终端上，diff 检查 20 行而不是 120 行——比较工作减少了 6 倍。

## Ink 类维护两个帧缓冲区：

` 分机
` 私有 frontFrame ：框架； // 当前显示在终端上

`

## 每个“框架”包含：

- `screen: Screen` — 单元缓冲区（打包 `Int32Array`）

- `viewport: Size` — 渲染时的终端尺寸

- `cursor: { x, y,visible }` — 终端光标的停放位置

- `scrollHint` — Alt-screen 模式的 DECSTBM（滚动区域）优化prompt

- `scrollDrainPending` — ScrollBox 是否还有剩余的滚动增量需要处理

每次渲染后，帧交换：`backFrame = frontFrame; frontFrame = newFrame`.旧的前框架成为下一个后框架，为 blit 优化提供“prevScreen”，并为单元级比较提供基线。

这种双缓冲区设计消除了分配。渲染器不会每帧创建一个新的“Screen”，而是重用后帧的缓冲区。交换是指针分配。该模式借用自图形编程，其中双缓冲通过确保显示器读取完整帧而渲染器写入另一帧来防止撕裂。在终端上下文中，撕裂不是问题（BSU/ESU 协议处理该问题）；值得关注的是，每 16 毫秒分配和丢弃包含 48KB 以上类型数组的“Screen”对象会带来 GC 压力。

渲染调度以 16 毫秒（约 60 fps）使用 lodash `throttle`，并启用前缘和后缘：

` 分机
` const deferredRender = () =>queueMicrotask(this.onRender);
这个.scheduleRender = 节流阀 (deferredRender, FRAME_INTERVAL_MS, {
领先：真实，
尾随： true ，
}); `
`

微任务延迟并非偶然。 `resetAfterCommit` 在 React 的布局效果阶段之前运行。如果渲染器在这里同步运行，它将错过`useLayoutEffect`中设置的光标声明。微任务在布局效果之后运行，但在相同的事件循环滴答内 - 终端看到一个单一的、一致的框架。

对于滚动操作，4ms (FRAME_INTERVAL_MS >> 2) 的单独“setTimeout”可提供更快的滚动帧，而不会干扰油门。滚动突变完全绕过 React：“ScrollBox.scrollBy()”直接改变 DOM 节点属性，调用“markDirty()”，并通过微任务安排渲染。没有 React 状态更新，没有协调开销，没有为单个轮事件重新渲染整个消息列表。

调整大小处理是同步的，而不是去抖的。当终端调整大小时，“handleResize”会立即更新尺寸以保持布局一致。对于替代屏幕模式，它会重置帧缓冲区并将“ERASE_SCREEN”推迟到下一个原子 BSU/ESU 绘制块中，而不是立即写入。同步写入擦除将使屏幕在渲染所需的约 80 毫秒内保持空白；将其推迟到原子块中意味着旧内容保持可见，直到新框架完全准备好为止。

替代屏幕管理又增加了一层。 `AlternateScreen` 组件在安装时进入 DEC 1049 备用屏幕缓冲区，限制终端行的高度。它使用“useInsertionEffect”（而不是“useLayoutEffect”）来确保“ENTER_ALT_SCREEN”转义序列在第一个渲染帧之前到达终端。使用“useLayoutEffect”就太晚了：第一帧将渲染到主屏幕缓冲区，在切换之前产生可见的闪烁。 `useInsertionEffect` 在布局效果之前和浏览器（或终端）绘制之前运行，从而实现无缝过渡。

## 基于池的内存：为什么实习很重要

200 列 x 120 行的终端有 24,000 个单元。如果每个单元格都是一个带有“char”字符串、“style”字符串和“hyperlink”字符串的 JavaScript 对象，则每帧有 72,000 个字符串分配，加上单元格本身的 24,000 个对象分配。在 60fps 下，即每秒 576 万次分配。 V8 的垃圾收集器可以处理这个问题，但不能没有显示为丢帧的暂停。 GC 暂停通常为 1-5 毫秒，但它们是不可预测的：它们可能会在流令牌更新期间发生，从而在用户观看输出时导致明显的卡顿。

Claude Code 通过打包类型数组和三个临时池完全消除了这个问题。结果：单元缓冲区的每帧对象分配为零。唯一的分配是在池本身中（摊销，因为大多数字符和样式都在第一帧上保留并随后重用）和 diff 生成的补丁字符串中（不可避免，因为 stdout.write 需要字符串或 Buffer 参数）。

单元布局每个单元使用两个“Int32”字，存储在连续的“Int32Array”中：

` 分机
` word0：charId（32 位，CharPool 索引）
word1：styleId[31:17] |超链接 ID[16:2] |宽度[1:0] `
`

同一缓冲区上的并行“BigInt64Array”视图支持批量操作 - 清除一行是对 64 位字的单个“fill()”调用，而不是对各个字段进行清零。

CharPool 将字符串转换为整数 ID。它具有 ASCII 的快速路径：128 个条目的“Int32Array”将字符代码直接映射到池索引，完全避免了“Map”查找。多字节字符（表情符号、CJK 表意文字）会转化为“Map<string, number>”。索引 0 始终为空格，索引 1 始终为空字符串。

StylePool 将 ANSI 样式代码数组实习为整数 ID。巧妙的部分：每个 ID 的位 0 编码样式是否对空格字符（背景色、反色、下划线）有可见效果。仅前景样式的 ID 为偶数；空间上可见的样式获得奇怪的 ID。这使渲染器可以通过单个位掩码检查跳过不可见的空间 - `if (!(styleId & 1) && charId === 0) continue` - 而无需查找样式定义。该池还缓存任意两个样式 ID 之间的预序列化 ANSI 转换字符串，因此从“粗红色”到“暗绿色”的转换是缓存的字符串连接，而不是比较和序列化操作。

HyperlinkPool 实习生 OSC 8 超链接 URI。索引 0 表示没有超链接。

所有三个水池均在前后框架上共享。这是一个关键的设计决策。由于池是共享的，因此驻留 ID 跨帧有效：blit 优化可以将打包单元字直接从“prevScreen”复制到当前屏幕，而无需重新驻留。 diff 可以将 ID 作为整数进行比较，而无需进行字符串查找。如果每个帧都有自己的池，则 blit 需要重新驻留每个复制的单元（通过旧 ID 查找字符串，然后将其驻留在新池中），这将抵消大部分 blit 的性能优势。

池会定期重置（每 5 分钟），以防止长时间会话期间无限制的增长。迁移过程将前框架的活细胞重新植入新鲜池中。

## CellWidth 使用 2 位分类处理双宽字符：

值 含义 0（窄）标准单列字符 1（宽）CJK/emoji 头部单元格，占据两列 2 (SpacerTail) 宽字符的第二列 3 (SpacerHead) 软换行连续标记
它存储在“word1”的低 2 位中，从而可以免费对打包单元进行宽度检查 - 一般情况下不需要进行字段提取。

## 附加的每单元元数据存在于并行数组中，而不是打包单元中：

- `noSelect: Uint8Array` — 每个单元格标志，排除文本选择中的内容。用于不应出现在复制文本中的 UI 镶边（边框、指示符）

- `softWrap: Int32Array` — 指示自动换行连续的每行标记。当用户选择软换行上的文本时，选择逻辑知道不要在换行点插入换行符

- `damage: Rectangle` — 当前帧中所有写入单元的边界框。差异仅检查此矩形内的行，跳过完全未更改的区域

这些并行数组避免扩大打包单元格式（这会增加 diff 内部循环中的缓存压力），同时提供选择、复制和优化所需的元数据。

“Screen”还公开了一个“createScreen()”工厂，该工厂获取维度和池引用。创建屏幕通过“BigInt64Array”视图上的“fill(0n)”将“Int32Array”归零——一个本地调用，可在微秒内清除整个缓冲区。这在调整大小（当需要新的帧缓冲区时）和池迁移（当旧屏幕的单元重新插入新池时）期间使用。

## REPL 组件

REPL (`REPL.tsx`) 大约有 5,000 行。它是代码库中最大的单个组件，并且有充分的理由：它是整个交互体验的协调器。一切都流经它。

## 该组件大致分为九个部分：

- 导入（约 100 行）——引入引导状态、命令、历史记录、Hook、组件、键绑定、成本跟踪、通知、群/团队支持、语音集成

- 功能标记导入 - 通过“feature()”保护和“require()”有条件加载语音集成、主动模式、简短工具和协调器代理

- 状态管理 - 广泛的“useState”调用，涵盖消息、输入模式、待处理权限、对话框、成本阈值、会话状态、工具状态和代理状态

- QueryGuard — 管理活动 API 调用生命周期，防止并发请求相互干扰

- 消息处理——处理来自query loop的传入消息、标准化排序、管理流状态

- 工具权限流——协调工具使用块和 PermissionRequest 对话框之间的权限请求

- 会话管理——恢复、切换、导出会话

- 键绑定设置 — 连接键绑定提供程序：`KeybindingSetup`、`GlobalKeybindingHandlers`、`CommandKeybindingHandlers`

- 渲染树 — 由以上所有内容组成最终 UI

## 它的渲染树组成了全屏模式下的完整界面：

“OffscreenFreeze”是针对终端渲染的性能优化。当消息滚动到视口上方时，其 React 元素将被缓存，并且其子树将被冻结。这可以防止屏幕外消息中基于计时器的更新（旋转器、经过时间计数器）触发终端重置。如果没有这个，即使用户正在查看消息 47，消息 3 中的旋转指示器也会导致完全重新绘制。

该组件自始至终都是由 React Compiler 编译的。编译器使用槽数组插入每个表达式记忆，而不是手动使用“useMemo”和“useCallback”：

` 分机
` const $=_c(14); // 14 个记忆槽
让 t0;
if ($[ 0 ]!== dep1 || $[ 1 ]!== dep2) {
t0 = 昂贵的计算(dep1, dep2);

} `
`

这种模式出现在代码库中的每个组件中。它提供了比“useMemo”（在Hook级别进行记忆）更精细的粒度——渲染函数中的各个表达式都有自己的依赖项跟踪和缓存。对于像 REPL 这样的 5,000 行组件，这可以消除每次渲染数百次潜在的不必要的重新计算。

## 选择和搜索突出显示

文本选择和搜索突出显示作为屏幕缓冲区覆盖进行操作，在主渲染之后但在差异之前应用。

文本选择仅适用于替代屏幕。 Ink 实例包含“SelectionState”跟踪锚点和焦点、拖动模式（字符/单词/行）以及捕获的已滚动到屏幕外的行。当用户单击并拖动时，选择处理程序会更新这些坐标。在“onRender”期间，“applySelectionOverlay”遍历受影响的行并使用“StylePool.withSelectionBg()”就地修改单元格样式 ID，这会返回添加了反色视频的新样式 ID。屏幕缓冲区的这种直接突变就是“prevFrameContaminate”标志存在的原因 - 前帧的缓冲区已被覆盖修改，因此下一帧不能信任它进行 blit 优化，并且必须执行完全损坏的差异。

鼠标跟踪使用 SGR 1003 模式，该模式通过列/行坐标报告单击、拖动和运动。 `App`组件实现了多次点击检测：双击选择单词，三次点击选择一行。该检测使用 500 毫秒超时和 1 个单元格位置容差（鼠标可以在单击之间移动一个单元格，而无需重置多次单击计数器）。超链接点击有意延迟此超时 - 双击链接选择单词而不是打开浏览器，这符合用户对文本编辑器的期望行为。

丢失释放恢复机制处理用户在终端内开始拖动、将鼠标移到窗口外然后释放的情况。终端报告按下和拖动，但不报告释放（发生在其窗口之外）。如果没有恢复，选择将永久停留在拖动模式。恢复的工作原理是在没有按下任何按钮的情况下检测鼠标运动事件——如果我们处于拖动状态并接收到无按钮运动事件，我们就推断按钮被释放到窗口之外并最终确定选择。

搜索突出显示有两种并行运行的机制。基于扫描的路径 (`applySearchHighlight`) 遍历可见单元格以查找查询字符串并应用 SGR 反向样式。基于位置的路径使用来自“scanElementSubtree()”的预先计算的“MatchPosition[]”，存储与消息相关的信息，并使用堆叠的 ANSI 代码（反色 + 黄色前景 + 粗体 + 下划线）将它们应用在已知的偏移量上，并使用“当前匹配”黄色突出显示。黄色前景与 inverse 结合成为黄色背景 — 当 inverse 处于活动状态时，终端会交换 fg/bg。下划线是黄色与现有背景颜色冲突的主题的后备可见性标记。

游标声明解决了一个微妙的问题。终端模拟器在物理光标位置呈现 IME（输入法编辑器）预编辑文本。 CJK 用户编写字符需要将光标置于文本输入的插入符处，而不是终端自然将其停放的屏幕底部。 `useDeclaredCursor` Hook允许组件在每一帧之后声明光标应该在哪里。 Ink 类从“nodeCache”读取声明的节点位置，将其转​​换为屏幕坐标，并在差异之后发出光标移动序列。屏幕阅读器和放大器也会跟踪物理光标，因此这种机制有利于辅助功能以及 CJK 输入。

在主屏幕模式下，声明的光标位置与“frame.cursor”分开跟踪（它必须保留在日志更新的相对移动不变量的内容底部）。在替代屏幕模式下，问题更简单：每个帧都以“CSI H”（光标主页）开始，因此声明的光标只是在帧末尾发出的绝对位置。

## 流式 Markdown

渲染 LLM 输出是终端 UI 面临的最艰巨的任务。令牌一次到达一个，每秒 10-50 个，每个令牌都会更改可能包含代码块、列表、粗体文本和内联代码的消息内容。幼稚的方法——重新解析每个标记上的整个消息——在规模上将是灾难性的。

## Claude Code 使用了三种优化：

令牌缓存。 模块级 LRU 缓存（500 个条目）存储由内容哈希键控的“marked.lexer()”结果。缓存在虚拟滚动期间能够承受 React 卸载/重新安装周期。当用户滚动回之前可见的消息时，将从缓存中提供 Markdown 令牌，而不是重新解析。

快速路径检测。 `hasMarkdownSyntax()` 通过单个正则表达式检查前 500 个字符的 Markdown 标记。如果没有找到语法，它会直接构造一个单段落标记，绕过完整的 GFM 解析器。这可以在纯文本消息上每次渲染节省大约 3 毫秒 - 当您每秒渲染 60 帧时，这一点很重要。

惰性语法突出显示。 代码块突出显示是通过 React `Suspense` 加载的。 “MarkdownBody”组件立即以“highlight={null}”作为后备渲染，然后与 cli-highlight 实例异步解析。用户立即看到代码（无样式），然后在一两帧后弹出颜色。

流媒体案例增加了一个问题。当令牌从模型到达时，降价内容会逐渐增长。在消息传输过程中，重新解析每个标记上的整个内容将是 O(n^2)。快速路径检测很有帮助——大多数流内容是纯文本段落，完全绕过解析器——但对于带有代码块和列表的消息，LRU 缓存提供了真正的优化。缓存键是内容哈希，因此当 10 个令牌到达并且只有最后一段发生更改时，将重用未更改前缀的缓存解析结果。 Markdown 渲染器仅重新解析发生更改的尾部。

“StreamingMarkdown”组件与静态“Markdown”组件不同。它处理仍在生成内容的情况：不完整的代码栅栏（没有关闭栅栏的`````）、部分粗体标记和截断的列表项。流式变体在解析方面更加宽容——它不会在未闭合语法上出错，因为闭合语法尚未到达。当消息完成流式传输时，组件将转换到静态“Markdown”渲染器，该渲染器应用完整的 GFM 解析和严格的语法检查。

代码块的语法突出显示是渲染管道中最昂贵的每元素操作。使用 cli-highlight 来突出显示 100 行代码块可能需要 50-100 毫秒。加载突出显示库本身需要 200-300 毫秒（它捆绑了数十种语言的语法定义）。这两种成本都隐藏在 React“Suspense”背后：代码块立即呈现为纯文本，突出显示库异步加载，当它解析时，代码块会用颜色重新呈现。用户可以立即看到代码，并在稍后着色——这比库加载时出现 300 毫秒的空白帧要好得多。

## 应用此：高效渲染流输出

终端渲染管道是消除工作的一个案例研究。驱动设计的三个原则：

实习一切。 如果您有一个出现在数千个单元格中的值（样式、字符、URL），请将其存储一次并通过整数 ID 引用它。整数比较是一条CPU指令。字符串比较是一个循环。当内部循环以 60fps 每帧运行 24,000 次时，整数上的 `===` 和字符串上的 `===` 之间的差异就是平滑滚动和可见滞后之间的差异。

差异在正确的水平。 单元级比较听起来很昂贵——每帧需要 24,000 次比较。但每个单元格（打包字）需要进行两次整数比较，并且在稳态框架上，差异在检查第一个单元格后会脱离大多数行。另一种方法是重新渲染整个屏幕并将其写入标准输出，每帧将产生 100KB 以上的 ANSI 转义序列。差异通常会产生不到 1KB 的大小。

将热路径与 React 分开。 滚动事件以鼠标输入频率到达（可能每秒数百次）。通过 React 的协调器（状态更新、协调、提交、布局、渲染）路由每个事件，每个事件都会增加 5-10 毫秒的延迟。通过直接改变 DOM 节点并通过微任务调度渲染，滚动路径保持在 1 毫秒以下。 React 只参与最终的绘制，无论如何它都会运行。

这些原则适用于任何流输出系统，而不仅仅是终端。如果您正在构建一个呈现实时数据的 Web 应用程序（日志查看器、聊天客户端、监控仪表板），则需要进行相同的权衡。实习生重复值。与前一帧的差异。将热路径排除在响应式框架之外。

第四个原则，特定于长时间运行的会话：定期清理。 随着新角色和风格的加入，Claude Code的池子单调增长。在几个小时的会话中，池可能会积累数千个不再被任何活动单元引用的条目。 5 分钟的重置周期限制了这种增长：每 5 分钟，就会创建新的池，前框架的单元会被迁移（重新插入新的池中），而旧的池就会变成垃圾。这是一种分代收集策略，应用于应用程序级别，因为 JavaScript GC 无法了解池条目的语义活跃度。

在普通对象上使用“Int32Array”的决定除了 GC 压力之外还有一个更微妙的好处：内存局部性。当 diff 比较 24,000 个单元格时，它会遍历连续的类型化数组。现代 CPU 预取顺序内存访问，因此整个屏幕比较在 L1/L2 缓存中运行。每个单元对象的布局会将单元分散在堆上，将每次比较都变成缓存未命中。性能差异是可测量的：在 200x120 屏幕上，类型化数组差异在 0.5 毫秒内完成，而等效的基于对象的差异需要 3-5 毫秒——与其他管道阶段结合使用时，足以超出 16 毫秒的帧预算。

第五个原则适用于渲染为固定大小网格的任何系统：跟踪损坏边界。 每个屏幕上的“damage”矩形记录了渲染期间写入的单元格的边界框。 diff 参考该矩形并完全跳过其外部的行。当流式消息占据 120 行终端的底部 20 行时，差异检查 20 行，而不是 120 行。结合 blit 优化（仅针对重新渲染区域而不是 blit 区域填充损坏矩形），这意味着常见情况（一条消息流式传输，而对话的其余部分是静态的）会触及屏幕缓冲区的一小部分。

更广泛的教训是：渲染系统的性能并不是让任何单个操作变得更快。这是关于完全消除运营。 blit 消除了重新渲染。损坏矩形消除了差异。池共享消除了重新实习。打包单元消除了分配。每次优化都会删除整个类别的工作，并且它们会以乘法方式叠加。

用数字来表示：在 200x120 终端上，最坏情况下的帧（一切都很脏，没有位块传输，全屏损坏）大约需要 12 毫秒。最好情况下的帧（一个脏节点，位块传输其他所有内容，3 行损坏矩形）需要不到 1 毫秒。系统大部分时间都处于最佳情况下。流令牌到达会触发一个脏文本节点，该节点会弄脏其祖先直至消息容器，该消息容器通常为屏幕的 10-30 行。 blit 处理其他 90-110 行。损坏矩形将差异限制在脏区域。池查找是整数运算。流式传输一个令牌的稳态成本主要由 Yoga 布局（重新测量脏文本节点及其祖先）和 Markdown 重新解析决定，而不是由渲染管道本身决定。
