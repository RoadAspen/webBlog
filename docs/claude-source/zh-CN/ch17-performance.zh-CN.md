# 第 17 章：性能——每一毫秒和令牌都很重要

## 高级工程师的剧本

代理系统中的性能优化不是一个问题。是五：

- 启动延迟 — 从击键到第一个有用输出的时间。用户放弃那些感觉启动缓慢的工具。

- 令牌效率——有用内容消耗的上下文窗口与开销的比例。上下文窗口是最受限制的资源。

- API 成本 — 每回合的美元金额。即时缓存可以将这种情况减少 90%，但前提是系统在各个轮次之间保持缓存稳定性。

- 渲染吞吐量 — 流输出期间每秒的帧数。第13章介绍了渲染架构；本章介绍了保持快速的性能测量和优化。

- 搜索速度——每次击键在 270,000 个路径的代码库中查找文件的时间。

Claude Code 使用从明显（记忆）到微妙（用于预过滤模糊搜索的 26 位位图）的技术来攻击这五种技术。关于方法论的说明：这些不是理论上的优化。 Claude Code 附带 50 多个启动分析检查点，对 100% 的内部用户和 0.5% 的外部用户进行采样。下面的每项优化都是由该仪器的数据驱动的，而不是直觉。

## API预连接

`apiPreconnect.ts` 在初始化期间向 Anthropic API 发出一个 `HEAD` 请求，将 TCP+TLS 握手（100-200ms）与设置工作重叠。在交互模式下，重叠是无限的——当用户打字时连接就会升温。该请求在“applyExtraCACertsFromConfig()”和“configureGlobalAgents()”之后触发，因此预热的连接使用正确的传输配置。

## 快速路径调度和延期进口

CLI 入口点包含专用子命令的早期返回路径——“claude mcp”从不加载 React REPL，“claude daemon”从不加载工具系统。仅在需要时才通过动态“import()”加载重型模块：OpenTelemetry（~400KB +~700KB gRPC）、事件日志记录、错误对话框、上游代理。 “LazySchema”将 Zod 模式构建推迟到首次验证，从而将成本推高到启动成本之上。

## 在上下文窗口中保存令牌

## 时隙预留：默认8K，升级64K

## 影响最大的单项优化：

默认输出槽预留为 8,000 个令牌，截断时会升级到 64,000 个。 API 为模型的响应保留容量“max_output_tokens”。默认 SDK 值为 32K-64K，但生产数据显示 p99 输出长度为 4,911 个令牌。默认超额储备 8-16 倍，每回合浪费 24,000-59,000 个代币。 Claude Code 的上限为 8K，对于罕见的截断（<1% 的请求）重试为 64K。对于 200K 窗口，可用上下文提高了 12-28%——而且是免费的。

## 工具结果预算

限制值 目的 每个工具字符 50,000 超过时将结果持久保存到磁盘 每个工具令牌 100,000 ~400KB 文本上限 每消息聚合 200,000 个字符 防止 N 个并行工具一回合超出预算
每条消息的聚合是关键的洞察力。如果没有它，“读取 src/ 中的所有文件”可能会产生 10 个并行读取，每个读取返回 40K 字符。

## 上下文窗口大小调整

默认的 200K 令牌窗口可通过模型名称或实验处理上的“[1m]”后缀扩展到 1M。当使用量接近极限时，4 层压缩系统会逐步总结旧内容。令牌计数基于 API 的实际“使用”字段，而不是客户端估计 - 考虑即时缓存积分、思考令牌和服务器端转换。

## Prompt Cache架构

Anthropic 的Prompt Cache基于精确的前缀匹配。如果单个令牌更改了中间前缀，则后面的所有内容都是缓存未命中。 Claude Code 构建了整个prompt，因此稳定的部分放在前面，不稳定的部分放在最后。

当“shouldUseGlobalCacheScope()”返回 true 时，动态边界之前的system prompt条目将获取“scope: 'global'”——运行相同 Claude Code 版本的两个用户共享前缀缓存。当存在 MCP 工具时，全局作用域被禁用，因为 MCP 模式是针对每个用户的。

## 在渲染中节省 CPU

第 13 章深入介绍了渲染架构——打包类型数组、基于池的驻留、双缓冲和单元级比较。在这里，我们重点关注保持快速的性能测量和自适应行为。

终端渲染器通过“throttle(deferredRender, FRAME_INTERVAL_MS)”将速度限制在 60fps。当终端模糊时，间隔加倍至 30fps。滚动排水框架以四分之一间隔运行，以获得最大滚动速度。这种自适应节流可确保渲染不会消耗过多的 CPU。

React 编译器（`react/compiler-runtime`）自动记忆整个代码库中的组件渲染。手动`useMemo`和`useCallback`容易出错；编译器通过构造正确地得到它。预分配的冻结对象 (`Object.freeze()`) 消除了常见渲染路径值的分配 - 在替代屏幕模式下每帧保存一个分配，复合数千帧。

有关完整渲染管道的详细信息——“CharPool”/“StylePool”/“HyperlinkPool”实习系统、blit 优化、损坏矩形跟踪、OffscreenFreeze 组件——请参阅第 13 章。

## 节省搜索内存和时间

每次击键都会运行模糊文件搜索，搜索 270,000 多个路径。三个优化层将其保持在几毫秒以内。

## 分数限制拒绝和融合索引扫描

位图中幸存的路径在昂贵的边界/驼峰命名法评分之前面临分数上限检查。如果最佳情况分数无法超过当前的 top-K 阈值，则跳过该路径。

实际匹配将位置查找与使用“String.indexOf()”的间隙/连续奖励计算融合在一起，这在 JSC (Bun) 和 V8 (Node) 中都是 SIMD 加速的。该引擎的优化搜索速度明显快于手动字符循环。

## 具有部分可查询性的异步索引

对于大型代码库，“loadFromFileListAsync()”每约 4 毫秒的工作就会产生一次事件循环（基于时间，而不是基于计数 - 适应机器速度）。它返回两个承诺：“可查询”（解析第一个块，启用立即部分结果）和“完成”（完整索引完成）。用户可以在文件列表可用后 5-10 毫秒内开始搜索。

产量检查使用 `(i & 0xff) === 0xff` — 一个无分支模 256 来分摊 `performance.now()` 的成本。

## 内存相关性侧面查询

一项优化位于代币效率和 API 成本的交叉点。如第 11 章所述，内存系统使用轻量级 Sonnet 模型调用（而不是主要 Opus 模型）来选择要包含哪些内存文件。与不包含不相关内存文件所节省的令牌相比，成本（快速模型上的最大输出令牌为 256 个）可以忽略不计。在浪费的上下文中，单个不相关的 2,000 个令牌内存的成本比 API 调用中的侧查询成本更高。

## 推测工具执行

在完整响应完成之前，“StreamingToolExecutor”开始执行流入的工具。只读工具（Glob、Grep、Read）可以并行执行；写入工具需要独占访问权限。 `partitionToolCalls()` 函数将连续的安全工具分组为批次：[Read, Read, Grep, Edit, Read, Read] 变成三个批次 - [Read, Read, Grep] 并发，[Edit] 串行，[Read, Read] 并发。

结果始终按照确定性模型推理的原始工具顺序生成。当 Bash 工具出错时，同级中止控制器会终止并行子进程，从而防止资源浪费。

## 流媒体和原始 API

Claude Code 使用原始流 API，而不是 SDK 的“BetaMessageStream”帮助程序。帮助器在每个“input_json_delta”上调用“partialParse()”——工具输入长度为 O(n^2)。 Claude Code 累积原始字符串并在块完成时解析一次。

如果没有块到达，流式看门狗（CLAUDE_STREAM_IDLE_TIMEOUT_MS，默认 90 秒）将中止并重试，并在代理失败时回退到非流式“messages.create()”。

## 应用此：代理系统的性能

审核您的上下文窗口预算。 您的“max_output_tokens”保留和实际的 p99 输出长度之间的差距是浪费的上下文。设置严格的默认值并在截断时升级。

缓存稳定性设计。 prompt中的每个字段都是稳定或不稳定的。将稳定放在第一位，将波动放在最后。将任何对话中对稳定前缀的更改视为具有美元成本的错误。

并行启动 I/O。 模块加载受 CPU 限制。钥匙串读取和网络握手受 I/O 限制。在导入之前启动 I/O。

使用位图预过滤器进行搜索。 在昂贵的评分之前，廉价的预过滤器会拒绝 10-90% 的候选者，这是每个条目 4 字节的重大胜利。

衡量重要的地方。 Claude Code 有 50 多个启动检查点，内部采样率为 100%，外部采样率为 0.5%。没有衡量的绩效工作只是猜测。

最后的观察：大多数优化在算法上并不复杂。位图预过滤器、循环缓冲区、记忆、实习——这些是 CS 基础知识。复杂之处在于知道在哪里应用它们。启动分析器会告诉您毫秒在哪里。 API 使用字段告诉您令牌在哪里。缓存命中率告诉您钱在哪里。测量第一，优化第二，始终如一。
