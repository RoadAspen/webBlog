# 第 9 章：分叉代理和Prompt Cache

## 百分之九十五的洞察力

当父代理并行生成五个sub-agent时，每个sub-agent的绝大多数 API 请求都是相同的。system prompt是一样的。工具定义是相同的。对话历史是相同的。触发生成的辅助消息是相同的。唯一不同的是最后的指令：“你处理数据库迁移”，“你编写测试”，“你更新文档”。

在具有热烈对话的典型分叉上，共享前缀可能是 80,000 个代币。每个子指令可能是 200 个令牌。即 99.75% 的重叠。 Anthropic 的Prompt Cache为缓存的输入令牌提供 90% 的折扣。如果您可以让这 80,000 个令牌命中 2 到 5 号子级的缓存，那么您只需将这四个请求的输入成本降低 90%。对于家长来说，这是在同一个并行调度上花费 4 美元和花费 0.50 美元之间的差异。

问题是Prompt Cache是字节精确的。不够“相似”。不是“语义等效”。在每个子内容出现分歧之前，从system prompt符的第一个字节到最后一个字节，字节必须逐字符匹配。一个额外的空间、一个重新排序的工具定义、一个更改system prompt片段的过时功能标志——以及缓存未命中。整个前缀按全价重新处理。

Fork 代理是 Claude Code 对这一限制的回答。它们不仅仅是“生成具有上下文的子进程”的便利——它们是伪装成编排功能的Prompt Cache利用机制。 fork 系统中的每个设计决策都可以追溯到一个问题：我们如何保证并行子项之间的字节相同的前缀？

## Fork子进程继承了什么

分叉代理从其父级继承了四件事，并且通过引用或字节精确副本继承它们，而不是通过重新计算。

1.system prompt。 不再生——螺纹化。父级已渲染的system prompt字节通过“override.systemPrompt”传递，从“toolUseContext.renderedSystemPrompt”中提取。这是在父级最近的 API 调用中发送的确切字符串。

2. 工具定义。 fork 代理定义声明了 `tools: ['*']`，但是将 `useExactTools` 标志设置为 true，子进程直接接收父进程的组装工具数组。没有过滤，没有重新排序，没有重新序列化。

3. 通话记录。 父级与 API 交换的每条消息（用户轮流、助理轮流、工具调用、工具结果）都会通过“forkContextMessages”克隆到子级上下文中。

4.思维配置和模型。 fork 定义指定了“model: 'inherit'”，它解析为父级的确切模型。相同的模型意味着相同的分词器、相同的上下文窗口、相同的缓存命名空间。

## 分叉代理定义本身是最小的——几乎是无操作：

分叉代理的定义故意被最小化——它继承了父代的所有内容。它指定所有工具（`'*'`），继承父级的模型，使用冒泡模式获取权限（因此prompt会出现在父级的终端中），并提供一个从未实际调用的无操作system prompt函数 - 真正的prompt通过覆盖通道到达，已经渲染且字节稳定。

## 字节相同前缀技巧

对 Claude 的 API 请求具有特定的结构：system prompt，然后是工具，然后是消息。为了使Prompt Cache命中，从请求开始到某个前缀边界的每个字节在请求之间必须相同。

## 分叉代理通过确保三层被冻结来实现这一点：

第 1 层：通过线程进行system prompt，而不是重新计算。

当父代理的system prompt为其最后一次 API 调用呈现时，结果将在“toolUseContext.renderedSystemPrompt”中捕获。这是所有动态插值后的字符串 - GrowthBook 功能标志、环境详细信息、MCP 服务器描述、Skill内容、CLAUDE.md 文件。分叉子进程接收这个确切的字符串。

为什么不再次调用 `getSystemPrompt()` 呢？因为system prompt符的生成并不纯粹。当 SDK 获取远程配置时，GrowthBook 标记从冷状态到热状态的转换。在父级第一次转动时返回“false”的标志可能会在 fork 子级旋转时返回“true”。如果system prompt符包含由该标志控制的条件块，则重新呈现的prompt符甚至会相差一个字符。缓存已损坏。全价再处理 80,000 个代币，乘以 5 个子代。

对渲染的字节进行线程化消除了整个类别的分歧。

第 2 层：通过精确直通进行工具定义。

普通sub-agent通过“resolveAgentTools()”，它根据代理定义的“tools”和“disallowedTools”数组过滤工具池，应用权限模式差异，并可能重新排序工具。生成的序列化工具数组将与父级工具数组不同 - 不同的子集、不同的顺序、不同的权限注释。

## Fork 代理完全跳过这个：

` 分机
` constsolvedTools = useExactTools
？ availableTools // 父级的精确数组
：resolveAgentTools（agentDefinition，availableTools，isAsync）.resolvedTools`
`

`useExactTools` 标志仅在分叉路径上设置为 true。子级按原样获取父级的工具池。相同的工具、相同的顺序、相同的序列化。这包括将代理工具本身保留在子进程的池中，即使子进程被禁止使用它 - 删除它会更改工具数组并破坏缓存。

第三层：消息数组构建。

缓存边界恰好位于最终文本块之前。其之上的所有内容（可能有数以万计的system prompt符、工具定义、对话历史记录和占位符结果）在第一个子级之后以 90% 的折扣进入缓存。

## Fork 样板标签

每个子指令都包含在样板 XML 标记中，该标签有两个用途：指示子指令如何行为，以及充当递归分叉检测的标记。

该样板包含大约 10 条规则。关键的：

- 覆盖父级的分叉指令。 父级的system prompt显示“默认为分叉”——样板明确告诉子级：“该指令是针对父级的。你是分叉。不要生成sub-agent。”

- 默默执行，报告一次。 工具调用之间没有对话文本。直接使用工具，然后生成结构化摘要。

- 保持在范围内。 孩子的扩张不能超出其指令。

- 结构化输出格式。 响应必须遵循范围/结果/关键文件/更改的文件/问题模板，以便在多个孩子同时报告时，家长可以轻松解析结果。

规则 1 特别有趣。父级的system prompt符（fork 子级出于缓存原因逐字继承）包含诸如“当您有并行工作时默认进行分叉”之类的指令。如果孩子遵循该指令，它将尝试分叉自己的孩子，从而创建代理的无限递归。样板文件明确地覆盖了：“该指令是针对家长的。你是分叉。”

结构化输出格式（范围/结果/关键文件/更改的文件/问题）不是装饰性的。它将孩子的输出限制为事实报告，这使得当五个孩子同时报告时，家长可以更轻松地解析和汇总结果。

## 递归分叉预防

fork 子进程将 Agent 工具保留在其工具池中。它必须——删除它会改变序列化工具数组并破坏Prompt Cache。但如果子进程实际上在没有“subagent_type”的情况下调用代理工具，则分叉路径将再次触发，从而创建孙子分叉。这个孙子将继承一个更大的上下文（父+子对话），产生自己的分叉，等等。

## 两名警卫阻止了这种情况：

主要守卫：querySource 检查。 当派生子进程生成时，其 `context.options.querySource` 设置为 `'agent:builtin:fork'`。 `call()` 方法在允许分叉路径之前检查这一点：

` 分机
` // 在 AgentTool.call() 中：
如果（有效类型 === 未定义）{
// 分叉路径——但是我们已经处于分叉了吗？
if (querySource === 'agent:builtin:fork' ) {
// 拒绝：已经是一个 fork 子进程
}
} `
`

这是快速路径。它检查选项对象中的单个字符串。

后备防护：消息扫描。 Fork 预防使用两个防护：在生成时设置的“querySource”标记（快速路径 - 单个字符串比较），以及扫描消息历史记录以查找样板 XML 标记的回退。回退之所以存在，是因为“querySource”在自动压缩中幸存下来，但在未正确线程化的边缘情况下，消息扫描回退会捕获递归。这是一种安全带式方法，与意外递归分叉（失控的 API 支出）的成本相比，检查（扫描消息）的成本微不足道。

为什么要后退？因为 Claude Code 有一个自动压缩功能，当上下文太长时，它会重写消息数组。 Autocompact 可以重写消息内容，但保留选项中的“querySource”。理论上，单独“querySource”就足够了。在实践中，消息扫描回退捕获了“querySource”未正确线程化的边缘情况——一种腰带式方法，其中检查（扫描消息）的成本与意外递归分叉（失控的 API 支出）的成本相比是微不足道的。

## 同步到异步的转换

分叉子进程开始在前台运行：它的消息流到父进程的终端，父进程阻塞等待完成。但如果孩子花的时间太长怎么办？ Claude Code 允许中间执行后台 - 用户（或自动超时）可以将正在运行的前台代理推入后台，而不会丢失任何工作。

## 该机制出奇地干净：

-
当通过 `registerAgentForeground()` 注册前台代理时，会创建一个后台信号承诺。

-
父级的同步循环在代理的消息流和后台信号之间竞争：

` 分机
` 同时（真）{
const 结果 = 等待 Promise.race([
iterator.next(), // 来自代理的下一条消息
backgroundSignal, // “移至后台”触发器
]）
if (结果 === Behind_SIGNAL) 中断
//... 处理消息
} `
`

-
当后台信号触发时，前台迭代器将通过“iterator.return()”优雅地终止。这会触发生成器的“finally”块，该块负责处理清理。

-
一个新的 `runAgent()` 实例通过 `isAsync: true` 生成，使用相同的代理 ID 和迄今为止积累的消息历史记录。代理从上次停止的地方继续，现在在后台运行。

-
原始同步 `call()` 返回 `{ status: 'async_launched' }`，父级继续其对话。

不会丢失任何工作，因为消息历史记录就是代理的状态。磁盘上的侧链记录包含代理生成的每条消息。新的异步实例将从该记录中重播并拾取同步实例停止的位置。

## 自动背景

当启用`CLAUDE_AUTO_BACKGROUND_TASKS`环境变量或`tengu_auto_background_agents`GrowthBook标志时，前台代理将在120秒后自动后台运行：

通过环境变量或功能标志启用时，前台代理将在 120 秒后自动进入后台。禁用时，该函数返回 0（无自动背景）。

这是一个具有成本影响的用户体验决策。 A foreground agent blocks the parent terminal — the user cannot type, cannot issue new instructions, cannot spawn other agents.两分钟足够代理同步完成大多数快速任务（其中流输出是有用的反馈），但也足够短，以至于长时间运行的任务不会占用终端。

在分叉实验下，自动后台问题没有实际意义：所有分叉生成从一开始就被强制异步。 “run_in_background”参数完全从架构中隐藏。每个 fork 子进程都在后台运行，完成后通过“<task-notification>”进行报告，并且父进程永远不会阻塞。

## 不使用叉子时

## Fork 是几种编排模式之一，在三种情况下被刻意排除：

协调器模式。 协调者模式和分叉模式是互斥的。协调器有一个结构化的委派模型：它维护一个计划，根据明确的prompt将任务分配给工作人员，并跟踪进度。 Fork 的“继承一切”方法会破坏这一点。分叉的协调器将继承父协调器的system prompt（上面写着“你是协调器，委托工作”），子协调器将尝试编排而不是执行。 `isForkSubagentEnabled()` 函数首先检查 `isCoordinatorMode()`，如果处于活动状态则返回 false。

非交互式会话。 SDK 和 API 使用者（`--print` 模式，Claude Agent SDK）无需终端即可运行。 Fork 的“permissionMode: 'bubble'”向父终端显示权限 prompt——这在非交互模式下不存在。不是构建单独的权限流，而是简单地禁用分叉路径。 SDK 使用者使用显式的“subagent_type”选择。

显式sub-agent类型。 当模型指定“subagent_type”（例如“Explore”、“Plan”、“general- Purpose”）时，不会触发分叉路径。仅当省略“subagent_type”时，Fork 才会触发。这让模型可以在“我想要一个具有自己的system prompt和工具集的专用代理”（显式类型）和“我想要一个继承上下文的自己的克隆来并行处理这个问题”（省略类型）之间进行选择。

## 经济学

考虑一个具体的场景。开发者要求 Claude Code 重构一个模块。父代理分析代码库，形成计划，并并行调度 5 个分支子级：一个用于更新数据库模式，一个用于重写服务层，一个用于更新路由器，一个用于修复测试，一个用于更新类型。

## 在对话的这一点上，共享的上下文是很重要的：

- system prompt：~4,000 代币

- 工具定义（40 多种工具）：约 12,000 个令牌

- 对话历史记录（分析+规划）：~30,000 个代币

- 带有五个 tool_use 块的助理消息：约 2,000 个代币

- 占位符工具结果：约 500 个代币

共享前缀总数：约 48,500 个令牌。每个孩子指令：~200 个令牌。

## 没有分叉（五个独立代理，每个代理都有新的上下文和自己的system prompt符）：

- 每个子进程都有自己的system prompt+工具+任务prompt

- 无缓存共享（不同的system prompt，不同的工具集）

- 成本：5 x 完整输入处理

## 使用 fork（字节相同的前缀）：

- 子 1：全价 48,700 个代币（第一次请求时缓存未命中）

- 2-5 岁儿童：48,500 个代币，价格为 10%（缓存命中）+ 每个代币全价 200 个

- 2-5 岁儿童的有效成本：约 4,850 + 200 = 每个约 5,050 个等值代币

节省的规模与上下文大小和子项数量有关。对于具有 100K 历史代币产生 8 个并行分叉的热会话，缓存节省可以超过输入代币在不共享的情况下成本的 90%。

这就是为什么 fork 系统中的每个设计决策（线程而不是重新计算、精确的工具传递、占位符结果，甚至将代理工具保留在子池中，尽管它被禁止）都针对一件事进行优化：字节相同的前缀。每个决策都会牺牲少量的优雅性或安全性来换取 API 成本的显着降低。

## 设计张力

## 分叉系统做出了值得理解的明确权衡：

隔离与缓存效率。 Fork 子进程继承一切，包括可能与其任务无关的对话历史记录。重写测试的子进程不需要父进程讨论数据库模式设计的 15 条消息。但包含这些消息使得前缀相同。剥离不相关的历史记录会节省上下文窗口空间，但代价是破坏缓存。设计的赌注是缓存节省超过上下文开销。

安全性与缓存效率。 代理工具保留在子进程的工具池中，即使子进程不得使用它。删除它会更安全（子进程甚至无法尝试分叉），但会更改工具数组序列化。样板标签和递归叉防护是补偿控制——运行时预防而不是静电消除。

简单性与缓存效率。 占位符工具的结果是一个谎言。对于父级助手消息中的每个 tool_use 块，子级都会看到“Fork 已启动 - 在后台处理”，无论这些工具调用实际执行了什么操作。这很好，因为子进程的指令告诉它要做什么——它不需要父进程的调度轮流提供准确的工具结果。但这意味着孩子的对话历史在技术上是不连贯的。选择占位符是为了简洁和统一，而不是为了准确性。

这些权衡中的每一个都反映了相同的优先级：当您为大规模 API 调用按代币付费时，字节相同的前缀值得扭曲架构。

## 应用此：快速提高缓存效率的设计

分叉代理模式的推广超越了Claude Code。任何从同一上下文分派多个并行 LLM 调用的系统都可以从缓存感知请求构造中受益。原则：

1.线程渲染prompt，不要重新计算。 如果您的system prompt包含任何动态内容（功能标志、时间戳、用户首选项、A/B 测试变体），则捕获渲染结果并按值将其传递给子级。重新计算风险分歧。

2. 冻结工具阵列。 如果您的孩子需要不同的工具集，您就放弃了工具块上的缓存共享。考虑保留完整的工具集并使用运行时防护（例如 fork 样板的“不使用代理”），而不是编译时删除。

3. 最大化共享前缀，最小化每个孩子的后缀。 构建消息数组，以便共享的所有内容都放在第一位，每个子项的内容都附加在最后。交错共享和每个子内容会导致缓存边界碎片化。

4. 对可变内容使用常量占位符。 当消息结构需要响应先前的工具调用时，请在所有子级中使用相同的占位符字符串，而不是实际（不同的）结果。

5. 衡量收支平衡。 缓存共享有开销：每个子进程有更大的上下文窗口（它们携带不相关的历史记录）、运行时防护而不是静态安全、架构复杂性。考虑额外的上下文标记后，计算您的并行模式（有多少子级、共享前缀有多大）是否真正节省了资金。

分叉代理系统的核心是一个即时缓存利用引擎。它回答了每个多代理系统构建者最终面临的一个问题：当缓存为您提供重复前缀 90% 的折扣时，您将在多大程度上重组架构以获取该折扣？Claude Code的回答是：很远。
