# 第 11 章：记忆——通过对话学习

## 无状态问题

到目前为止，每一章都描述了单个会话中存在的机制。agent loop运行，工具执行，sub-agent协调，当进程退出时，所有这些都消失了。下一次对话以相同的system prompt、相同的工具定义、相同的模型开始，并且对之前发生的事情零了解。

这是无状态架构的根本限制。开发者在周一纠正了模型的测试方法，周二模型又犯了同样的错误。用户解释他们的角色、项目的限制、他们对代码风格的偏好，并且每个新会话都要求他们再次解释。这个模型并不是健忘的——它从来不知道。每一次对话都是一个独立的宇宙。

这个问题并不只是理论上的，它会以非常具体的方式消耗用户信任。用户今天说“记住，我们的测试要连真实数据库，不要 mock”，下周模型还是会写出 mock 测试。用户明明已经说明自己是高级工程师，不需要入门级解释，下一个会话却又从教程式讲解开始。如果没有记忆，每次会话都像从零入职一样，智能体永远是第一天上班的新同事。

业界的标准解决方案是检索增强生成（RAG）：将文档嵌入到向量中，将它们存储在向量数据库中，并在查询时检索相关块。这非常适合知识库——文档、常见问题解答、参考材料。但它在架构上与代理跨会话实际需要记住的内容不匹配。智能体的记忆不是知识库。它是观察的集合：用户是谁，他们纠正了什么，项目当前的限制是什么，在哪里可以找到东西。这些观察结果很小，经常变化，并且必须是人工可编辑的。矢量数据库解决了错误的问题。

Claude Code 的内存系统完全是一个不同的赌注：磁盘上的文件、Markdown 格式、LLM 驱动的调用、没有基础设施。我们的赌注是，存储的简单性与检索的智能性相结合，会产生一个比两者都复杂的系统更好的系统。

## 设计理念会影响整个系统：

- 人类可读。 想要查看 Claude Code 记住的内容的用户可以在任何文本编辑器中打开 `~/.claude/projects/<slug>/memory/MEMORY.md`。无需特殊工具，无需解密，无需导出命令。

- 人工可编辑。 陈旧的记忆可以用 vim 来纠正。错误的内存可以用“rm”删除。用户对代理的知识拥有完全的代理权。

- 版本可控。 团队记忆可以提交到 git。内存更改 diff 很干净，因为它们是 Markdown。

- 零基础设施。 内存系统可以离线工作，无需服务器即可工作，可以在任何具有文件系统的操作系统上工作。因为没有架构，所以没有迁移路径。

- 可调试。 当内存出现异常时，诊断路径是“ls”和“cat”，而不是查询日志和数据库检查。

模型通过 `FileWriteTool` 和 `FileEditTool` 读写记忆，这和它编辑源代码时使用的是同一套工具（见第 6 章）。这里没有专门的 memory API。system prompt只是在教模型一套两步写入协议：先创建记忆文件，再更新索引；模型则用已有能力去执行这套新规则。这正体现了“工具复用”这条架构原则：记忆系统不是硬塞进智能体里的专用子系统，而是智能体在既有能力基础上涌现出的行为。

基于文件的选择在这里发挥作用还有更深层的原因。对于人工智能代理来说，内存与传统应用程序中的内存有根本的不同。传统应用程序的数据库拥有权威状态——系统数据的真实来源。特工的记忆保存着观察结果——在某个时间点是真实的事情，并且可能仍然是真实的，也可能不是真实的。文件自然地传达了这种认识论状态。它们具有显示观察记录时间的修改时间。知道观察结果错误的人可以阅读、编辑和删除它们。数据库意味着永久性和权威性； Markdown 文件建议有人写下并可能需要更新的注释。存储介质传达了数据的本质——这些是工作笔记，而不是福音。

## 四类分类法

并不是所有的事情都值得记住。记忆系统将所有记忆限制为四种类型：

这四种类型是：用户、反馈、项目和参考。

分类法是围绕一个标准设计的：这些知识是否可以从当前项目状态中得出？ 代码模式、架构、文件结构、git 历史记录——所有这些都可以通过阅读代码库来重新导出。他们被排除在外。这四种类型捕获了无法重新派生的内容。

用户记忆记录了有关该人的信息：他们的角色、目标、职责、专业水平。刚接触 React 的高级 Go 工程师与初次接触的程序员会得到不同的解释。

反馈记忆捕获有关如何处理工作的指导——包括纠正和确认。系统明确指示模型记录两者：“如果您只保存更正，您将偏离用户已经验证的方法。”每个反馈记忆都有一个特定的结构：规则本身，然后是包含原因的“**为什么：**”行（通常是过去的事件），然后是包含触发条件的“**如何应用：**”行。

项目记忆记录了正在进行的工作环境——谁在做什么、为什么、何时做。prompt强调将相对日期转换为绝对日期：“星期四”变为“2026-03-05”，因此几周后记忆仍然可以解释。

参考存储器是书签——指向外部系统中信息所在位置的指针。一个 Linear 项目 URL、一个 Grafana 仪表板、一个 Slack 通道。这些告诉模型去哪里寻找，而不是寻找什么。

## 作为过滤器的分类法

这四种类型不仅仅是类别——它们还是一个过滤器。通过准确定义什么算作内存，系统隐式定义了什么不算内存。如果没有分类法，急切的模型将保存所有内容：代码模式、架构图、错误消息。一切都可以从代码库中派生出来。保存它会创建一个并行的、可能过时的信息副本，该副本更好地源自其来源。

这种分类法还可以防止一个更微妙的失败：记忆作为拐杖。如果模型将架构决策保存为内存，它就会停止读取代码库来理解架构。通过排除可导出的信息，系统迫使模型保持在代码的当前状态。

排除列表是明确的：代码模式、git 历史记录、调试解决方案、CLAUDE.md 中的任何内容、临时任务详细信息。即使用户明确要求保存，这些排除也适用。如果用户说“记住这个公关列表”，模型就会被指示反驳——“其中有什么令人惊讶或不明显的地方？”这个令人惊讶的部分值得保留。原始列表不是。该指令通过 evals 进行验证，添加排除覆盖指令时，从 0/2 变为 3/3。

## 作为合同的 Frontmatter

## 每个内存文件都使用 YAML frontmatter 和三个必填字段：

` 分机
` ---
名称：{{内存名称}}
描述: {{ 一行描述 -- 用于决定相关性 }}
类型：{{ 用户，反馈，项目，参考}}
---`
`

‘描述’是最承载的字段。相关性选择器（十四行诗侧查询，下面讨论）用它来决定是否显示该内存。像“测试东西”这样的模糊描述要么匹配范围太广，要么根本不匹配。像“集成测试必须击中真实数据库，而不是模拟——被模拟分歧 Q4 烧毁”这样的具体描述与重要的对话完全匹配。描述是内存的搜索索引——不是由搜索引擎消耗，而是由可以理解细微差别、上下文和意图的语言模型消耗。

前部内容也是扫描系统在调用期间读取的文件的唯一部分。 `scanMemoryFiles()` 仅读取每个文件的前 30 行以提取标头。在明确选择并加载文件之前，主体是私有的。

## 召回之路

书写记忆是必要的，但还不够。更难的问题是检索：给定用户的查询，可能有数百个内存文件中的哪一个应该加载到模型的上下文中？加载所有这些将耗尽代币预算。不加载任何内容都会违背目的。加载错误的信息会在不相关的信息上浪费代币，同时错过会改变模型行为的知识。

召回系统分两层运作。 “MEMORY.md”索引始终在会话开始时加载到上下文中，以提供方向。各个内存文件通过 LLM 支持的相关性查询按需显示，每轮最多选择五个内存。

## 完整的召回流程

步骤 2 中的异步预取是关键的性能决策。当主模型达到调用上下文有用的程度时，辅助查询通常已经完成。用户不会体验到额外的延迟。

## 陈旧性

过时系统解决了实际使用中出现的故障模式。用户报告说，旧的记忆——包含文件：对后来发生更改的代码的引用——被模型断言为事实。这一引文使陈旧的主张听起来更具权威性，而不是更少。

解决办法不是过期。旧记忆不会被删除——它们可能包含多年有效的机构知识。相反，系统会附加年龄警告：

陈旧度函数计算内存的使用期限（以天为单位）。今天或昨天的记忆不会收到任何警告（该函数返回一个空字符串）。所有较旧的内容都会与内存内容一起注入警告：一条消息，说明年龄（以天为单位），并警告代码行为声明或文件：行引用可能已过时，建议针对当前代码进行验证。

今天或昨天的记忆没有任何警告。所有较旧的内容都会与内存内容一起注入陈旧警告。人类可读的格式——“今天”、“昨天”、“47 天前”——之所以存在，是因为模型在日期运算方面很差。原始 ISO 时间戳不会像“47 天前”那样触发过时推理。这是对模型行为的实证观察，通过评估进行验证：“根据记忆进行推荐之前”的动作prompt框架得分为 3/3，而更抽象的“相信你记得的内容”得分为 0/3，且正文相同。

有一种值得一提的哲学张力。陈旧系统将记忆视为假设，而不是事实。但该模型的自然倾向是自信地呈现信息。过时警告正在与模型自己的声音作斗争——利用其遵循指令的能力来克服其产生信心的倾向。

## MEMORY.md 作为始终加载的索引

每个对话都以上下文中的“MEMORY.md”开始。它不是内存——它是一个索引，是实际内存文件的目录。

## 该指数有两个硬顶：

该索引有两个硬上限：200 行和 25,000 字节。

200 行上限符合正常增长。 25KB 字节上限捕获了一种观察到的故障模式：用户打包的长行保持在 200 行以下，但消耗了大量的代币预算。在第 97 个百分位数处，只有 197 行的 MEMORY.md 的大小为 197KB。当任一上限触发时，可操作的指导会告诉用户要修复什么：“将索引条目保留在约 200 个字符的一行内；将详细信息移至主题文件中。”

这种两层架构——轻量级的永远在线索引加上大量的点播内容——是允许内存扩展的设计。具有 150 个内存的项目有 150 行索引，可能消耗 3,000 个令牌，而不是 150 个完整文件消耗 100,000 个令牌。

从个人记忆到共享知识的转变是自然的。测试策略、部署约定、**构建系统**中的已知陷阱 - 这些都需要在团队中共享。

## 团队记忆

团队内存是位于“<autoMemPath>/team/”的自动内存目录的子目录，位于功能标志后面并需要启用自动内存。架构嵌套是经过深思熟虑的：禁用自动内存会间接禁用团队内存。

## 纵深防御

团队记忆引入了个人记忆所没有的攻击面。团队同步的文件来自其他用户，恶意队友可能会尝试路径遍历。该安全模型使用三层防御。

第 1 层：输入清理。 `sanitizePathKey()` 函数针对空字节、URL 编码遍历 (`%2e%2e%2f`)、Unicode 规范化攻击（规范化为 `../` 的全角字符）、反斜杠和绝对路径进行验证。

第 2 层：字符串级路径验证。 清理后，“path.resolve()”规范剩余的“..”段，并根据团队目录前缀检查解析的路径（包括尾部分隔符以防止“team-evil/”匹配“team/”）。

第 3 层：符号链接解析。 `realpathDeepestExisting()` 解析最深的现有祖先上的符号链接，捕获字符串级验证无法检测到的攻击。如果“team/evil”是指向“/etc/”的符号链接，则字符串验证会看到有效的前缀，但“realpath”会显示真正的目标。

所有验证失败都会产生“PathTraversalError”。没有局部的成功，没有后退。失败关闭。

## 范围指南

prompt会向模型介绍私有内存与共享内存。用户记忆始终是私人的。参考记忆通常是团队的。反馈记忆默认为私有，除非它们代表项目范围的约定。交叉检查指令——“在保存私人反馈记忆之前，检查它是否与团队反馈记忆相矛盾”——可以防止根据首先调用的记忆而出现不可预测的冲突指导。

## KAIROS 模式：仅附加每日日志

标准内存假定离散会话。 KAIROS 模式（Claude Code 的助手模式）打破了这一假设——会话是长期存在的，可能会运行数天。两步写入模式无法扩展到连续操作。

## 解决方案是捕获和整合之间的架构分离：

在 KAIROS 模式下，模型附加到以日期命名的日志文件 (`<autoMemPath>/logs/YYYY/MM/YYYY-MM-DD.md`)。每个条目都是一个带有时间戳的短项目符号。该模型被指示：“不要重写或重新组织日志”——捕获期间的重组会丢失整合所需的时间顺序信号。

prompt中的路径被描述为一种模式，而不是今天的字面日期。这是一个缓存优化：当日期在午夜发生变化时，内存prompt会被缓存并且不会失效。该模型从单独的“date_change”附件中获取当前日期。

## /梦想巩固

合并分四个阶段进行：定向（列出目录、读取索引、浏览现有文件）、收集（搜索日志、检查漂移的内存）、合并（写入或更新文件、合并而不是重复）、修剪（更新 200 行以下的索引、删除过时的指针）。强调合并到现有文件而不是创建新文件很重要 - 没有它，内存目录将随着使用量线性增长。

## 合并锁

锁文件`.consolidate-lock`有双重用途：它的内容是持有者的PID（互斥），它的mtime是`lastConsolidatedAt`（调度状态）。当三个门通过时，自动梦想就会触发，从最便宜的优先进行评估：自上次整合以来的小时数超过 24 小时，此后修改的会话超过 5 个，并且没有其他进程持有锁。崩溃恢复通过“process.kill(pid, 0)”检测失效的 PID，并设置一小时的过时超时来防止 PID 重用。

## 背景提取

主代理拥有主动写入记忆的完整指令。但代理人是不完美的——而且这种不完美是可以预见的。当用户说“记住始终使用集成测试”，然后立即询问“现在修复登录错误”时，模型的注意力完全转移到错误上。内存节省指令已被处理但可能无法执行。

在每个完整的query loop结束时，分叉代理（共享父代理的Prompt Cache）分析​​最近的消息并写入主代理错过的任何记忆。当主代理已经在当前轮次范围内写入内存时，提取代理会跳过该范围。提取代理的工具预算有限：只读工具加上仅对内存目录路径的写访问。它的prompt指示两回合策略：第 1 回合并行读取，第 2 回合并行写入。

这种互动是合作性的，而不是竞争性的。主代理的prompt始终包含完整的保存说明。当主代理保存时，后台代理延迟。如果没有，后台代理就会捕捉到这个间隙。这种模式（具有后台安全网的主要路径）使内存捕获更加可靠，而不会给主要交互带来负担。单独这样做是不够的。

## 路径解析和安全性

## 自动记忆路径通过优先级链解析：

- `CLAUDE_COWORK_MEMORY_PATH_OVERRIDE` — Cowork 的完整路径覆盖。

- settings.json 中的 `autoMemoryDirectory` — 仅受信任的设置源。项目设置被有意排除。

- 默认计算路径 — `~/.claude/projects/<sanitized-git-root>/memory/`。

排除项目设置是一项安全决定。恶意存储库可以使用“autoMemoryDirectory：“~/.ssh””提交“.claude/settings.json”，并且内存文件的权限划分将授予模型对 SSH 密钥的自动写入访问权限。通过限制对策略、标志、本地和用户设置的覆盖（没有可提交到存储库的设置），可以关闭此攻击向量。

isAutoMemPath() 函数在前缀检查之前规范化路径以防止遍历，并且尾随分隔符约定确保前缀匹配需要目录边界。

## 启用/禁用链

自动内存是否处于活动状态由 isAutoMemoryEnabled() 确定，实现其自己的优先级链：环境变量、裸模式、无持久存储的 CCR、设置、默认启用。禁用后，prompt部分将被删除（因此模型不会接收内存指令），并且后台进程将停止（提取内存、自动梦想、团队同步）。两个门必须对齐——仅删除prompt并不能停止提取代理，提取代理有自己的prompt。

## 应用此：设计代理内存

内存系统的复杂性在于行为层——prompt指令、LLM支持的召回、过时管理、背景提取——而不是存储基础设施。这种复杂性的分布本身就是一种设计原则。

在代理内存方面，文件胜过数据库。 文件是可检查、可编辑和版本可控的。透明度建立信任。当替代方案是用户无法轻松读取的数据库时，文件仅凭信任就能获胜。

限制保存的内容，而不仅仅是保存方式。 可推导性测试——这些知识可以从当前的项目状态重新推导吗？ ——消除大部分潜在记忆，同时保留真正重要的记忆。

使用法学硕士来回忆，而不是关键字或嵌入。 LLM 侧查询可以理解上下文、对话中已有内容的原因、处理否定，并且不需要维护索引。延迟成本是真实的，但有界，并且隐藏在主模型处理的后面。

警告过时，不要过期。 机构知识可能多年有效。附加年龄警告可以让模型将旧记忆视为假设而不是事实。人类可读的年龄格式会触发正确的推理，而原始时间戳则不会。

建立捕获安全网。 主要特工会怀念回忆。审查最近对话的背景提取代理使系统更加可靠，而不会给主要交互带来负担。当主代理保存时，后台代理延迟。

代理现在可以跨会话学习 - 积累有关用户、他们的偏好、项目状态以及他们所做的更正的知识。记忆系统做出了一个哲学承诺：代理与其用户的关系应该随着时间的推移而加深，而不是在每次交互时重置。基于文件的实现使这一承诺变得切实可行——在磁盘上可见、可由人类编辑、与代码一起进行版本控制。特工的记忆不是黑匣子。它是文件夹中的笔记集合，以模型和人类都可以阅读的语言编写。
