# 第 1 章：AI Agent 架构

## 你将看到什么

传统的 CLI 更像一个函数。它接收参数、执行任务，然后退出。`grep` 不会自己决定再去运行 `sed`，`curl` 也不会根据下载内容自动打开文件并打补丁。这个约定很简单：一个命令，一个动作，一个可预期的输出。

Agent CLI 打破了这个约定的每一部分。它接收自然语言 prompt，自主决定调用什么工具、按什么顺序执行、如何评估结果，并不断循环，直到任务完成或被用户叫停。“程序”不再是一串预先写死的指令，而是一个围绕语言模型展开的循环，在运行时动态生成自己的执行步骤。工具调用是副作用，模型推理则承担了控制流的角色。

Claude Code 是 Anthropic 对这个想法的生产级实现：一个包含近 2000 个文件的 TypeScript 单体工程，把终端变成由 Claude 驱动的完整开发环境。它服务于数十万开发者，因此每一个架构决策都会产生真实世界的影响。本章的目标，是帮你建立一套心智模型。整个系统可以抽象为六个核心部分，它们由一条统一的数据流串联起来。理解了从一次按键到最终输出的“**黄金路径**”，后面的每一章都只是对这条路径某一段的放大观察。

接下来是回顾性分解——这六个抽象并不是在白板上预先设计的。他们摆脱了将生产代理运送到大量用户群的压力。了解它们的本来面目，而不是按照它们的计划，为本书的其余部分设定了正确的期望。

## 六个关键抽象

Claude Code 构建于六个核心抽象之上。其他一切 - 400 多个实用程序文件、分叉的终端渲染器、vim 仿真、成本跟踪器 - 都是为了支持这六个而存在的。

以下是每个人的作用及其存在的原因。

1. query loop（`query.ts`，约 1,700 行）。异步生成器是整个系统的心跳。它流式传输模型响应、收集工具调用、执行它们、将结果附加到消息历史记录并循环。每个交互——REPL、SDK、sub-agent、无头“--print”——都流经这个单一函数。它生成 UI 使用的“Message”对象。它的返回类型是一个名为“Terminal”的可区分联合，它准确编码了​​循环停止的原因：正常完成、用户中止、代币预算耗尽、停止Hook干预、最大轮数或不可恢复的错误。生成器模式（而不是回调或事件发射器）提供自然的背压、干净的取消和类型化的终端状态。第 5 章完整介绍了循环的内部结构。

2. 工具系统（`Tool.ts`、`tools.ts`、`services/tools/`）。工具是代理可以在世界上执行的任何操作：读取文件、运行 shell 命令、编辑代码、搜索网络。这种简单的目的隐藏着重要的机制。每个工具都实现了丰富的接口，涵盖身份、模式、执行、权限和渲染。工具不仅仅是函数——它们带有自己的权限逻辑、并发声明、进度报告和 UI 渲染。系统将工具调用划分为并发和串行批处理，并且流式执行器在模型完成其响应之前启动并发安全工具。第 6 章介绍了完整的工具接口和执行管道。

3. 任务（`Task.ts`、`tasks/`）。任务是后台工作单元——主要是sub-agent。它们遵循一个状态机：`待处理 -> 运行 -> 已完成 |失败 |杀了`。 `AgentTool` 生成一个新的 `query()` 生成器，具有自己的消息历史记录、工具集和权限模式。任务为 Claude Code 提供了递归功能：代理可以委托给sub-agent，sub-agent可以进一步委托。

4. 状态（两层）。系统在两个级别上维护状态。可变单例（“STATE”）保存约 80 个会话级基础设施字段：工作目录、模型配置、成本跟踪、遥测计数器、会话 ID。它在启动时设置一次并直接突变——没有反应性。最小的响应式存储（34 行，Zustand 形状）驱动 UI：消息、输入模式、工具批准、进度指示器。这种分离是有意为之的：基础设施状态很少发生变化，不需要触发重新渲染； UI 状态必须不断变化。第 3 章深入介绍了两层架构。

5. 内存（`memdir/`）。代理跨会话的持久上下文。三层：项目级（存储库中的“CLAUDE.md”文件）、用户级（“~/.claude/MEMORY.md”）和团队级（通过符号链接共享）。在会话开始时，系统扫描所有内存文件，解析 frontmatter，并且 LLM 选择哪些内存与当前对话相关。内存是 Claude Code “记住”您的代码库约定、架构决策和调试历史记录的方式。

6. Hook（`hooks/`、`utils/hooks/`）。用户定义的生命周期拦截器可在 4 种执行类型中触发 27 个不同事件：shell 命令、单次 LLM prompt、多轮代理对话和 HTTP Webhook。Hook 可以阻止工具执行、修改输入、注入附加上下文或短路整个query loop。**权限系统**本身部分是通过Hook 实现的——“PreToolUse”Hook 可以在交互式权限 prompt触发之前拒绝工具调用。

## 黄金之路：从击键到输出

通过系统跟踪单个请求。用户键入“向登录功能添加错误处理”并按 Enter 键。

关于此流程需要注意三件事。

首先，query loop 是一个生成器，而不是回调链。REPL 通过 `for await` 从中提取消息，这意味着背压是天然存在的：如果 UI 跟不上，生成器就会暂停。这是相对于事件发射器或可观察流的一次有意选择。

其次，工具执行与模型流重叠。 “StreamingToolExecutor”在启动并发安全工具之前不会等待模型完成。当模型仍在生成其响应的其余部分时，“Read”调用可以完成并返回其结果。这是推测执行 - 如果模型的最终输出使工具调用无效（罕见但有可能），则结果将被丢弃。

第三，整个循环是可重入的。当模型进行工具调用时，结果将附加到消息历史记录中，并且循环使用更新的上下文再次调用模型。没有单独的“工具结果处理”阶段——这都是一个循环。该模型通过不再进行任何工具调用来决定何时完成。

## 权限系统

Claude Code 在您的计算机上运行任意 shell 命令。它编辑您的文件。它可以生成子进程、发出网络请求以及修改 git 历史记录。如果没有 **权限系统**，这将是一场安全灾难。

模式行为 `bypassPermissions` 一切都允许，不做检查，仅限内部或测试。`dontAsk` 全部允许，但仍会记录，不弹用户 prompt。`auto` 由轻量级分类器（LLM）决定允许还是拒绝。`acceptEdits` 文件编辑自动批准，其他变更仍然需要 prompt。`default` 标准交互模式，由用户批准每个操作。`plan` 只读，所有变更都会被阻止。`bubble` 将决策上抛给父 agent，用于 sub-agent 模式。

“自动”模式值得特别关注。它运行一个单独的轻量级 LLM 调用，根据对话记录对工具调用进行分类。分类器看到工具输入的紧凑表示，并决定操作是否与用户的要求一致。这种模式可以让 Claude Code 半自主地工作——批准例行操作，同时标记任何看起来偏离用户意图的内容。

## 多提供商架构

Claude Code 通过四种不同的基础设施路径与 Claude 进行对话，这些路径对系统的其余部分都是透明的。

关键的见解是，Anthropic SDK 为每个云提供商提供了包装类，这些提供商提供与直接 API 客户端相同的接口。 `getAnthropicClient()` 工厂读取环境变量和配置以确定要使用哪个提供程序、构造适当的客户端并返回它。从那时起，“callModel()”和所有其他消费者都将其视为通用的 Anthropic 客户端。

提供者选择在启动时确定并存储在“STATE”中。query loop从不检查哪个提供程序处于活动状态。这意味着从 Direct API 切换到 Bedrock 是配置更改，而不是代码更改 — agent loop、工具系统和权限模型完全与提供商无关。

“feature()”函数来自“bun:bundle”，Bun 的内置捆绑器 API。在构建时，每个功能标志都会解析为布尔文字。当标志为 false 时，捆绑器的死代码消除会完全删除“require()”调用——该模块从未加载，从未包含在捆绑包中，也从未发货。

该模式是一致的：顶级“feature()”防护包装了“require()”调用。使用“require()”而不是“import”，具体是因为当防护为 false 时，动态“require()”可以被捆绑器完全消除，而动态“import()”则不能（它返回捆绑器必须保留的 Promise）。

有一个讽刺值得注意。早期 npm 版本发布的源映射包含“sourcesContent”——完整的原始 TypeScript 源，包括仅供内部使用的代码路径。功能标志成功地剥离了运行时代码，但将源代码留在了映射中。这就是Claude Code源代码成为公开可读的方式。

## 各部分如何连接

这六个抽象形成了一个依赖图：

内存作为 system prompt 的一部分输入 query loop。query loop 驱动工具执行。工具结果再以消息的形式反馈回 query loop。任务则是拥有独立消息历史的递归 query loop。Hook 会在定义好的节点拦截 query loop。状态被所有模块读取和写入，而响应式存储则负责桥接到 UI。

query loop 和工具系统之间的循环依赖，是这个系统最关键的特征。该模型生成工具调用。工具执行并产生结果。结果将附加到消息历史记录中。模型看到结果并决定下一步做什么。此循环一直持续到模型停止生成工具调用或外部约束（令牌预算、最大轮数、用户中止）终止它为止。

## 应用此

如果您正在构建一个代理系统（LLM 决定在运行时采取哪些操作的任何系统），那么以下是 Claude Code 架构中可传输的模式。

生成器循环模式。 使用异步生成器作为agent loop，而不是回调或事件发射器。生成器为您提供自然的背压（消费者按照自己的节奏拉动）、干净的取消（生成器上的“.return()”）以及终端状态的类型化返回值。它解决的问题是：在基于回调的agent loop中，很难知道循环何时“完成”以及为什么。生成器使终止成为类型系统的首要部分。

自描述工具界面。 每个工具都应该声明自己的并发安全性、权限要求和渲染行为。不要将此逻辑放入“了解”每个工具的中央协调器中。它解决的问题是：中央编排器成为一个神对象，每次添加工具时都必须更新。自描述工具线性扩展——添加工具 N+1 需要对现有代码进行零更改。

将基础设施状态与反应状态分开。 并非所有状态都需要触发 UI 更新。会话配置、成本跟踪和遥测属于普通的可变对象。消息历史记录、进度指示器和批准队列属于响应式存储。它解决的问题是：使所有内容都成为响应式，增加了订阅开销和复杂性，使得在启动时更改一次并被读取一千次的状态。两层匹配两种访问模式。

权限模式，而不是权限检查。 定义一小组命名模式（计划、默认、自动、旁路）并通过该模式解决每个权限决策。不要通过工具实现分散“if (isAllowed)”检查。它解决的问题是：权限执行不一致。当每个工具都经过相同的基于模式的解析链时，您可以通过了解哪种模式处于活动状态来推断系统的安全状况。

通过任务构建递归 Agent 架构。sub-agent 应该是同一个 agent loop 的新实例，拥有自己的消息历史，而不是特殊分支代码。权限提升通过 `bubble` 模式向上流动。它解决的问题是：如果 sub-agent 逻辑和主 agent loop 分叉，就很容易在行为和错误处理上出现细微差异；而如果 sub-agent 复用同一套 loop，它就能天然继承同样的保证。
