# 第16章：远程控制和云执行

## 代理超越本地主机

到目前为止，每一章都假设 Claude Code 在代码所在的同一台机器上运行。终端是本地的。文件系统是本地的。模型响应流回拥有键盘和工作目录的进程。

当您想要从浏览器控制 Claude Code、在云容器内运行它或将其作为 LAN 上的服务公开时，这种假设就被打破了。代理需要一种方法来接收来自 Web 浏览器、移动应用程序或自动化管道的指令 - 将权限 prompt转发给不在终端的人员，并通过可能代表代理注入凭据或终止 TLS 的基础设施传输其 API 流量。

Claude Code 通过四个系统解决了这个问题，每个系统处理不同的拓扑：

这些系统具有共同的设计理念：读取和写入是不对称的，重新连接是自动的，并且故障会优雅地降级。

## Bridge v1：轮询、调度、生成

v1 网桥是基于环境的远程控制系统。当开发者运行“claude Remote-control”时，CLI 会向环境 API 进行注册，轮询工作，并为每个会话生成一个子进程。

在注册之前，会运行一系列预检检查：运行时功能门控、OAuth 令牌验证、组织策略检查、失效令牌检测（同一过期令牌连续三次失败后的跨进程退避）以及主动令牌刷新，以消除大约 9% 的注册，否则这些注册在第一次尝试时就会失败。

一旦注册，网桥就会进入长轮询循环。工作项以会话（带有包含会话令牌、API 基本 URL、MCP 配置和环境变量的“秘密”字段）或运行状况检查的形式到达。桥接器会限制每 100 个空轮询的“无工作”日志消息。

每个会话都会生成一个子 Claude Code 进程，通过 stdin/stdout 上的 NDJSON 进行通信。权限请求通过网桥传输流至用户批准或拒绝的 Web 界面。往返必须在大约 10-14 秒内完成。

## Bridge v2：直接会话和 SSE

v2 桥消除了整个环境 API 层 — 无需注册、无需轮询、无需确认、无需心跳、无需取消注册。动机：v1 要求服务器在分派工作之前了解机器的功能。 V2 将生命周期折叠为三个步骤：

- 使用 OAuth 凭据创建会话：`POST /v1/code/sessions`。

- 连接桥：`POST /v1/code/sessions/{id}/bridge`。返回“worker_jwt”、“api_base_url”和“worker_epoch”。每个“/bridge”调用都会改变纪元——这就是注册。

- 开放传输：SSE 用于读取，`CCRClient` 用于写入。

传输抽象（`ReplBridgeTransport`）将 v1 和 v2 统一在一个公共接口后面，因此消息处理不需要知道它正在与哪一代通信。

当 SSE 连接由于 401 断开时，传输会使用新的“/bridge”调用中的新凭据进行重建，同时保留序列号游标 — 不会丢失任何消息。写入路径使用每个实例的“getAuthToken”闭包而不是进程范围的环境变量，从而防止并发会话中的 JWT 泄漏。

## 冲水门

一个微妙的排序问题：桥接器需要发送对话历史记录，同时接受来自 Web 界面的实时写入。如果在历史刷新期间实时写入到达，则消息可能会乱序传递。 `FlushGate` 在刷新 POST 期间对实时写入进行排队，并在完成时按顺序排出它们。

## 令牌刷新和纪元管理

v2 桥会在到期前主动刷新工作 JWT。新纪元告诉服务器这是具有新凭据的同一个工作人员。 Epoch 不匹配（409 响应）会被积极处理：连接关闭并且异常会释放调用者，从而防止裂脑情况。

## 消息路由和回声重复数据删除

两代网桥共享“handleIngressMessage()”作为中央路由器：

- 解析 JSON，规范控制消息键。

- 将“control_response”路由到权限处理程序，将“control_request”路由到请求处理程序。

- 根据“recentPostedUUIDs”（回显重复数据删除）和“recentInboundUUIDs”（重新传递重复数据删除）检查 UUID。

- 转发经过验证的用户消息。

## BoundedUUIDSet：O(1) 查找，O(容量) 内存

桥接器存在回声问题——消息可能会在读取流上回声，或者在传输切换期间传送两次。 `BoundedUUIDSet` 是一个由循环缓冲区支持的 FIFO 有界集合：

` 分机
` 类 BoundedUUIDSet {
私有缓冲区：字符串[]
私有集：设置<字符串>
私人头 = 0

添加（uuid：字符串）：无效{
if (这个.set.size >= 这个.capacity) {
这个.set. 删除（这个.buffer[这个.head]）
}
这个.buffer[这个.head] = uuid
这个.set. 添加（uuid）
这个.head = (这个.head + 1) % 这个.capacity
}

has ( uuid: string ): boolean { 返回这个.set。 有（uuid）}
} `
`

两个实例并行运行，每个实例的容量为 2000。通过 Set 查找 O(1)，通过循环缓冲区逐出 O(capacity) 内存，没有计时器或 TTL。未知的控制请求子类型会得到错误响应，而不是沉默——防止服务器等待永远不会到来的响应。

## 非对称设计：持久读取、HTTP POST 写入

CCR 协议使用非对称传输：通过持久连接（WebSocket 或 SSE）读取流，通过 HTTP POST 写入。这反映了沟通模式的根本不对称。

读取是高频率、低延迟、由服务器启动的——在令牌流期间每秒数百条小消息。持久连接是唯一明智的选择。写入频率较低，由客户端发起，并且需要确认——每分钟一条消息，而不是每秒一条。 HTTP POST 通过 UUID 提供可靠的交付、幂等性以及与负载均衡器的自然集成。

尝试将它们统一在单个 WebSocket 上会产生耦合：如果 WebSocket 在写入期间丢失，您需要重试逻辑，并且必须区分“未发送”和“已发送但确认丢失”。独立的通道可以独立优化每个通道。

## 远程会话管理

`SessionsWebSocket` 管理 CCR WebSocket 连接的客户端。其重新连接策略区分故障类型：

失败策略 4003（未经授权） 立即停止，不重试 4001（未找到会话） 最多 3 次重试，线性退避（压缩期间瞬态） 其他瞬态 指数退避，最多 5 次尝试
“isSessionsMessage()”类型保护接受任何带有字符串“type”字段的对象——故意允许。硬编码的白名单会在客户端更新之前默默地删除新的消息类型。

## 直接连接：本地服务器

Direct Connect 是最简单的拓扑：Claude Code 作为服务器运行，客户端通过 WebSocket 连接。没有云中介，没有 OAuth 令牌。

会话有五种状态：“开始”、“运行”、“分离”、“停止”、“停止”。元数据保留在“~/.claude/server-sessions.json”中，以便在服务器重新启动时恢复。 `cc://` URL 方案为本地连接提供了干净的寻址。

## 上游代理：容器中的凭证注入

上游代理在 CCR 容器内运行并解决特定问题：将组织凭据注入到来自代理可能执行不受信任命令的容器的出站 HTTPS 流量中。

## 设置顺序经过仔细排序：

- 从“/run/ccr/session_token”读取会话令牌。

- 通过 Bun FFI 设置 `prctl(PR_SET_DUMPABLE, 0)` — 阻止进程堆的相同 UID ptrace。如果没有这个，prompt注入的“gdb -p $PPID”可能会从内存中刮掉令牌。

- 下载上游代理 CA 证书并与系统 CA 捆绑包连接。

- 在临时端口上启动本地 CONNECT-to-WebSocket 中继。

- 取消链接令牌文件 — 令牌现在仅存在于堆上。

- 导出所有子进程的环境变量。

每一步都无法打开：错误禁用代理而不是终止会话。正确的权衡——失败的代理意味着某些集成将无法工作，但核心功能仍然可用。

## Protobuf 手工编码

通过隧道的字节被包装在“UpstreamProxyChunk”protobuf 消息中。该模式很简单 - `message UpstreamProxyChunk { bytes data = 1; }` - Claude Code 手动编码为十行，而不是引入 protobuf 运行时：

` 分机

` 导出函数encodeChunk（数据：Uint8Array）：Uint8Array {
const varint: 数字 [] = []
让 n = 数据。 长度
while (n > 0x7f ) { varint. 推（（n＆0x7f）| 0x80）； n >>>= 7 }
变体。 推 (n)
const out = new Uint8Array ( 1 + varint.length + data.length )
out[ 0 ] = 0x0a // 字段 1，线路类型 2
出来。 设置（varint，1）
出来。 设置（数据，1 + varint.length）
返回
} `
`

十行代码取代了完整的 protobuf 运行时。单字段消息并不能证明依赖性是合理的——位操作的维护负担远低于供应链风险。

## 应用此：设计远程代理执行

独立的读写通道。 当读取是高频流而写入是低频 RPC 时，统一它们会产生不必要的耦合。让每个通道独立故障和恢复。

限制您的重复数据删除内存。 BoundedUUIDSet 模式提供固定内存重复数据删除。任何至少一次传送系统都需要有界的重复数据删除缓冲区，而不是无界的 Set。

使重连策略与故障信号成比例。 永久失败不应重试。暂时性失败应通过退避重试。不明确的失败应该以较低的上限重试。

在敌对环境中仅保留堆秘密。 从文件中读取令牌、禁用 ptrace 并取消链接文件可以消除文件系统和内存检查攻击向量。

辅助系统故障打开。 上游代理无法打开，因为它提供增强功能（凭据注入），而不是核心功能（模型推理）。

远程执行系统编码了更深层次的原则：代理的核心循环（第 5 章）应该不知道指令来自何处以及结果去往何处。桥接器、Direct Connect 和上游代理是传输层。无论用户是坐在终端还是 WebSocket 的另一侧，消息处理、工具执行和权限流都是相同的。
