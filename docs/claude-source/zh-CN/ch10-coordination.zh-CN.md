# 第 10 章：任务、协调和集群

## 单线程的限制

第 8 章展示了如何创建sub-agent——从代理定义构建隔离执行上下文的十五步生命周期。第 9 章展示了如何通过快速缓存利用来经济地实现并行生成。但创建代理和管理代理是不同的问题。本章讨论第二个。

单个agent loop（一个模型、一个对话、一次一个工具）可以完成大量工作。它可以读取文件、编辑代码、运行测试、搜索网络以及推理复杂问题。但它已经达到了天花板。

天花板不是智力。这是并行性和范围。从事大型重构的开发者需要更新 40 个文件，在每批之后运行测试，并验证没有出现任何问题。代码库迁移同时涉及前端、后端和数据库层。在后台运行测试套件时，彻底的代码审查会读取数十个文件。这些并不是更难的问题——而是更广泛的问题。他们需要能够同时做多项事情、将工作委托给专家并协调结果。

Claude Code 对这个问题的答案不是一种机制，而是分层的编排模式，每种模式都适合不同的工作形式。即发即忘命令的后台任务。经理-工人层次结构的协调员模式。组建团队进行点对点协作。以及将它们联系在一起的统一通信协议。

编排层跨越“tools/AgentTool/”、“tasks/”、“coordinator/”、“tools/SendMessageTool/”和“utils/swarm/”中的大约 40 个文件。尽管有这样的广度，该设计还是以所有模式共享的单个状态机为基础。理解状态机——“Task.ts”中的“Task”抽象——是理解其他一切的先决条件。

本章跟踪完整的堆栈，从基本的任务状态机到最复杂的多代理拓扑。

## 任务状态机

Claude Code 中的每个后台操作（shell 命令、sub-agent、远程会话、工作流程脚本）都作为任务进行跟踪。任务抽象位于“Task.ts”中，并提供编排层其余部分构建的统一状态模型。

## 七种类型

## 系统定义了七种任务类型，每种类型代表不同的执行模型：

七个任务类型是：“local_bash”（后台 shell 命令）、“local_agent”（后台sub-agent）、“remote_agent”（远程会话）、“in_process_teammate”（群队友）、“local_workflow”（工作流程脚本执行）、“monitor_mcp”（MCP 服务器监视器）和“dream”（推测性后台思考）。

`local_bash` 和 `local_agent` 是主力——分别是后台 shell 命令和后台sub-agent。 `in_process_teammate` 是群体原语。 `remote_agent` 连接到远程 Claude Code Runtime 环境。 `local_workflow` 运行多步骤脚本。 `monitor_mcp` 监视 MCP 服务器的运行状况。 “dream”是最不寻常的——一个后台任务，让代理在等待用户输入时进行推测性思考。

## 每种类型都有一个单字符 ID 前缀，用于即时视觉识别：

类型前缀 示例 ID `local_bash` `b` `b4k2m8x1` `local_agent` `a` `a7j3n9p2` `remote_agent` `r` `r1h5q6w4` `in_process_teammate` `t` `t3f8s2v5` `local_workflow` `w` `w6c9d4y7` `monitor_mcp` `m` `m2g7k1z8` `梦想` `d` `d5b4n3r6`
任务 ID 使用单字符前缀（a 代表代理、b 代表 bash、t 代表队友等），后跟从不区分大小写的安全字母表中抽取的 8 个随机字母数字字符（数字加小写字母）。这会产生大约 2.8 万亿个组合，足以抵御针对磁盘上任务输出文件的强力符号链接攻击。

当您在日志行中看到“a7j3n9p2”时，您立即知道它是后台代理。当您看到“b4k2m8x1”时，这是一个 shell 命令。该前缀对于人类读者来说是一种微观优化，但在可能有数十个并发任务的系统中，它很重要。

## 五种状态

## 生命周期是一个没有循环的简单有向图：

“pending”是注册和首次执行之间的短暂状态。 “运行”意味着任务正在积极地工作。三种终端状态是“完成”（成功）、“失败”（错误）和“终止”（由用户、协调器或中止信号明确停止）。辅助函数可防止与死任务交互：

` 分机
` 导出函数 isTerminalTaskStatus ( 状态: TaskStatus ): boolean {
返回状态 === '已完成' || 状态 === '失败' || 状态 === '被杀'
} `
`

这个函数随处可见——消息注入防护、逐出逻辑、孤立清理以及决定是对消息进行排队还是恢复失效代理的 SendMessage 路由。

## 基础状态

每个任务状态都扩展“TaskStateBase”，它携带所有七种类型共享的字段：

` 分机
` 导出类型 TaskStateBase = {
id: string // 带前缀的随机 ID
type: TaskType // 鉴别器
status: TaskStatus // 当前生命周期位置
description: string // 人类可读的摘要
toolUseId?: string // 生成此任务的 tool_use 块
startTime: number // 创建时间戳
endTime?: number // 终端状态时间戳
TotalPausedMs?: number // 累计暂停时间
outputFile: string // 流式输出的磁盘路径

outputOffset: number // 读取游标以进行增量输出
notification: boolean // 是否向父级报告完成情况
} `
`

有两个领域值得关注。 `outputFile` 是异步执行和父级会话之间的桥梁 - 每个任务都将其输出写入磁盘上的文件，父级可以通过 `outputOffset` 增量读取它。 “notified”可防止重复的完成消息；一旦父级被告知任务完成，标志就会翻转为“true”，并且不再发送通知。如果没有这个保护，在通知队列的两次连续轮询之间完成的任务将生成重复的通知，从而使模型误认为两个任务完成，而实际上只有一个任务完成。

## 代理任务状态

`LocalAgentTaskState` 是最复杂的变体，包含管理后台sub-agent的完整生命周期所需的一切：

` 分机
` 导出类型 LocalAgentTaskState = TaskStateBase & {
类型：'本地代理'
代理 ID: 字符串
prompt：字符串
选定的代理？：代理定义
代理类型：字符串
型号?: 字符串
abortController?: 中止控制器
endingMessages: string [] // 通过 SendMessage 排队
isBackgrounded: boolean // 这原本是前台代理吗？
keep: boolean // UI 正在保留此任务
diskLoaded: boolean // 侧链转录本已加载
evictAfter?: number // GC 截止时间
进度？：代理进度
最后报告的工具计数：数量
最后报告的令牌计数：数字
//...附加生命周期字段
} `
`

三个字段揭示了重要的设计决策。 “pendingMessages”是收件箱——当“SendMessage”针对正在运行的代理时，消息会在这里排队，而不是立即注入。消息在工具轮边界处被耗尽，这保留了代理的轮次结构。 `isBackgrounded` 区分了天生异步的代理和那些以前台同步代理开始、后来通过用户按键进入后台的代理。 `evictAfter` 是一种垃圾收集机制：非保留的已完成任务在其状态从内存中清除之前会获得一段宽限期。

所有任务状态都以“Record<string, TaskState>”的形式存储在“AppState.tasks”中，并以前缀 ID 为键。这是一个平面地图，而不是一棵树——系统不会在状态存储中对父子关系进行建模。父子关系隐含在对话流中：父级持有生成子级的“toolUseId”。

## 任务注册表

## 每个任务类型都由具有最小接口的“Task”对象支持：

` 分机
` 导出类型任务 = {
名称：字符串
type: 任务类型
kill (taskId: string, setAppState: SetAppState): Promise <void>
} `
`

## 注册表收集所有任务实现：

` 分机
` 导出函数 getAllTasks(): 任务[] {
返回[
本地Shell任务，
本地代理任务，
远程代理任务，
梦想任务，
... (LocalWorkflowTask? [LocalWorkflowTask]: []),
... (MonitorMcpTask? [MonitorMcpTask]: []),
]
} `
`

请注意条件包含 - `LocalWorkflowTask` 和 `MonitorMcpTask` 是功能门控的，并且在运行时可能不存在。 “Task”界面被刻意最小化。早期的迭代包括“spawn()”和“render()”方法，但当很清楚生成和渲染从未被多态调用时，这些方法被删除。每个任务类型都有自己的生成逻辑、自己的状态管理和自己的渲染。真正需要按类型分派的唯一操作是“kill()”，因此这就是接口所需的全部操作。

这是通过减法进行界面演变的一个例子。最初的设计设想所有任务类型都将共享一个通用的生命周期接口。在实践中，类型的差异足够大，以至于共享接口变成了虚构——shell 命令的“spawn()”和进程内队友的“spawn()”几乎没有任何共同点。该团队没有维护一个有漏洞的抽象，而是删除了除了真正受益于多态性的一种方法之外的所有内容。

## 沟通模式

仅当父任务可以观察其进度并接收其结果时，在后台运行的任务才有用。 Claude Code 支持三个通信通道，每个通道都针对不同的访问模式进行了优化。

## 前景：生成器链

当代理同步运行时，父级直接迭代其“runAgent()”异步生成器，生成每条消息备份调用堆栈。这里有趣的机制是后台逃生舱口——同步循环在“来自代理的下一条消息”和“后台信号”之间竞争：

` 分机
` const agentIterator = runAgent ({... params })[Symbol.asyncIterator]()

而（真）{
const nextMessagePromise = agentIterator. 下一个 ()
const raceResult = 背景Promise
？ 等待诺言。 race([nextMessagePromise.then(...),backgroundPromise])
: { 类型: '消息', 结果: 等待 nextMessagePromise }

if (raceResult.type === '背景' ) {
// 用户触发的后台处理——转换为异步
等待代理迭代器。 返回（未定义）
void runAgent ({... params, isAsync: true })
返回{数据：{状态：'async_launched'}}
}

代理消息。 推送（消息）
} `
`

如果用户在执行中决定同步代理应成为后台任务，则前台迭代器将被干净地返回（触发其“finally”块以进行资源清理），并且代理将重新生成为具有相同 ID 的异步任务。转换是无缝的 - 不会丢失任何工作，代理会使用与父级 ESC 键取消链接的异步中止控制器从中断处继续。

这是一个真正难以正确完成的状态转换。前台代理共享父级的中止控制器（ESC 杀死两者）。后台代理需要自己的控制器（ESC 不应该杀死它）。代理的消息需要从前台生成器流传输到后台通知系统。任务状态需要翻转“isBackgrounded”，以便 UI 知道将其显示在背景面板中。所有这一切都必须以原子方式发生——在转换中没有消息丢失，没有僵尸迭代器继续运行。下一条消息和后台信号之间的“Promise.race”是使这成为可能的机制。

## 背景：三个通道

后台代理通过磁盘、通知和排队消息进行通信。

磁盘输出文件。 每个任务都会写入“outputFile”路径 - JSONL 格式的代理记录的符号链接。父级（或任何观察者）可以使用“outputOffset”增量读取该文件，该文件跟踪文件已被消耗的程度。 `TaskOutputTool` 将其公开给模型：

` 分机
` 输入模式 = z. 严格对象（{
任务 ID：z。 字符串 (),
块：z。 布尔值 ()。 默认（ true ），
超时：z。 数（）。 默认（30000），
}) `
`

当“block: true”时，工具会进行轮询，直到任务达到最终状态或超时到期。这是协调器生成工作人员并等待其结果的主要机制。

任务通知。 当后台代理完成时，系统会生成一个 XML 通知并将其排队以传递到父级会话中：

` 分机
` <任务通知>
<任务id>a7j3n9p2</任务id>
<工具使用id>toolu_abc123</工具使用id>
<输出文件>/path/to/output</输出文件>
<状态>已完成</状态>
<摘要>代理“调查身份验证错误”已完成</摘要>
< result >在 src/auth/validate.ts:42 中发现空指针...</ result >
<用法>
<total_tokens>15000</total_tokens>
<工具使用数>8</工具使用数>
<持续时间_毫秒>12000</持续时间_毫秒>
</用法>
</任务通知> `
`

通知作为用户角色消息注入到父级对话中，这意味着模型在其正常消息流中看到它。它不需要特殊的工具来检查完成情况——它们作为上下文到达。任务状态上的“notified”标志可防止重复交付。

命令队列。 “LocalAgentTaskState”上的“pendingMessages”数组是第三个通道。当“SendMessage”针对正在运行的代理时，消息将排队：

` 分机
` if ( isLocalAgentTask (task) && task.status === '正在运行' ) {
queuePendingMessage（agentId，input.message，setAppState）
return { data: { success: true, message: '消息已排队...' } }
} `
`

这些消息通过“drainPendingMessages()”在工具轮边界处排出，并作为用户消息注入到代理的对话中。这是一个至关重要的设计选择——消息在工具轮次之间到达，而不是在执行中间到达。代理完成当前的想法，然后接收新信息。没有竞争条件，没有损坏的状态。

## 进度追踪

## “ProgressTracker”提供对座席活动的实时可见性：

` 分机
` 导出类型 ProgressTracker = {
工具使用计数：数量
latestInputTokens: number // 累计（最新值，不是总和）
cumulativeOutputTokens: number // 轮次求和
previousActivities: ToolActivity [] // 最近使用的 5 个工具
} `
`

输入和输出令牌跟踪之间的区别是经过深思熟虑的，反映了 API 计费模型的微妙之处。每个 API 调用的输入令牌都是累积的，因为每次都会重新发送完整的对话 — 第 15 轮包括之前的所有 14 轮，因此 API 报告的输入令牌计数已经反映了总数。保留最新值是正确的聚合。输出令牌是每轮的——模型每次都会生成新的令牌——因此求和是正确的聚合。如果出现此错误，要么会严重超算（对累积输入令牌求和），要么会严重低估（仅保留最新的输出令牌）。

`recentActivities` 数组（最多 5 个条目）提供了代理正在执行的操作的人类可读流：“读取 src/auth/validate.ts”、“Bash：npm test”、“编辑 src/auth/validate.ts”。这出现在 VS Code sub-agent面板和终端的后台任务指示器中，使用户可以了解代理工作，而无需他们阅读完整的记录。

对于后台代理，进度通过“updateAsyncAgentProgress()”写入“AppState”，并通过“emitTaskProgress()”作为 SDK 事件发出。 VS Code sub-agent面板使用这些事件来呈现实时进度条、工具计数和活动流。进度跟踪不仅仅是装饰性的，它是告诉用户后台代理是否正在取得进展或陷入循环的主要反馈机制。

## 协调器模式

协调器模式将 Claude Code 从具有后台助手的单个代理转变为真正的管理器-工作器架构。它是系统中最固执己见的编排模式，它的设计揭示了对法学硕士应该和不应该如何委派工作的深入思考。

## 协调员模式解决的问题

标准agent loop具有单个对话和单个上下文窗口。当它生成后台代理时，后台代理独立运行并通过任务通知报告结果。这对于简单的委派（“在我继续编辑的同时运行测试”）效果很好，但对于复杂的多步骤工作流程来说却很困难。

考虑代码库迁移。代理需要：(1) 了解 200 个文件的当前模式，(2) 设计迁移策略，(3) 对每个文件应用更改，以及 (4) 验证没有任何损坏。步骤 1 和 3 受益于并行性。步骤 2 需要综合步骤 1 的结果。步骤 4 取决于步骤 3。按顺序执行此操作的单个代理将花费其大部分令牌预算重新读取文件。多个后台代理在没有协调的情况下执行此操作会产生不一致的更改。

协调器模式通过将“思考”代理与“执行”代理分开来解决这个问题。协调员处理步骤 1 和 2（派遣研究人员，然后进行综合）。工作人员处理步骤 3 和 4（应用更改、运行测试）。协调员看到全貌；工人们看到他们的具体任务。

## 激活

## 单个环境变量可以切换开关：

` 分机
` 导出函数 isCoordinatorMode (): boolean {
如果（功能（'COORDINATOR_MODE'））{
返回 isEnvTruthy (process.env. CLAUDE_CODE_COORDINATOR_MODE )
}
返回错误
} `
`

在会话恢复时，“matchSessionMode()”检查恢复的会话的存储模式是否与当前环境匹配。如果它们不同，则翻转环境变量以匹配。这可以防止出现令人困惑的情况，即协调器会话作为常规代理恢复（失去其工作人员的意识）或常规会话作为协调器恢复（失去对其工具的访问权限）。会话的模式是事实的来源；环境变量是运行时信号。

## 工具限制

协调员的力量不是来自于拥有更多的工具，而是来自于拥有的工具更少。在协调器模式下，协调器代理恰好获得三个工具：

- 代理 — 生成工人

- SendMessage — 与现有工作人员沟通

- TaskStop — 终止正在运行的工作进程

就是这样。没有文件读取。没有代码编辑。没有 shell 命令。协调器不能直接接触代码库。这个限制不是限制——它是核心设计原则。协调员的工作是思考、计划、分解和综合。工人做工作。

## 相反，工人获得完整的工具集减去内部协调工具：

` 分机
` const INTERNAL_WORKER_TOOLS = 新集 ([

TEAM_CREATE_TOOL_NAME，
TEAM_DELETE_TOOL_NAME，
SEND_MESSAGE_TOOL_NAME，
SYNTHETIC_OUTPUT_TOOL_NAME ，
]) `
`

工作人员无法生成自己的子团队或向同事发送消息。它们通过正常的任务完成机制报告结果，协调器对它们进行综合。

## 370 行system prompt符

协调器system prompt符是代码库中关于如何使用 LLM 进行编排的最具指导性的文档。它运行大约 370 行代码，并对来之不易的委托模式课程进行编码。主要教导：

“永远不要委托理解。” 这是中心论点。协调员必须将研究结果综合为包含文件路径、行号和确切更改的具体prompt。该prompt明确指出了反模式，例如“根据您的发现，修复错误”——该prompt将理解委托给工作人员，迫使其重新派生协调员已有的上下文。正确的模式是：“在第 42 行的 `src/auth/validate.ts` 中，从 OAuth 流程调用时，`userId` 参数可以为 null。添加一个返回 401 响应的 null 检查。”

“并行性是你的超能力。” prompt建立了清晰的并发模型。只读任务可以自由并行运行——研究、探索、文件读取。写入密集型任务按文件集序列化。协调员应该推断出哪些任务可以重叠，哪些任务必须排序。一个好的协调器同时产生五个研究工作人员，等待所有研究工作人员，综合，然后产生三个接触不相交文件集的实施工作人员。一个糟糕的协调器会产生一个工作人员，等待，产生下一个工作人员，再次等待——序列化本来可以并行的工作。

任务工作流程阶段。 该prompt定义了四个阶段：

- 研究——工作人员并行探索代码库、读取文件、运行测试、收集信息

- 综合——协调员（不是工作人员）阅读所有研究结果并建立统一的理解

- 实施——工作人员收到来自综合的精确指令

- 验证——工作人员运行测试并验证更改

协调员不应跳过阶段。最常见的失败模式是从研究直接跳到实施而不进行综合。当这种情况发生时，协调员会将理解委托给实施工作人员——每个工作人员都必须从头开始重新派生上下文，从而导致不一致的更改和浪费的令牌。

继续与生成的决定。 当工作人员完成并且协调员有后续工作时，它应该向现有工作人员发送一条消息（通过 SendMessage）还是生成一个新工作人员（通过代理）？该决策是上下文重叠的函数：

- 高重叠，相同文件：继续。工作人员已经在其上下文中拥有文件内容，理解模式，并且可以在其先前的工作基础上进行构建。产生新的内容将迫使重新读取相同的文件并重新得出相同的理解。

- 低重叠，不同领域：新鲜生成。刚刚调查身份验证系统的工作人员携带 20,000 个特定于身份验证上下文的令牌，这对于 CSS 重构任务来说是沉重的负担。开始清洁更便宜。

- 高度重叠，但工人失败了：产生新鲜的，并明确指导出了什么问题。继续一个失败的工人通常意味着与混乱的环境作斗争。重新开始“之前的尝试失败是因为X，避免Y”更可靠。

- 后续需要工作人员的输出：继续，输出包含在 SendMessage 中。 Worker 不需要重新导出自己的结果。

工人prompt写作和反模式。 该prompt教协调员如何编写有效的工作prompt并明确标记不良模式：

反模式：“根据你的研究结果，实施修复。” 这代表理解力。该工作人员不是进行研究的人 - 协调员阅读了研究结果。

反模式：“修复 auth 模块中的错误。” 没有文件路径，没有行号，没有错误描述。工作人员必须从头开始搜索整个代码库。

反模式：“对所有其他文件进行相同的更改。” 哪些文件？有什么变化？协调员知道；它应该枚举它们。

好的模式：“在第 42 行的 `src/auth/validate.ts` 中，从 `src/oauth/callback.ts:89` 调用时，`userId` 参数可以为 null。添加 null 检查：如果 `userId` 为 null，则返回 `{ error: 'unauthorized', status: 401 }`。然后更新 `src/auth/__tests__/validate.test.ts` 中的测试以覆盖 null案”。

编写特定prompt的费用由协调员一次性承担。其好处——工人在第一次尝试时就能正确执行——是巨大的。模糊的prompt创造了一种虚假的经济：协调员节省了 30 秒的prompt写作时间，而工作人员则浪费了 5 分钟的探索时间。

## 进程中的队友

进程内的队友与领导者在同一个 Node.js 进程中运行，通过“AsyncLocalStorage”进行隔离。他们的状态通过团队特定领域扩展了基础：

` 分机
` 导出类型 InProcessTeammateTaskState = TaskStateBase & {
类型：'in_process_teammate'
身份：队友身份
prompt：字符串
messages?: Message [] // 上限为 50
待处理的用户消息：字符串[]
是否空闲：布尔值
关闭请求：布尔值
等待计划批准：布尔值
权限模式：权限模式
onIdleCallbacks?: 数组 <() => void >
当前WorkAbortController?: AbortController
} `
`

50 个条目的“消息”上限值得解释。在开发过程中，分析显示每个进程中的代理在 500 多次转弯时积累了大约 20MB 的 RSS。据观察，Whale 会话（运行扩展工作流程的高级用户）在 2 分钟内启动了 292 个代理，将 RSS 提升至 36.8GB。 UI 表示的 50 条消息上限是一个内存安全阀。特工的实际对话继续进行，并具有完整的历史记录；仅面向 UI 的快照被截断。

“isIdle”标志启用工作窃取模式。空闲的队友不会消耗令牌或 API 调用 - 它只是在等待下一条消息。 `onIdleCallbacks` 数组让系统能够Hook从活动到空闲的转换，从而启用诸如“等待所有队友完成，然后继续”之类的编排模式。

“currentWorkAbortController”与队友的主中止控制器不同。中止当前工作控制器会取消队友正在进行的回合，但不会杀死队友。这实现了“重定向”模式：领导者发送更高优先级的消息，队友当前的工作被中止，队友接收新消息。主中止控制器在中止时会完全杀死队友。两个级别的干扰对应两个级别的意图。

`shutdownRequested` 标志实现协作终止。当领导者发送关闭请求时，该标志被设置。队友可以在自然停止点检查它并优雅地结束——完成当前文件写入、提交更改或发送最终状态更新。这比硬删除更温和，硬删除可能会使文件处于不一致的状态。

## 邮箱

队友通过基于文件的邮箱系统进行通信。当“SendMessage”以队友为目标时，消息将写入磁盘上收件人的邮箱文件中：

` 分机
` 等待 writeToMailbox (recipientName, {
来自：发件人姓名，
文本：内容，
总结，
时间戳：新的日期（）。 到ISOString(),
颜色：发件人颜色，
}, 团队名称) `
`

消息可以是纯文本、结构化协议消息（关闭请求、计划批准）或广播（`to: "*"` 发送给除发送者之外的所有团队成员）。轮询器Hook处理传入的消息并将其路由到队友的对话中。

基于文件的方法非常简单。没有消息代理，没有事件总线，没有共享内存通道。文件是持久的（在进程崩溃时幸存）、可检查的（你可以“cat”邮箱）并且便宜（没有基础设施依赖性）。对于每个会话消息量以数十个而不是每秒数千个来衡量的系统来说，这是正确的权衡。 Redis 支持的消息队列会增加操作复杂性、依赖性和故障模式 - 所有这些都是为了文件系统调用可以轻松处理的吞吐量要求。

广播机制值得一提。当消息发送到“*”时，发送者会迭代团队文件中的所有团队成员，跳过自身（不区分大小写的比较），并分别写入每个成员的邮箱：

` 分机
` for ( teamFile.members 的 const 成员) {
if (member.name.toLowerCase() === senderName.toLowerCase()) continue
收件人。 推送（成员.名称）
}
for ( const 收件人姓名) {
等待 writeToMailbox (recipientName, { from: senderName, text: content,... }, teamName)
} `
`

没有扇出优化 - 每个接收者都会获得单独的文件写入。同样，对于代理团队的规模（通常为 3-8 名成员）来说，这已经完全足够了。如果一个团队有 100 名成员，则需要重新考虑这一点。但是，阻止 36GB RSS 场景的 50 条消息内存上限也隐式限制了有效团队规模。

## 路由调度

`call()` 方法遵循优先级排序的调度链：

1. 桥接消息（`bridge:<session-id>`）。通过 Anthropic 的远程控制服务器进行跨机器通信。这是最广泛的覆盖范围 - 不同机器上的两个 Claude Code 实例，可能位于不同的大陆，通过中继进行通信。该系统在发送桥接消息之前需要明确的用户同意——这是一种安全检查，可防止一个代理单方面与远程实例建立通信。如果没有此门，受损或混乱的代理可能会将信息泄露到远程会话。同意检查使用“postInterClaudeMessage()”，它处理通过远程控制中继的序列化和传输。

2. UDS 消息（`uds:<socket-path>`）。通过 Unix 域套接字进行本地进程间通信。这适用于在同一台计算机上但在不同进程中运行的 Claude Code 实例 - 例如，托管一个实例的 VS Code 扩展和托管另一个实例的终端。 UDS 通信快速（无网络往返）、安全（文件系统权限控制访问）且可靠（内核处理传递）。 “sendToUdsSocket()”函数序列化消息并将其写入“to”字段中指定的套接字路径。对等点通过扫描活动 UDS 端点的“ListPeers”工具发现彼此。

3. 进程内sub-agent路由（普通名称或代理 ID）。这是最常见的路径。路由逻辑：

- 在“agentNameRegistry”中查找“input.to”

- 如果找到并运行：`queuePendingMessage()` — 消息等待下一个工具轮边界

- 如果找到但处于最终状态： `resumeAgentBackground()` — 代理会透明地重新启动

- 如果不在“AppState”中：尝试从磁盘记录中恢复

4. 团队邮箱（团队上下文处于活动状态时的回退）。指定收件人收到写入其邮箱文件的邮件。 “*”通配符会触发向所有团队成员的广播。

## 结构化协议

除了纯文本之外，SendMessage 还带有两个正式协议。

关闭协议。 领导者向队友发送“{ type: 'shutdown_request', Reason: '...' }”。队友用“{ type: 'shutdown_response', request_id,approve: true/false, Reason }”进行响应。如果获得批准，进程中的队友将中止他们的控制器；基于 tmux 的队友会收到 `gracefulShutdown()` 调用。该协议是合作性的——如果正在进行关键工作，队友可以拒绝关闭请求，而领导者必须处理这种情况。

计划批准协议。 以计划模式操作的队友必须在执行前获得批准。他们提交计划，领导者以“{ type: 'plan_approval_response', request_id,approval, Feedback }”进行响应。只有团队领导才能发出批准。这创建了一个审查门——领导者可以在接触任何文件之前检查员工的预期方法，从而及早发现误解。

## 自动恢复模式

路由系统最优雅的特性是透明的代理恢复。当“SendMessage”针对已完成或已杀死的代理时，它不会返回错误，而是复活该代理：

` 分机
` if (task.status!== '正在运行') {
const 结果 = 等待resumeAgentBackground ({
代理 ID,
prompt：输入消息，
toolUseContext：上下文，
可以使用工具，
})
返回{
数据：{
成功：真实，
消息：`代理“${ input. to }”已停止；继续显示您的消息`
}
}
} `
`

`resumeAgentBackground()` 函数根据磁盘记录重建代理：

- 读取侧链 JSONL 转录本

- 重建消息历史记录，过滤孤立的思维障碍和未解决的工具使用

- 重建内容替换状态以提高缓存稳定性

- 从存储的元数据解析原始代理定义

- 使用新的中止控制器重新注册为后台任务

- 使用恢复的历史记录和新消息作为prompt调用“runAgent()”

从协调者的角度来看，向死亡代理发送消息和向活动代理发送消息是相同的操作。路由层处理复杂性。这意味着协调员不需要跟踪哪些代理还活着——他们只需发送消息，系统就会计算出来。

其影响是重大的。如果没有自动恢复，协调器将需要维护代理活动的心理模型：“‘researcher’还在运行吗？让我检查一下。它已完成。我需要生成一个新的代理。但是等等，我应该使用相同的名称吗？它会有相同的上下文吗？”通过自动恢复，所有这些都可以简化为：“向‘研究人员’发送消息。”如果它还活着，则该消息将排队。如果它死了，它会带着完整的历史复活。协调器的即时复杂性急剧下降。

当然，这是有代价的。从磁盘记录中恢复意味着重新读取潜在的数千条消息、重建内部状态以及使用完整上下文窗口进行新的 API 调用。对于长期代理来说，这在延迟和令牌方面都可能很昂贵。但替代方案——要求协调员手动管理代理生命周期——更糟糕。协调员是法学硕士。它擅长推理问题和编写指令。簿记方面很糟糕。自动恢复完全消除了簿记类别，从而发挥了法学硕士的优势。

## 该实现委托给“stopTask()”，它根据任务类型进行调度：

- 在“AppState.tasks”中查找任务

- 调用 `getTaskByType(task.type).kill(taskId, setAppState)`

- 对于代理：中止控制器，将状态设置为“已杀死”，启动驱逐计时器

- 对于 shell：杀死进程组

该工具有一个旧别名“KillShell”——提醒人们，任务系统是从更简单的起源发展而来的，其中唯一的后台操作是 shell 命令。

终止机制因任务类型而异，但模式是一致的。对于代理来说，终止意味着中止中止控制器（这会导致“query()”循环在下一个屈服点退出），将状态设置为“已终止”，并启动逐出计时器，以便在宽限期后清除任务状态。对于 shell，终止意味着向进程组发送信号 — 首先发送“SIGTERM”信号，如果进程在超时内未退出，则发送“SIGKILL”信号。对于正在处理的队友，杀死也会触发团队关闭通知，以便其他成员知道队友已经离开。

驱逐计时器值得注意。当代理被杀死时，其状态不会立即清除。它会在“AppState.tasks”中停留一段宽限期（由“evictAfter”控制），以便 UI 可以显示终止状态，可以读取任何最终输出，并且仍然可以通过 SendMessage 自动恢复。宽限期过后，状态将被垃圾收集。这与用于已完成任务的模式相同 - 系统区分“已完成”（结果可用）和“忘记”（状态已清除）。

## 在模式之间进行选择

（命名注意事项：代码库还包含管理结构化待办事项列表的“TaskCreate”/“TaskGet”/“TaskList”/“TaskUpdate”工具，这是一个与此处描述的后台任务状态机完全独立的系统。“TaskStop”在“AppState.tasks”上运行；“TaskUpdate”在项目跟踪数据存储上运行。命名重叠是历史性的，也是模型混乱的经常性来源。）

有了三种可用的编排模式——后台委托、协调器模式和群体团队——自然的问题是何时使用每种模式。

当父级有一两个独立任务需要卸载时，简单委派（具有“run_in_background：true”的代理工具）适用。在继续编辑的同时在后台运行测试。在等待构建时搜索代码库。家长保持控制，准备好后检查结果，并且永远不需要复杂的通信协议。开销很小——一个任务状态条目、一个磁盘输出文件、一个完成通知。

当问题分解为研究阶段、综合阶段和实施阶段时，以及当协调员需要在指导下一步之前对多个工作人员的结果进行推理时，协调员模式是合适的。协调员无法接触文件，这迫使关注点完全分离：思考发生在一种环境中，做事发生在另一种环境中。 370 行的system prompt不是仪式——它编码的模式可以防止 LLM 委托最常见的失败模式，即委托理解而不是委托行动。

Swarm 团队适合长时间运行的协作会话，在这些会话中，代理需要点对点通信，工作是持续进行的而不是面向批处理的，并且代理可能需要根据传入消息进行空闲和恢复。邮箱系统支持协调器模式（同步spawn-wait-synthesize）不支持的异步模式。计划审批门添加了审查层。权限转发可以维护安全性，而不需要每个代理都拥有完整的权限。

## 实用的决策表：

场景模式 为何在编辑时运行测试 简单委派 一个后台任务，无需协调 搜索所有用途的代码库 简单委派 一劳永逸，完成后读取输出 重构跨 3 个模块的 40 个文件 协调器 研究阶段查找模式、综合计划更改、每个模块并行执行 具有审查门的多日功能开发 集群 长寿命代理、计划批准协议、对等通信 修复已知位置的错误 两者都不是 - 单一代理 编排开销超过了集中、顺序工作的好处 迁移数据库模式 + 更新API + 更新前端协调器 共享研究/规划阶段后的三个独立工作流 结对编程与用户监督 集群与计划模式 工人提出，领导批准，工人执行
这些模式在原则上并不相互排斥，但在实践中却是相互排斥的。协调器模式禁用 fork sub-agent。 Swarm 团队有自己的通信协议，不与协调器任务通知混合。该选择是在会话启动时通过环境变量和功能标志做出的，它塑造了整个交互模型。

最后一个观察：最简单的模式几乎总是正确的起点。大多数任务不需要协调器模式或群体团队。一个偶尔有背景委托的单一代理处理绝大多数的开发工作。复杂的模式存在于问题真正广泛、真正并行或真正长期运行的 5% 的情况下。在单文件错误修复上使用协调器模式就像为静态网站部署 Kubernetes — 技术上可行，但架构上不合适。

## 编排成本

在研究编排层从哲学上揭示的内容之前，值得先承认它的实际成本。

每个后台代理都是一个单独的 API 对话。它有自己的上下文窗口、自己的令牌预算和自己的Prompt Cache槽。一个产生 5 个研究人员的协调器正在进行 6 个并发 API 调用，每个调用都有自己的system prompt、工具定义和 CLAUDE.md 注入。令牌开销并不小——仅system prompt就可能有数千个令牌，并且每个工作人员都会重新读取其他工作人员可能已经读取过的文件。

通信通道增加了延迟。磁盘输出文件需要文件系统 I/O。任务通知是在工具轮边界传递的，而不是立即传递的。命令队列引入了完整的往返延迟 - 协调器发送一条消息，该消息等待工作器完成当前的工具使用，工作器处理该消息，并将结果写入磁盘以供协调器读取。

状态管理增加了复杂性。七种任务类型、五种状态以及每个任务状态的数十个字段。逐出逻辑、垃圾收集计时器、内存上限——所有这些都存在，因为无限制的状态增长导致了实际的生产事件（36.8GB RSS）。

这些都不意味着编排是错误的。这意味着编排是一种有成本的工具，应该权衡成本与收益。当搜索需要连续 5 分钟时，运行 5 个并行工作线程来搜索代码库是值得的。运行协调器来修复一个文件中的拼写错误是纯粹的开销。

## 编排层揭示了什么

该系统最有趣的方面不是任何单独的机制 - 任务状态、邮箱和通知 XML 都是简单的工程。有趣的是它们如何组合在一起所产生的设计理念。

协调员prompt的“永远不要委托理解”不仅仅是LLM编排的好建议。它是关于基于上下文窗口的推理的基本局限性的陈述。具有新上下文窗口的工作人员无法理解协调员在阅读 50 个文件并综合三份研究报告后所理解的内容。弥合这一差距的唯一方法是协调员将其理解提炼成具体的、可操作的prompt。模糊的授权不仅效率低下，而且从理论上讲，它也是有损失的。

SendMessage 中的自动恢复模式揭示了人们更喜欢表面上的简单性而不是实际的简单性。实现很复杂——读取磁盘记录、重建内容替换状态、重新解析代理定义。但界面很简单：发送一条消息，无论收件人是活着还是死了，它都会起作用。基础设施吸收了复杂性，以便模型（和用户）可以用更简单的术语进行推理。

进程内队友的 50 条消息内存上限提醒我们，编排系统是在真正的物理限制下运行的。 292 个代理在 2 分钟内达到 36.8GB RSS 并不是理论上的问题 - 它发生在生产中。这些抽象很优雅，但它们运行在内存有限的硬件上，当用户将其推向极端时，系统必须优雅地降级。

分层架构本身也有一个教训。任务状态机是不可知的——它不知道协调器或群。通信通道是不可知的——SendMessage 不知道它是由协调器、群领导者还是独立代理调用。协调器prompt位于顶部，在不改变底层机制的情况下添加方法。每一层都可以独立理解、独立测试、独立演化。当团队添加 swarm 系统时，他们不需要修改任务状态机。当他们添加协调器prompt时，不需要修改SendMessage。

这是精心设计的编排的标志：原语是通用的，模式是由它们组成的。协调员只是一个具有有限工具和详细system prompt的代理。群体领导者只是具有团队上下文和邮箱访问权限的代理。后台工作者只是一个具有独立中止控制器和磁盘输出文件的代理。七种任务类型、五种状态和四种路由模式组合起来产生大于各部分之和的编排模式。

在编排层，Claude Code 不再是单线程工具执行器，而是更接近于开发团队。任务状态机提供簿记功能。沟通渠道提供信息流。协调员prompt提供了方法。群体系统为不符合严格层次结构的问题提供了点对点拓扑。它们共同使语言模型能够完成单个模型调用无法完成的任务：并行、协调地处理广泛的问题。

下一章将研究**权限系统**——安全层，它决定这些代理中的哪些人可以做什么，以及危险操作如何从工人升级到人类。没有权限控制的编排将会导致错误的发生。**权限系统**确保更多的代理意味着更多的能力，而不是更多的风险。
