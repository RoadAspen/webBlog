# 第 7 章：并发工具执行

## 等待的成本

第 6 章跟踪了单个工具调用的生命周期——从 API 响应中的原始“tool_use”块到输入验证、权限检查、执行和结果格式化。该管道处理一种工具。但该模型很少只需要一个。

典型的Claude Code交互每轮涉及三到五个工具调用。 “读取这两个文件，grep 查找此模式，然后编辑此函数。”该模型在单个响应中发出所有这些内容。如果每个工具需要 200 毫秒，则依次运行它们将花费一整秒。如果 Read 和 Grep 调用是独立的（而且确实如此），并行运行它们会将时间缩短至 200 毫秒。五对一改进，免费。

但并非所有工具都是独立的。修改“config.ts”的编辑不能与修改“config.ts”的另一个编辑同时运行。创建目录的 Bash 命令必须在将文件写入该目录的 Bash 命令之前完成。并发性不是工具的全局属性。它是具有特定输入的特定工具调用的属性。

这是驱动整个并发系统的洞察力：安全性是针对每个调用的，而不是针对每个工具类型的。 `Bash("ls -la")` 可以安全地并行化。 `Bash("rm -rf build/")` 不是。相同的工具，不同的输入，不同的并发分类。系统在做出决定之前必须检查输入。

Claude Code 实现了两层并发优化。第一个是批量编排：在完全接收到模型的响应后，将工具调用分为并发组和串行组，然后适当地执行每个组。第二种是推测执行：在模型仍在传输其响应时开始运行工具，在响应完成之前收集结果。这两种机制共同消除了大部分原本用于等待的挂钟时间。

该算法从左到右遍历数组。对于每个工具调用：

- 按名称查找工具定义。

- 通过“safeParse()”使用工具的 Zod 模式解析输入。如果解析失败，该工具将被保守地归类为非并发安全的。

- 在工具定义上调用“isConcurrencySafe(parsedInput)”。这就是按输入进行分类的地方。 Bash 工具解析命令字符串，检查每个子命令是否是只读的（`ls`、`grep`、`cat`、`git status`），并且仅当整个复合命令是纯读取时才返回 `true`。读取工具始终返回“true”。编辑工具始终返回“false”。该调用包含在 try-catch 中 - 如果“isConcurrencySafe”抛出异常（例如，Bash 命令字符串无法由 shell-quote 库解析），则该工具默认为串行。

- 合并或创建批次。 如果当前工具是并发安全的并且最近的批次也是并发安全的，则追加到该批次。否则，开始新一批。

结果是在并发组和单独的串行条目之间交替的一系列批次。来看一个具体的例子：

分区是贪婪且保序的。连续的安全工具累积到一个批次中。任何不安全的工具都会中断运行并开始新的批次。这意味着模型发出工具调用的顺序很重要 - 如果它在两次读取之间交错写入，您将获得三批而不是两批。在实践中，模型倾向于将它们的读取聚集在一起，这是算法优化的常见情况。

`all()` 实用程序是具有有限并发性的 `Promise.all` 的生成器感知变体。它同时启动最多 N 个生成器，从先完成的生成器中生成结果，并在每个生成器完成时启动下一个排队的生成器。该机制类似于信号量保护的任务池，但适用于产生中间结果的异步生成器。

上下文修饰符队列是微妙的部分。一些工具会生成上下文修饰符 - 为后续工具转换“ToolUseContext”的函数。当工具并发运行时，您无法立即应用这些修饰符，因为同一批次中的其他工具正在读取相同的上下文。相反，修改器被收集在由工具使用 ID 键入的映射中：

` 分机

实际上，当前的并发安全工具都没有生成上下文修饰符——代码库中的注释明确地承认了这一点。但基础设施之所以存在，是因为 MCP 服务器可以添加工具，并且自定义只读 MCP 工具可能合法地想要修改上下文（例如，更新“已看到的文件”集）。

## 连续批次

串行执行很简单。每个工具都会运行，其上下文修饰符会立即应用，下一个工具会看到更新后的上下文：

` 分机
` for ( const toolUse of toolUseMessages) {
for wait ( const update of runToolUse (toolUse, /*... */ )) {
if (update.contextModifier) {
currentContext = update.contextModifier。 修改上下文（当前上下文）
}
产量 { 消息： update.message，newContext： currentContext }
}
} `
`

这是关键的区别。串行工具可以改变后续工具的世界。编辑修改文件；下一次阅读会看到修改后的版本。 Bash 命令创建一个目录；下一个 Bash 命令将写入其中。上下文修饰符是这种依赖关系的形式化：它们让工具说“执行环境已经改变，具体方式如下”。

## 流媒体工具执行器

批量编排消除了模型响应到达后不必要的序列化。但还有一个更大的机会：模型的响应需要时间来传输。典型的多工具响应可能需要 2-3 秒才能完全到达。第一个工具调用在 500 毫秒后即可解析。为什么要等剩下的2秒呢？

`StreamingToolExecutor` 类实现推测执行。当模型传输其响应时，每个“tool_use”块在完全解析后就会被交给执行器。执行器立即开始运行它——而模型仍在生成下一个工具调用。当响应完成流式传输时，多个工具可能已经完成。

连续总时间：3.1秒。流传输总计：2.6 秒 — 工具 1 和 2 在流传输期间完成，节省了 16% 的挂钟时间。

储蓄复利。当模型请求五个只读工具并且响应需要 3 秒才能传输时，所有五个工具都可以在这 3 秒内启动和完成。流后排出阶段没有什么可做的。用户几乎在模型响应的最后一个字符出现后立即看到结果。

## 工具生命周期

## 执行器跟踪的每个工具都会经历四种状态：

- 已排队：“tool_use”块已被解析和注册。等待并发条件允许执行。

- 正在执行：工具的“call()”函数正在运行。结果累积在缓冲区中。

- 完成：执行完成。结果已准备好交付给对话。

- 产生：结果已发出。终端状态。

## addTool()：在流期间排队

` 分机
` addTool（块：ToolUseBlock，assistantMessage：AssistantMessage）：void `
`

每次完整的“tool_use”块到达时，由流响应解析器调用。方法：

- 查找工具定义。如果未找到，则立即创建一个带有错误消息的“已完成”条目——对不存在的工具进行排队是没有意义的。

- 使用与“partitionToolCalls()”相同的逻辑解析输入并确定“isConcurrencySafe”。

- 推送状态为“已排队”的“TrackedTool”。

- 调用“processQueue()”——这可能会立即启动该工具。

对“processQueue()”的调用是“即发即忘”(“void this.processQueue()”)。执行者不等待它。这是故意的：从流解析器的事件处理程序中调用 addTool() ，并且阻塞会停止响应解析。该工具开始在后台执行，同时解析器继续使用流。

## processQueue()：准入检查

## 准入检查是单个谓词：

` 分机
` // 伪代码 — 说明互斥规则
canRun = noToolsRunning || (newToolIsSafe && allRunningAreSafe) `
`

## 当且仅当满足以下条件时，工具才能开始执行：

- 当前没有工具正在执行（队列为空），或者

- 新工具和所有当前正在执行的工具都是并发安全的。

这是一份互斥合同。非并发工具需要独占访问——没有其他工具可以运行。并发工具可以与其他并发工具共享跑道，但执行集中的单个非并发工具会阻止所有人。

“processQueue()”方法按顺序迭代所有工具。对于每个排队的工具，它都会检查“canExecuteTool()”。如果该工具可以运行，则它会启动。如果非并发工具尚无法运行，则循环将中断 - 它完全停止检查后续工具，因为非并发工具必须保持顺序。如果并发工具无法运行（被正在执行的非并发工具阻止），则循环将继续 - 但实际上这几乎没有帮助，因为非并发阻止程序之后的并发工具通常都依赖于其结果。

## executeTool()：核心执行循环

这种方法才是真正复杂的地方。它管理中止控制器、错误级联、进度报告和上下文修改器。

儿童中止控制器。 每个工具都有自己的“AbortController”，它是共享同级控制器的子级。

该层次结构深三层：查询级控制器（由 REPL 拥有，在用户 Ctrl+C 时触发）是同级控制器（由流执行器拥有，在 Bash 错误时触发）的父级，同级控制器是每个工具的单独控制器的父级。中止同级控制器会杀死所有正在运行的工具。中止工具的单独控制器只会杀死该工具 - 但如果中止原因不是同级错误，它也会冒泡到查询控制器。这种冒泡可以防止系统默默地丢弃执行器，例如，当权限拒绝应该结束整个回合时。

这种冒泡对于拒绝许可至关重要。当用户在权限对话框中拒绝某个工具时，该工具的中止控制器将触发。该信号必须到达query loop，以便它可以结束转弯。如果没有它，query loop将继续下去，就好像什么也没发生一样，向模型发送过时的拒绝消息。

同级错误级联。 当工具产生错误结果时，执行器检查是否取消同级工具。规则：只有 Bash 错误才会级联。 当 shell 命令出错时，执行器会记录失败，捕获出错工具的描述，并中止同级控制器 - 这会取消批处理中的所有其他正在运行的工具。

其理由是务实的。 Bash 命令通常形成隐式依赖链：`mkdir build && cp src/* build/ && tar -czf dist.tar.gz build/`。如果“mkdir”失败，则运行“cp”和“tar”是没有意义的。立即取消同级可以节省时间并避免令人困惑的错误消息。

相比之下，Read 和 Grep 错误是独立的。如果一个文件读取失败，因为该文件被删除，这对并发 grep 搜索不同目录没有影响。取消 grep 会无缘无故地浪费工作。

## 错误级联为同级工具生成合成错误消息：

` 分机
` 已取消：并行工具调用 Bash(mkdir build) 出错 `
`

描述包括出错工具的命令或文件路径的前 40 个字符，为模型提供足够的上下文来理解出了什么问题。

进度消息与结果分开处理。虽然结果按顺序缓冲和生成，但进度消息（状态更新，如“正在读取文件...”或“正在搜索...”）会转到“pendingProgress”数组，并通过“getCompletedResults()”立即生成。当新进度到达时，解析回调会唤醒“getRemainingResults()”循环，从而防止 UI 在长时间运行的工具期间出现冻结状态。

队列重新处理。 每个工具完成后，会再次调用“processQueue()”：

` 分机
` 无效的承诺。 最后（（）=> {
作废这个。 进程队列()
}) `
`

这就是被并发批处理阻塞的串行工具的启动方式。当最后一个并发工具完成时，后续非并发工具的“canExecuteTool()”检查通过，并开始执行。

## 结果收获

流式执行器公开了两种收集方法，专为响应生命周期的两个不同阶段而设计。

`getCompletedResults()` — 中流收获。 这是在流 API 响应块之间调用的同步生成器。它按顺序遍历工具数组并为任何已完成的工具生成结果：

`getCompletedResults()` 是一个同步生成器，它按提交顺序遍历工具数组。对于每个工具，它首先会耗尽所有待处理的进度消息。如果该工具完成，它会产生结果并将其标记为已产生。关键规则：如果非并发工具仍在执行，则遍历会中断 - 即使后续工具已经完成，之后也不会产生任何结果。串行工具之后的结果可能取决于其上下文修改，因此它们必须等待。对于并发工具，此限制不适用；该循环会跳过执行并发工具并继续检查后续条目。

这种打破就是秩序保持机制。如果非并发工具仍在执行，则在它之后不会产生任何结果 - 即使后续工具已经完成。串行工具之后的结果可能取决于其上下文修改，因此它们必须等待。对于并发工具，此限制不适用；该循环会跳过执行并发工具并继续检查后续条目。

`getRemainingResults()` — 流后耗尽。 在完全收到模型的响应后调用。这个异步生成器循环直到生成每个工具：

`getRemainingResults()` 是流后消耗。它循环直到每个工具都被生成。在每次迭代中，它都会处理队列（启动任何新解锁的工具），通过 getCompletedResults() 生成任何已完成的结果，然后 - 如果工具仍在执行但没有任何新任务完成 - 使用 Promise.race 来空闲等待以先完成者为准：任何执行工具的承诺或进度可用信号。这可以避免忙轮询，同时在发生事情时仍然醒来。当没有工具完成并且没有新的工具可以启动时，执行器将等待任何正在执行的工具完成（或进度到达）。这可以避免忙轮询，同时在发生事情时仍然醒来。

## 订单保存

结果是按照收到工具的顺序产生的，而不是按照工具完成的顺序产生的。这是一个经过深思熟虑的设计选择。

考虑一个请求 `[Read("a.ts"), Read("b.ts"), Read("c.ts")]` 的模型响应。三者同时开始。 `c.ts` 首先完成（它较小），然后是 `a.ts`，然后是 `b.ts`。如果结果按完成顺序产生，则对话将显示：

` 分机
` 工具结果：c.ts 内容
工具结果：a.ts 内容
工具结果：b.ts内容`
`

但模型按照 a-b-c 的顺序发出它们。对话历史记录必须符合模型的期望，否则下一回合将会混淆哪个结果对应哪个请求。通过按到达顺序让步，对话保持连贯：

` 分机
` 工具结果：a.ts 内容（第二完成，第一生成）
工具结果：b.ts内容（完成第三，产生第二）
工具结果：c.ts 内容（首先完成，产生第三）`
`

成本很小：如果工具 1 较慢而工具 2-5 较快，则快速结果将保留在缓冲区中，直到工具 1 完成。但另一种情况——谈话不连贯——情况要糟糕得多。

## 丢弃（）：流式后备逃生舱

当 API 响应流中途失败（网络错误、服务器断开连接）时，系统会重试新的 API 调用。但流执行器可能已经从失败的尝试中启动了工具。这些结果现在是孤立的——它们对应于从未被完全接收到的响应。

` 分机
` 丢弃 (): void {
这个.discarded = true
} `
`

## 设置 `discarded = true` 会导致：

- `getCompletedResults()` 立即返回，没有结果。

- `getRemainingResults()` 立即返回，没有结果。

- 任何开始执行的工具都会检查“getAbortReason()”，查看“streaming_fallback”，并获取合成错误而不是实际运行。

废弃的执行器被废弃。为重试尝试创建一个新的执行程序。

## 工具并发属性

每个内置工具都通过“isConcurrencySafe()”方法声明其并发特性。这种分类不是任意的——它反映了该工具对共享状态的实际影响。

工具并发安全条件基本原理始终读取 - 纯读取。无副作用。 Grep Always — 纯读。包裹 ripgrep。 Glob Always — 纯读。文件列表。 始终获取 — HTTP GET。无局部副作用。 WebSearch Always — 对搜索提供商的 API 调用。 Bash 有时只读命令只有 `isReadOnly()` 解析命令并对子命令进行分类。 `ls`、`git status`、`cat`、`grep` 是安全的。 `rm`、`mkdir`、`mv` 不是。 从不编辑 — 修改文件。对同一文件的两次并发编辑会损坏该文件。 从不写入 — 创建或覆盖文件。同样的腐败风险。 NotebookEdit 从不 — 修改 `.ipynb` 文件。
Bash 工具的分类值得详细阐述。它使用 splitCommandWithOperators() 分解复合命令（&&、||、;;、||），然后根据已知安全集对每个子命令进行分类：

- 搜索命令：`grep`、`rg`、`find`、`fd`、`ag`、`ack`

- 读取命令：`cat`、`head`、`tail`、`wc`、`jq`、`less`、`file`、`stat`

- 列表命令：`ls`、`tree`、`du`、`df`

- 中性命令：`echo`、`printf`（没有副作用，但不“读取”）

仅当每个非中性子命令都在搜索、读取或列表集中时，复合命令才是只读的。 `ls -la && cat README.md` 是安全的。 `ls -la && rm -rf build/` 不是 - `rm` 污染了整个命令。

## 中断行为契约

当工具执行时，用户可以键入新消息。应该发生什么？答案取决于工具。

每个工具都声明一个“interruptBehavior()”方法，该方法返回“cancel”或“block”：

- `'cancel'` ：立即停止工具，丢弃部分结果，并处理新的用户消息。由部分执行无害的工具使用（读取、搜索）。

- `'block'` ：保持工具运行直至完成。用户的新消息正在等待。由中断会使系统处于不一致状态的工具使用（写入运行中、长时间运行的 bash 命令）。这是默认设置。

## 流执行器跟踪当前工具集的可中断状态：

通过检查所有当前正在执行的工具来更新可中断状态：仅当每个正在执行的工具都支持取消时，该集合才是可中断的。如果即使一个工具的中断行为是“阻止”，则整个集合将被视为不可中断。

当所有执行工具都支持取消时，UI 仅显示“可中断”指示符。如果即使有一个工具被“阻止”，则整套工具将被视为不可中断。这是保守但正确的：您无法有意义地中断一个工具无论如何都会继续运行的批处理。

当用户确实中断并且所有工具都可以取消时，中止控制器会因“中断”原因而触发。执行器的 getAbortReason() 方法单独检查每个工具的中断行为 - “取消”工具收到合成的“user_interrupted”错误，而“阻止”工具（不会出现在完全可中断集中，但代码处理边缘情况）继续运行。

## 上下文修饰符：仅串行合约

上下文修饰符是类型为 `(context: ToolUseContext) => ToolUseContext` 的函数。他们让一个工具说“我已经改变了后续工具需要了解的执行环境的一些内容。”

这种不对称是故意的。如果工具 A 修改上下文并且工具 B 读取该上下文，则它们具有数据依赖性。数据依赖性意味着它们不能同时运行。根据定义，如果两个工具是并发安全的，则任何一个工具都不应该依赖于另一个工具的上下文修改。系统通过推迟申请来强制执行此操作。

## 应用此

Claude Code 中的并发模式可推广到任何编排多个独立操作的系统。三个原则值得总结。

按安全划分，而不是按类型划分。 isConcurrencySafe(input) 方法接收解析后的输入，而不仅仅是工具名称。这种每次调用分类比静态的“此工具类型始终安全”声明更精确。在您自己的系统中，在决定是否并行化之前检查操作的参数。数据库读取可以安全地并行化；数据库写入同一行则不然。仅操作类型并不能告诉您足够的信息。

I/O 等待期间的推测执行。 流执行器在 API 响应仍在到达时启动工具。相同的模式适用于具有缓慢生产者和快速消费者的任何地方：开始处理较早的项目，同时仍在生成较晚的项目。 HTTP/2 服务器推送、编译器管道并行性和推测 CPU 执行都共享此结构。关键要求是您可以在完整的指令集可用之前识别独立的工作。

保留结果中的提交顺序。 按完成顺序生成结果很诱人——它可以最大限度地减少第一个结果的延迟。但是，如果消费者（在本例中为语言模型）期望结果按特定顺序排列，则重新排序会造成混乱，从而花费更多的时间来解决问题，而不是节省的延迟。缓冲完成的结果并按照请求的顺序释放它们。实现成本是简单的数组遍历；正确性的好处是绝对的。

流式执行器模式对于代理系统来说特别强大。每当您的agent loop涉及“思考，然后行动”循环（其中思考阶段产生多个独立行动）时，您都可以将思考的尾部与行动的开始重叠。节省的时间与思考时间与行动时间的比率成正比。对于语言模型代理来说，思考时间（API 响应生成）占主导地位，节省的成本是巨大的。

## 总结

Claude Code 的并发系统在两个层面上运行。分区算法（“partitionToolCalls”）将连续的并发安全工具分组为并行运行的批次，同时将不安全的工具隔离为串行批次，其中每个工具都会看到前一个工具的效果。流工具执行器（`StreamingToolExecutor`）更进一步，在模型响应流期间到达时推测性地启动工具，将工具执行与响应生成重叠。

安全模型在设计上是保守的。并发安全性是通过检查解析的输入来确定每次调用的。未知工具默认为串行。解析失败默认为串行。安全检查中的异常默认为串行。系统永远不会猜测某些东西可以安全地并行化——工具必须肯定地声明它。

错误处理遵循工具的依赖结构。 Bash 错误会级联到同级错误，因为 shell 命令通常形成隐式管道。读取和搜索错误是隔离的，因为它们是独立的操作。中止控制器层次结构（查询控制器、同级控制器、每个工具控制器）使每个级别都能够取消其范围，而不会中断上面的级别。

结果是一个系统从模型的工具请求中提取最大的并行性，同时保持对话历史记录反映连贯、有序的操作序列的不变性。该模型按照其请求的顺序查看结果。用户看到工具的完成速度与底层操作允许的速度一样快。这两者之间的差距（执行速度与呈现顺序）是通过缓冲来弥补的，而该缓冲区是整个系统中最简单的部分。
